# 代码改动执行顺序

本文档是在 `docs/代码影响清单.md` 基础上，进一步整理出的**建议实施顺序**。

目标不是一次性把所有旧实现全推翻，而是按“先收口主链路，再做减法，最后再清理验证体系”的顺序推进，避免边改边乱。

---

## 1. 总体原则

当前代码改动建议遵守这 5 条原则：

1. **先定主线，再改细节**
   - 不先确定前端主线，就不要大规模改 UI

2. **先收口嵌入主链路，再删旧功能**
   - 先保证“Header -> 按钮 -> 弹窗/兜底页 -> 当前结果”成立

3. **先做减法，不先扩功能**
   - 当前最需要的是收口，而不是继续加能力

4. **先收前端入口，再决定后端是否删能力**
   - 很多旧能力先从 UI 下线即可，后端不必第一时间跟着硬删

5. **最后再改 E2E / CI**
   - 不要一开始就去改测试和流水线

---

## 2. 推荐执行顺序总览

建议按下面 6 个阶段推进：

### 阶段 A：确定前端主线

先决定未来以哪套前端为主。

### 阶段 B：收口嵌入主链路

先围绕：

- Header 生成
- 节点按钮注入
- 页内弹窗
- 独立页兜底
- 当前结果展示

把主链路收稳。

### 阶段 C：收口后台信息架构

把设置页、节点页、结果页收成真正符合当前产品方向的结构。

### 阶段 D：下线历史 / 变化主线

先从前端入口、路由、按钮开始减法。

### 阶段 E：调整开发环境口径

把 Debian 原生开发 + Docker 联调这件事在脚本和 README 上收口。

### 阶段 F：最后再改 Playwright / CI

等主链路和页面边界稳定后，再统一改验证体系。

---

## 3. 阶段 A：确定前端主线

## 目标

先明确：

- 继续以 `legacy-app.ts` 为主
- 还是切到 `react-preview` 为主

这是整个后续改动的前置决策。

---

## 为什么必须先做

因为现在实际存在两套前端：

- `web/src/legacy-app.ts`
- `web/src/react-preview/*`

如果这一步不先拍板，后续所有需求都会出现：

- 改了一套，另一套没改
- 文档方向变了，但默认入口还是旧实现
- Playwright 不知道该测谁

---

## 我建议的方向

我建议：

> **尽快切到 React 版作为唯一主线。**

理由：

1. 当前仓库已经有 React 预览骨架，不是从零开始
2. 文档里前端迁移方向已经明确过
3. 现在再继续维护 legacy，后面只会越来越重
4. 收口 UI、弹窗、设置页、结果页，用 React 更适合做减法和重组

---

## 本阶段涉及文件

- `web/src/main.ts`
- `web/src/legacy-app.ts`
- `web/src/react-preview/App.tsx`
- `web/src/react-preview/bootstrap.tsx`

---

## 本阶段建议动作

### A1. 拍板 React 为主线

先明确：

- 默认入口未来应指向 React
- legacy 改为过渡实现，不再作为继续演进主线

### A2. 调整入口策略

从当时的旧状态：

- 默认 legacy
- `?ui=react` 才进 React

改成当前已完成状态：

- 默认 React
- legacy 仅临时保留或直接移除

### A3. 给 legacy 明确“退场状态”

不要继续在 legacy 上加新功能。

---

## 本阶段验收标准

1. 团队内部已经明确“以后只改哪套前端”
2. 文档和默认入口方向一致
3. 不再出现“同一需求要改两套前端”的默认预期

### 当前执行状态

- 已拍板：**React 为唯一前端主线**
- 已完成：**默认前端入口切到 React**
- legacy 当前仅保留为临时兼容入口，不再是默认实现

---

## 4. 阶段 B：收口嵌入主链路

## 目标

把当前产品真正最关键的主链路收稳：

1. 本服务生成 header
2. 用户复制到 Komari
3. Komari 节点详情页出现按钮
4. 点击后优先打开页内弹窗
5. 如果弹窗不稳，则打开独立页面
6. 用户看到当前 IP 质量结果

---

## 本阶段为什么优先级最高

因为这条链路就是当前产品成立的核心。

相比之下：

- 历史页
- 变化页
- 变化规则
- reporter 配置展示

这些都不是当前第一优先级。

---

## 本阶段涉及文件

### 后端

- `internal/service/header.go`
- `internal/httpx/handlers/embed.go`
- `internal/httpx/handlers/admin.go`
- `internal/config/config.go`

### 前端

- `web/src/react-preview/App.tsx`
- `web/src/react-preview/lib/api.ts`
- `web/src/react-preview/lib/types.ts`

### 可能相关

- `internal/app/app.go`
- `internal/render/static.go`

---

## 本阶段建议动作

### B1. 明确 Header 预览的地址策略

核心目标：

- Header 默认优先基于 `IPQ_PUBLIC_BASE_URL` 生成
- 不再继续把“当前站点 origin + /ipq”当成默认产品假设

注意：

- 不是删除 `BasePath`
- 而是把它降级为兼容能力

### B2. 验证当前 loader 的跨域弹窗链路

当前 loader 已经有：

- 按钮注入
- 节点状态检查
- 节点注册
- 登录跳转
- iframe 弹窗
- 独立页链接

这里不应该先推翻，而应该先验证：

1. 跨域独立部署时 iframe 是否可用
2. 登录完成后弹窗链路是否顺
3. 浏览器行为是否稳定

### B3. 给弹窗明确“失败降级规则”

建议实现上要明确：

- 弹窗优先
- 如果某些条件不满足，就清晰退回独立页

不要让用户遇到“弹窗半开不开”的中间态。

### B4. 收口当前结果页

在弹窗和独立页里，优先保证：

- 当前结果可读
- 页面重点清晰
- 不要被其他设置或历史能力干扰

---

## 本阶段验收标准

1. Header 生成结果默认符合独立部署思路
2. Komari 节点详情页按钮可稳定出现
3. 点击按钮后能优先打开页内弹窗
4. 弹窗不可用时能稳定打开独立页
5. 当前结果内容可正常展示

---

## 5. 阶段 C：收口后台信息架构

## 目标

让后台页面结构服务于当前产品目标，而不是延续历史阶段里堆出来的信息架构。

---

## 本阶段涉及文件

- `web/src/react-preview/App.tsx`
- `web/src/react-preview/lib/types.ts`
- 可能涉及 `web/src/react-preview/lib/result.ts`

如果暂时还没切 React 主线，则同类问题会落在：

- `web/src/legacy-app.ts`

---

## 本阶段建议动作

### C1. 收口节点详情页

节点详情页只突出：

- 当前 IP 质量结果
- 必要操作
- 接入相关最小信息

不要继续把历史、变化、过多内部配置混在主内容里。

### C2. 收口系统配置页

系统配置页当前应优先保留：

- Header 生成
- 公开地址相关信息
- 管理员设置
- 变化规则（如仍保留）

### C3. 降低 reporter 配置区权重

后端能力可保留，但前端展示不应抢主结果页注意力。

---

## 本阶段验收标准

1. 节点详情页一眼能看到当前结果重点
2. 系统配置页首屏只保留当前确实需要的配置
3. 页面不再像“分析台”而更像“结果页 + 配置页”

---

## 6. 阶段 D：下线历史 / 变化主线

## 目标

把已经不属于当前主产品方向的这条线降级或下线。

---

## 本阶段涉及文件

### 前端

- `web/src/react-preview/App.tsx`
- `web/src/react-preview/lib/changes.ts`
- `web/src/compare.ts`
- `web/src/legacy-app.ts`

### 后端

- `internal/httpx/handlers/nodes.go`
- `internal/httpx/handlers/admin.go`
- `internal/service/settings.go`
- `internal/service/nodes.go`

---

## 本阶段建议动作

### D1. 先删入口，不先删底层

优先移除：

- 历史页入口
- 变化页入口
- 变化规则设置入口

先从 UI 层下线。

### D2. 再收路由

等 UI 确认不需要后，再决定：

- `/nodes/:uuid/history`
- `/nodes/:uuid/changes`
- `/admin/change-priority`

哪些直接下线，哪些临时保留。

### D3. 底层数据先不急着删

例如：

- `NodeHistory`
- 历史存储
- compare 辅助逻辑

这些不必第一时间物理删除。

先完成产品层减法，再做代码层彻底清理。

---

## 本阶段验收标准

1. 前端主界面不再出现历史 / 变化主线入口
2. 用户主路径不再依赖变化规则能力
3. 当前结果页不再被历史能力打断节奏

---

## 7. 阶段 E：调整开发环境口径

## 目标

让开发环境和当前文档口径一致：

- Debian 原生日常开发
- Docker 用于联调 / E2E / CI

---

## 本阶段涉及文件

- `README.md`
- `.env.example`
- `compose.dev.yml`
- `deploy/dev/caddy/Caddyfile`
- `deploy/dev/workspace/*`

---

## 本阶段建议动作

### E1. 更新 README

不要再把 Docker 容器工作流写成唯一方式。

### E2. 明确区分“默认独立部署”和“子路径兼容部署”

当前应在文档与环境变量说明里写清：

- `BasePath` 是兼容能力
- `PublicBaseURL` 才是当前主部署口径下更关键的配置

### E3. Compose 保留，但调整定位

保留：

- `compose.dev.yml`
- `Caddyfile`

但在表达上改成：

- 联调模型
- E2E 模型
- CI 模型

---

## 本阶段验收标准

1. README 与文档口径一致
2. Docker 不再被描述为唯一开发方式
3. 配置说明里不再把 `/ipq/` 子路径写成默认产品前提

---

## 8. 阶段 F：最后收 Playwright / CI

## 目标

让自动化验证的重点跟当前产品重点一致。

---

## 本阶段涉及文件

- `web/playwright/verify-injection.mjs`
- `web/playwright/verify-stage1-flows.mjs`
- `web/playwright/verify-react-preview-nodes.mjs`
- `.github/workflows/ci.yml`

---

## 本阶段建议动作

### F1. 重写验证重点

验证重点收回到：

- Header 生成
- 注入按钮出现
- 页内弹窗打开
- 独立页兜底
- 当前结果展示

### F2. 去掉与当前主线无关的重验证

例如：

- history
- changes
- change-priority
- 复杂变化分层

### F3. 最后再调整 CI 健康检查与 E2E 目标

这块目前已经完成一轮收口：

- CI 健康检查已改为：
  - `http://localhost:8090/`
- Playwright 应用侧默认改走独立入口
- Komari 侧继续走：
  - `http://proxy:8080/`

---

## 本阶段验收标准

1. E2E 的关注点与当前产品方向一致
2. CI 不再强依赖已下线的历史/变化功能
3. 自动化验证更聚焦主链路

### 当前执行状态

- 已完成：CI / Compose / Playwright 默认入口从 `/ipq` 收口到根路径独立部署
- 已完成：history / changes 在 React 主线中的验证改为“应回退到节点详情”
- 当前剩余重点：继续验证 loader 在真实跨域弹窗场景下的稳定性

---

## 9. 不建议现在立刻做的事

下面这些动作当前不建议一开始就做：

### 9.1 不要先全删历史数据结构

例如：

- `NodeHistory`
- 历史存储逻辑

先把产品层入口删掉再说。

### 9.2 不要先改 Compose 大结构

现有 Compose 仍然能用于联调和 CI，价值还在。

### 9.3 不要继续在 legacy 和 React 两边同时加新需求

这会不断放大维护成本。

### 9.4 不要先大改 BasePath 路由体系

当前最关键的问题不是“必须消灭子路径支持”，而是“不能再依赖它作为默认产品前提”。

---

## 10. 我建议的第一轮实际改动顺序

如果马上开始动代码，我建议第一轮按这个顺序来：

### 第一轮

1. 拍板 React 为唯一前端主线
2. 默认入口切到 React
3. 梳理并验证当前 loader 弹窗链路
4. 收口当前结果页，只保留主内容

### 第二轮

1. 从前端移除历史 / 变化 / 规则入口
2. 收口设置页，只保留当前确实需要的内容
3. 降低 reporter 配置区权重

### 第三轮

1. 更新 README / 开发环境脚本说明
2. 重写 Playwright 主流程
3. 再调整 CI

---

## 11. 当前结论

如果用一句话概括当前建议的执行顺序，就是：

> **先定前端主线，先收口 Header 注入与当前结果主链路，再做减法，最后再改验证体系。**

这比“先删一堆旧功能”更稳，也更符合当前代码库状态。

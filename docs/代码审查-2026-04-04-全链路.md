## 1. 发现的问题

### 1.1 中：遗留 `NodeHistory` 表仍被保留，但已经不参与历史保留、占用估算和主查询，升级实例会出现“数据库体积比页面显示更大”的错位

相关位置：

- [database.go:30](/root/Komari-ip-history/internal/database/database.go#L30)
- [database.go:35](/root/Komari-ip-history/internal/database/database.go#L35)
- [database.go:118](/root/Komari-ip-history/internal/database/database.go#L118)
- [history_retention.go:61](/root/Komari-ip-history/internal/service/history_retention.go#L61)
- [history_retention.go:136](/root/Komari-ip-history/internal/service/history_retention.go#L136)

当前启动迁移逻辑会：

- 保留旧的 `NodeHistory` 表
- 在 `migrateLegacyNodeTargets(...)` 中把旧历史复制到新的 `NodeTargetHistory`

但后续主链路已经只依赖：

- `NodeTargetHistory`

与此同时：

- 历史保留清理只删 `NodeTargetHistory`
- 历史占用估算也只统计 `NodeTargetHistory`

这意味着如果用户是从旧版本升级而来：

- 旧 `NodeHistory` 记录会继续留在库里
- 页面显示的“当前历史占用”会偏小
- 即使自动清理跑过，SQLite 文件体积也可能看起来没有预期那么明显地下去

这不是新安装实例的问题，但对升级实例是实际的维护债。

### 1.2 中：历史主线的关键查询缺少更贴近实际访问模式的复合索引，历史量上去后会先卡数据库层

相关位置：

- [models.go:57](/root/Komari-ip-history/internal/models/models.go#L57)
- [models.go:63](/root/Komari-ip-history/internal/models/models.go#L63)
- [models.go:64](/root/Komari-ip-history/internal/models/models.go#L64)
- [nodes.go:277](/root/Komari-ip-history/internal/service/nodes.go#L277)
- [history_events.go:155](/root/Komari-ip-history/internal/service/history_events.go#L155)
- [history_retention.go:145](/root/Komari-ip-history/internal/service/history_retention.go#L145)

当前 `NodeTargetHistory` 只有这些单列索引语义：

- `node_target_id`
- `recorded_at`

但主要查询模式已经变成：

1. 历史列表 / 对比页  
   `WHERE node_target_id IN (...) AND recorded_at ... ORDER BY recorded_at`

2. 自动清理  
   `WHERE recorded_at < cutoff AND is_favorite = false`

这两类都更适合复合索引，而不是只靠单列索引。  
历史量上去后，问题通常不会先出在 React，而会先出在 SQLite 扫描和排序。

这不是功能 bug，但属于已经能预见的结构性性能债。

### 1.3 中：快照对比页仍然需要为当前 IP 串行拉取全量历史，历史量一大时首次进入会明显变重

相关位置：

- [App.tsx:456](/root/Komari-ip-history/web/src/react-preview/App.tsx#L456)
- [App.tsx:481](/root/Komari-ip-history/web/src/react-preview/App.tsx#L481)
- [App.tsx:486](/root/Komari-ip-history/web/src/react-preview/App.tsx#L486)
- [App.tsx:2434](/root/Komari-ip-history/web/src/react-preview/App.tsx#L2434)
- [nodes.go:248](/root/Komari-ip-history/internal/service/nodes.go#L248)

当前对比页为了保住滑块拖动的顺滑体验，采用的是：

- 前端按页把当前 IP 的完整历史全部拉下来
- 每页 `100` 条
- 串行请求直到拉完
- 每条都带完整 `result`

这个取舍是产品上接受过的，但它的上限也很清楚：

- 首次进入对比页会越来越慢
- 网络体积、后端 JSON 解码、浏览器内存都会一起涨

这不是当前必须改的问题，但已经是快照对比页最主要的性能瓶颈。

## 2. 当前结论

本轮没有发现新的阻塞性功能回归。

当前系统主链路是闭环的：

- 多目标 IP
- 历史变化列表
- 快照对比
- 收藏快照
- 历史保留策略
- 注入弹窗游客 / 管理员分流

更值得后续优先处理的顺序建议是：

1. 处理旧 `NodeHistory` 遗留表的收口问题
2. 为 `NodeTargetHistory` 增加更贴近查询模式的复合索引
3. 等真实数据量上来后，再决定是否进一步优化快照对比页的全量历史拉取策略

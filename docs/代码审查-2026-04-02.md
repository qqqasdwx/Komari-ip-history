# 代码审查（2026-04-02）

本次审查以当前主分支代码为准，重点检查：

- Komari 注入链路
- `embed=1` 详情页
- 认证与会话
- 节点详情接口负载
- 前端弹窗挂载逻辑

本次**只做审查，不改业务代码**。

## Findings

### 1. 高：跨站 iframe 场景下，IPQ 登录态大概率无法稳定建立

位置：

- `internal/httpx/handlers/auth.go:58`
- `internal/httpx/handlers/auth.go:80`
- `internal/service/header.go:349`

问题：

- 当前登录只通过 `c.SetCookie(...)` 写会话 cookie。
- 代码里没有设置 `SameSite=None`。
- 当前产品方向又明确要求支持“IPQ 独立部署，然后通过 Komari header 注入弹窗使用”。
- 一旦 Komari 和 IPQ 处于**不同站点**，弹窗 iframe 里的 IPQ 页面就属于第三方上下文，现代浏览器会对默认 SameSite 策略的 cookie 做严格限制。

影响：

- 在真实独立部署场景里，用户可能会遇到：
  - 弹窗里反复要求登录
  - 登录成功后 iframe 仍拿不到会话
  - `connect` 页和详情页在 iframe 中循环跳转/回到未登录态

判断依据：

- 当前默认弹窗实现就是通过 loader 打开跨站 iframe。
- 会话 cookie 代码没有显式处理跨站 iframe 所需的 SameSite 策略。

建议：

- 这条要么明确降级为“只保证同站点 iframe”，要么补齐跨站 cookie / 会话策略，并做真实浏览器验证。

### 2. 中：节点详情接口仍然无条件加载并返回全量历史，和当前产品方向不一致

位置：

- `internal/service/nodes.go:44`
- `internal/service/nodes.go:51`
- `internal/service/nodes.go:191`
- `internal/service/nodes.go:203`
- `web/src/react-preview/App.tsx:671`

问题：

- `GetNodeDetail` 现在仍然会：
  - 查询整个 `history`
  - 把完整历史记录挂到 `NodeDetail`
- 但当前 React 主界面已经不再展示历史主线，详情页也只消费：
  - `name`
  - `updated_at`
  - `current_result`
  - `report_config`

影响：

- 节点历史一旦积累起来，详情页接口会越来越重。
- Komari 弹窗打开详情时，也会为**没在页面里使用的数据**付出额外查询和传输成本。
- 这会直接拖慢：
  - 独立详情页
  - 弹窗 iframe 首次打开
  - 后续任何节点详情相关联调

建议：

- 至少把“当前详情页响应”和“历史查询”拆开。
- 如果历史不再是主功能，详情接口不应该默认带全量 history。

### 3. 中：loader 对整页 DOM 的监听和扫描过重，长期会拖慢 Komari 页面

位置：

- `internal/service/header.go:406`
- `internal/service/header.go:424`
- `internal/service/header.go:455`
- `internal/service/header.go:503`
- `internal/service/header.go:685`
- `internal/service/header.go:703`

问题：

- 当前 loader 注入后会：
  - 对 `document.documentElement` 建立全量 `MutationObserver`
  - 每次变化后重新跑 `sync()`
  - 在 `extractUUID()` / `findMountContainer()` 里多次 `querySelectorAll(...)`
  - 最后还会 fallback 到 `document.body.innerText`

影响：

- Komari 本身就是一个实时更新较多的后台页面。
- 在图表、状态卡片、列表频繁刷新时，这种“全局监听 + 全文扫描”的方式很容易造成：
  - 不必要的 CPU 消耗
  - 页面卡顿
  - 注入按钮闪动或重复重新挂载

建议：

- 把扫描范围收窄到更明确的挂载区域。
- 避免在每轮同步里读取 `body.innerText` 这种整页文本。
- 给 `sync()` 增加更明确的短路条件，而不是依赖 DOM 变化后反复全局搜索。

### 4. 中：Header 预览的“自动推导地址”仍然依赖请求头拼接，反代链路稍复杂时容易生成错误地址

位置：

- `internal/httpx/handlers/admin.go:17`
- `internal/httpx/handlers/admin.go:50`
- `internal/httpx/handlers/admin.go:69`
- `internal/httpx/handlers/admin.go:177`

问题：

- 当前“未手动填写接入地址”时，会从：
  - `X-Forwarded-Proto`
  - `X-Forwarded-Host`
  - `Request.Host`
 直接拼 `effective_public_base_url`
- 这在简单开发环境里可用，但在真实代理链更复杂时，`X-Forwarded-Host` 可能不是最终用户真正想写进 header 的地址。

影响：

- 管理员如果是通过：
  - 内网地址
  - 临时端口
  - 中间反代域名
访问 IPQ，再直接复制 header，得到的脚本地址可能不可达或不是最终生产地址。

建议：

- 当前逻辑更适合作为“开发默认值”，不适合作为生产可信来源。
- 文档里应明确提醒：生产环境应手动固定接入地址。

## Open Questions

### 1. 独立部署是否真的要支持“跨站 iframe 登录闭环”

如果答案是“要”，那 Finding 1 需要优先处理。  
如果答案是“不要，只接受同站点/同站域 iframe”，那应该在文档里明确写死，不然当前产品表述和实现假设冲突。

### 2. 历史数据是不是只打算保留为内部能力

如果历史只做内部留档，而不是用户主功能，那详情接口现在的返回结构还需要继续收口。

## Testing Gaps

- 没有自动化覆盖“不同站点下的 iframe 登录态”。
- 没有自动化覆盖“历史记录很多时，详情接口/弹窗打开耗时”的回归。
- 没有自动化覆盖“真实 Komari 页面高频 DOM 变化下，loader 是否持续重算”的性能风险。

## 总结

当前代码主线已经能跑通：

- 独立页
- Komari 注入按钮
- 弹窗 iframe
- IPQuality 风格详情展示

但如果按产品目标继续往“独立部署 + Komari 内弹窗”推进，当前最需要优先盯住的不是样式，而是：

1. 跨站 iframe 会话是否成立
2. 详情接口是否继续背着全量历史
3. loader 对 Komari 页面性能的长期影响

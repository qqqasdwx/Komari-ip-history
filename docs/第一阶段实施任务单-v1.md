# 第一阶段实施任务单 v1

本文档只覆盖第一阶段：

- 修正 `Komari` 未接入节点点击行为
- 收口弹窗 / 独立页职责
- 收口 cron 时区语义与页面显示

目标是把第一阶段拆成可以直接进入编码的任务单。

---

## 1. 第一阶段目标

完成后应满足：

1. `Komari` 弹窗只用于查看当前 IP 质量
2. 未接入节点点击图标时不再把后台页面注入到 `Komari` 弹窗
3. 未接入节点会在新标签页打开后台，并自动弹出对应节点配置弹窗
4. 保存 cron 时同时保存解析时区
5. 节点端执行 cron 时按保存时区执行，而不是按节点机器默认时区执行
6. 页面能明确告诉用户“当前 cron 按哪个时区解析”

## 1.1 当前实现状态（截至 2026-04-17）

本阶段相关能力已经大部分落地，但当前分支的实际改动范围已经超出第一阶段，后续阶段中的通知、对外 API、节点绑定等能力也已经提前进入实现。

以第一阶段目标为准，当前状态可归纳为：

- **已完成**
  1. `Komari` 弹窗已收口为查看页，未接入节点不再继续把后台流程塞回 embed 弹窗
  2. 未接入节点会走 `/#/nodes?report_config=<uuid>`，并在节点列表页打开上报设置
  3. 后端已支持 `reporter_timezone`，cron 预览与安装配置都带 `timezone`
  4. 安装脚本与生成的 cron 文件已显式写入 `CRON_TZ=<timezone>`
  5. 页面已明确展示“当前规则按哪个时区解析”，并展示最近 10 次执行时间

- **尚未完成**
  1. 严格意义上的“第一阶段”目标已经收口完成
  2. 当前剩余工作主要转向后续阶段中的契约统一与工程性加固，不再是本阶段主链路缺口

- **验收结论**
  - 第一阶段主链路已完成
  - 时区手动编辑与“配置完成，请返回 Komari 重新查看”提示都已落地

## 5. 接口请求/响应定义

## 5.1 `PUT /api/v1/nodes/:uuid/report-config`

请求体建议：

```json
{
  "schedule_cron": "0 12 * * *",
  "timezone": "Asia/Shanghai",
  "run_immediately": true
}
```

返回体建议增加：

```json
{
  "schedule_cron": "0 12 * * *",
  "timezone": "Asia/Shanghai",
  "run_immediately": true,
  "next_runs": [
    "2026-04-14T12:00:00+08:00"
  ]
}
```

## 5.2 `GET /api/v1/nodes/:uuid/report-config/preview`

支持查询参数：

- `cron`
- `timezone`
- `run_immediately`

返回时带：

- `timezone`
- `next_runs`

## 5.3 `GET /api/v1/report/nodes/:uuid/install-config`

返回体增加：

```json
{
  "node_uuid": "...",
  "report_endpoint": "...",
  "reporter_token": "...",
  "schedule_cron": "0 12 * * *",
  "timezone": "Asia/Shanghai",
  "run_immediately": true,
  "target_ips": []
}
```

# 项目理解

> 注意：本文件是早期“项目理解草稿”，保留用于回顾历史思路。  
> 当前权威口径请优先查看：
> - `docs/项目背景.md`
> - `docs/Komari集成方式重定稿.md`

本文档用于整理我目前基于现有资料形成的项目理解，供你审核。  
这不是实现方案，也不是最终设计稿，而是对“这个项目要做什么、先做什么、暂时不做什么”的完整梳理。

## 0. 已确认决策

以下内容已经由你确认，不再视为待讨论项：

1. 后端是独立服务，不直接改造 `Komari` 官方后端。
2. 独立后端必须支持配置 `Komari` 地址以及访问 `Komari` 所需的认证信息。
3. API 路径设计遵循主流、通用、易理解的风格，不走过度自定义路线。
4. 前端集成方式不是修改 `Komari` 官方源码，而是通过 `Komari` 的自定义 header 注入执行少量 JavaScript 来实现。
5. 这种注入方案不是临时过渡，而是既定设计方向。
6. 注入的 JavaScript 应尽量少，尽量只负责桥接和挂载。
7. 展示样式、弹窗结构、交互细节尽量由独立后端提供，不把大量样式和结构硬编码在注入脚本里。
8. 数据模型可以参考 `references/IPQuality/`，这是节点上用于检测 IP 质量的实际库。
9. 开发环境也属于当前要一并整理好的范围，而不只是写业务代码。
10. 项目本身有独立的 Web 前端，用于配置和展示，不是只有 API 服务。
11. 之所以还要注入到 `Komari`，是为了避免用户再单独打开一个页面；主要使用体验仍然希望发生在 `Komari` 内。
12. 该独立 Web 前端需要提供类似 `Grafana` 的分享/嵌入思路：先在本服务上配置好显示内容，再提供一个“一键复制 Komari header”的能力，用户把生成结果填入 `Komari` 即可。

## 1. 项目定位

本项目是一个与 `Komari` 平台集成的“节点 IP 质量监控与展示系统”。

核心目标不是单独做一个新的监控面板，而是：

1. 以 `Komari` 现有节点体系为基础。
2. 复用 `Komari` 节点的唯一标识 `UUID` 作为关联键。
3. 将节点的 IP 质量信息接入到 `Komari` 的服务端和前端展示链路中。
4. 最终让用户可以在 `Komari` 节点详情页里直接查看某个节点的 IP 质量状态。

换句话说，这个项目本质上是一个“围绕 Komari 节点对象扩展出的能力模块”。  
它会以“独立服务 + 可嵌入 Komari”的形式存在：

- 独立服务负责配置、展示、资源承载和接口能力
- `Komari` 侧通过 header 注入来复用这些能力

因此它不是简单的后端插件，也不是完全脱离 `Komari` 的独立站点，而是一个可以独立运行、但以嵌入 `Komari` 为主要使用方式的服务。

## 2. 业务背景

根据现有文档，每个 `Komari` 节点未来都会执行一个 `IPQuality` 脚本，用于获取该节点当前 IP 的质量信息。  
这类信息可能包括但不限于：

- 风险评分
- 黑名单数量
- 流媒体解锁状态
- 其他与 IP 信誉、可用性、用途相关的指标

这些数据的最终用途是：

1. 节点侧采集数据。
2. 服务端接收或获取这些数据。
3. 前端在 `Komari` 中展示这些数据。

目前整个项目的终态是“节点每天上报 IP 质量数据，平台端可以查看和管理”。  
但当前只处于阶段 1，因此先把平台端展示链路跑通，再考虑节点上报链路。

## 3. 当前阶段的边界

现有文档明确给出了“阶段 1”范围，因此当前阶段的重点不是完整产品，而是打通最小可运行链路。

阶段 1 只需要完成以下事情：

### 3.1 服务端侧

服务端需要先具备两个能力：

1. 能通过 `Komari` 官方接口或 SDK 获取节点列表。
2. 能提供一个“按节点查询 IP 质量状态”的接口。

并且这部分后端不是内嵌在 `Komari` 代码中的，而是一个独立服务。  
因此它至少需要具备以下配置能力：

- `Komari` 服务地址
- 访问 `Komari` 的认证信息
- 独立服务自身的监听地址
- 未来可能扩展的缓存、日志和上报相关配置

其中第 2 点在阶段 1 可以返回模拟数据，不要求接入真实的 IPQuality 上报结果。

这意味着阶段 1 的服务端目标不是“真实接收和存储节点上报”，而是：

- 确认能正确识别 Komari 中有哪些节点。
- 确认能围绕节点 UUID 返回一份对应的状态数据。
- 给前端提供一个稳定的调用入口。

### 3.2 前端侧

前端实际上分为两层：

1. 本项目自己的独立 Web 前端。
2. 注入到 `Komari` 节点详情页中的轻量前端桥接层。

其中，独立 Web 前端负责：

- 配置展示内容
- 承载实际展示页面或展示资源
- 提供管理界面
- 提供“一键复制 Komari header”的能力

而 `Komari` 内部看到的按钮和弹窗，则是通过 header 注入方式接入这些能力。

在 `Komari` 的节点详情页中，需要增加一个入口按钮：

- 按钮名称：`查看 IP 质量`

按钮点击后的行为：

1. 拿到当前节点的 UUID 或节点标识。
2. 调用服务端查询接口。
3. 将返回的状态数据显示在弹窗中。

阶段 1 中，弹窗展示的数据可以是模拟 JSON，只要满足“与节点 UUID 对应”即可。

这里有一个关键约束：  
前端不能修改 `Komari` 官方源码，必须通过 `Komari` 的自定义 header 中执行 JavaScript 来完成页面能力注入。

我理解这意味着：

- 页面增强是“外挂式前端集成”，不是 fork 官方前端代码。
- 注入脚本本身要尽量薄，只负责识别页面、插入挂载点、触发后端资源加载或调用。
- 实际的 UI 结构、样式和较复杂的交互，尽量由独立后端提供，而不是把 CSS 和大段模板都塞进 header 注入代码里。
- 本服务应提供一种“生成可复制 header 配置”的产品能力，而不是让用户手写注入代码。

### 3.3 数据层

阶段 1 明确不使用数据库。  
因此数据层只需要满足“演示链路”和“按节点返回数据”这两个需求即可。

允许的方式包括：

- 内存存储
- 轻量缓存
- 启动时初始化一组模拟数据
- 按节点 UUID 动态拼装模拟响应

重点是接口和前端交互先成型，而不是存储系统先成型。

### 3.4 节点端

阶段 1 明确不涉及节点端 `IPQuality` 上报。

这条边界非常重要，因为它意味着当前阶段不需要做：

- 节点脚本执行调度
- 上报协议设计
- 节点身份鉴权
- 上报频率控制
- 上报失败重试
- 历史数据留存

这些都应视为后续阶段内容。

## 4. 当前阶段要验证的核心链路

我理解阶段 1 实际是在验证下面这条链路：

1. `Komari` 中存在节点对象。
2. 我们能从 `Komari` 服务端或相关接口中获取节点列表。
3. 节点详情页中可以识别“当前正在查看哪个节点”。
4. 前端能够拿着这个节点标识去调用新增接口。
5. 服务端能根据这个节点标识返回一份状态数据。
6. 前端能把状态数据显示出来。

如果这条链路跑通，说明后续即使把“模拟数据”替换为“真实上报数据”，总体架构也不会推倒重来。

因此阶段 1 的本质是：

- 验证集成点
- 验证数据关联键
- 验证页面注入与交互方式
- 验证后端接口可用性

并且这里的“集成点”现在已经更明确了：

- 后端通过配置连接 `Komari`
- 前端通过 `Komari` 自定义 header 注入少量 JS
- 展示资源尽量由独立服务提供

## 5. 我理解的系统角色

从业务上看，这个项目里至少存在以下几个角色：

### 5.1 Komari 平台

这是现有主系统，负责：

- 节点管理
- 节点详情页展示
- 节点基础数据体系
- 用户访问入口

本项目需要尽量贴着它的技术栈和结构来做，避免将来集成成本过高。

### 5.2 节点

每个节点有唯一 `UUID`。  
在项目完整形态中，节点将周期性地产出 IP 质量信息。

当前阶段中，节点的意义主要是“作为被展示对象”，以及“作为模拟数据映射的索引对象”。

### 5.3 IPQuality 脚本或采集逻辑

这是未来负责生成 IP 质量数据的来源。

当前阶段中，它不参与实现，但它决定了未来数据结构大概率会包含多个维度字段，而不是单一状态值。

### 5.4 我们新增的独立后端服务

这部分负责：

- 从 Komari 获取节点列表
- 对外提供节点 IP 质量查询接口
- 在当前阶段返回模拟状态
- 在后续阶段接入真实上报数据
- 管理与 `Komari` 的连接配置和认证配置
- 提供前端注入所需的资源入口或展示资源

### 5.5 独立 Web 前端

这部分负责：

- 提供本项目自己的配置页面
- 提供本项目自己的展示页面或展示资源
- 管理“显示什么、怎么显示”的配置
- 生成适用于 `Komari` 的 header 注入代码
- 提供“一键复制 Komari header”的操作

### 5.6 前端注入展示层

这部分负责：

- 通过 `Komari` 自定义 header 注入极少量 JavaScript
- 在 `Komari` 节点详情页增加交互入口
- 调用独立后端 API
- 加载或承接由独立后端提供的展示资源
- 把状态数据以弹窗形式展示

## 6. 节点 UUID 的重要性

现有资料中反复强调：每个 `Komari` 节点都有唯一 `UUID`。

我理解这个 `UUID` 是整个系统最关键的关联键，至少在阶段 1 是这样。  
原因如下：

1. 前端需要知道当前展示的是哪个节点。
2. 服务端需要知道针对哪个节点返回状态。
3. 模拟数据需要和节点一一对应。
4. 后续真实上报大概率也要把数据绑定到节点 UUID。

因此可以认为：

- 节点 UUID 是阶段 1 的主索引。
- 未来即使接入真实上报，UUID 大概率仍然会是核心关联字段。

## 7. 我理解的阶段 1 数据形态

文档没有给出固定 JSON 结构，但现在已经有了一个明确参考来源：`references/IPQuality/`。  
因此我理解阶段 1 的数据模型虽然可以先用模拟值，但字段组织应尽量贴近该库最终产出的结果形态。

结合当前资料，我理解阶段 1 的数据至少应具备以下特征：

1. 能唯一对应某个节点。
2. 能被前端弹窗直观展示。
3. 字段设计应尽量贴近未来真实数据，而不是只返回一句字符串。

也就是说，虽然现在允许模拟数据，但这份模拟数据最好已经具备“未来真实 IP 质量记录”的雏形。  
结合 `references/IPQuality/ip.sh` 当前表现出来的能力，未来数据模型很可能会覆盖以下几类信息：

- 节点 UUID
- 更新时间
- 风险评分结果
- 多个风险数据源的评分或等级
- 邮件可用性与黑名单检测结果
- 流媒体解锁状态
- 流媒体区域信息
- 流媒体解锁类型
- 可能的 IP 类型、地区或网络环境补充信息

从 `references/IPQuality/` 的现状看，至少已经能观察到这些业务维度：

- 风险评分来源：`Scamalytics`、`ipapi`、`AbuseIPDB`、`IP2Location`、`IPQS`、`DB-IP`
- 黑名单与邮件能力：黑名单数量、可用性、清洁度、标记情况
- 流媒体与 AI 服务：`TikTok`、`Disney+`、`Netflix`、`YouTube`、`Amazon Prime Video`、`Reddit`、`ChatGPT`

这里的重点不是现在立刻把全部字段完全定死，而是阶段 1 的模拟数据不要脱离 `IPQuality` 的真实输出逻辑太远。

## 8. 技术路线理解

文档中明确要求技术栈与 `Komari` 官方保持一致，并指出阶段 1 采用 `Go`。

我理解这背后的考虑有几层：

1. 便于后续与 Komari 的代码结构和调用方式对齐。
2. 降低阅读和复用官方实现时的认知成本。
3. 降低后续从“外挂式接口”演进为“深度集成模块”的迁移成本。

这意味着当前项目在技术选择上不应该随意偏离，例如：

- 不应为了快而单独换成完全不同的后端语言。
- 不应把阶段 1 写成与未来完全不可兼容的临时脚本集合。

即使阶段 1 允许模拟数据，工程方向仍然应围绕“后续可继续迭代”来组织。

另外，前端集成方式虽然是注入式，而不是修改官方源码，但这并不意味着实现可以随意。  
相反，它要求更强的工程约束：

- 注入代码要足够轻
- 选择器和挂载策略要稳
- 样式和模板不要散落在注入脚本中
- 后端要承担更多资源提供与展示承载职责
- 独立 Web 前端要承担配置管理和“生成嵌入能力”的职责

## 9. 与官方源码的关系

仓库中的 `references/komari-official/` 是 Komari 官方源码参考。

我理解它在当前项目中的作用是：

1. 参考 Komari 的目录结构、技术栈和接口组织方式。
2. 查找节点详情页前端代码位置。
3. 查找服务端节点相关 API 的实现方式。
4. 确定我们应该通过什么方式获取节点列表。
5. 判断是“外挂式服务”更合适，还是“直接基于 Komari 代码做集成”更合适。

它的作用是“参考”和“对齐”，不是当前项目的主代码目录。

另外，`references/IPQuality/` 则是另外一个同样关键的参考源，用于：

1. 理解节点侧到底检测哪些维度。
2. 推导后端未来要接收或组织的数据结构。
3. 让阶段 1 的模拟数据尽量贴近真实业务输出。
4. 避免前端展示模型和节点实际产出严重脱节。

## 10. 我理解的阶段 1 验收标准

现有文档中的阶段性验收标准可以拆解成更具体的检查项：

### 10.1 服务端验收

- 能成功访问 Komari 的节点列表能力。
- 能拿到节点标识，尤其是节点 UUID。
- 能提供一个查询节点状态的接口。
- 该接口能根据不同节点返回对应结果。
- 能通过配置而不是写死代码的方式连接 `Komari` 地址和认证信息。

### 10.2 前端验收

- 节点详情页能看到 `查看 IP 质量` 按钮。
- 点击按钮后会发起请求，而不是纯静态展示。
- 请求目标与当前节点对应。
- 弹窗能正常打开并展示数据。
- 全过程不修改 `Komari` 官方源码。
- 注入代码本身保持轻量，不把大量样式和模板硬编码在 header 中。

### 10.3 独立 Web 前端验收

- 本服务存在独立的配置与展示页面。
- 可以在本服务中配置需要展示的内容。
- 可以生成适用于 `Komari` 的 header 配置内容。
- 存在“一键复制 Komari header”能力，便于用户直接粘贴到 `Komari`。

### 10.4 集成验收

- 同一个节点多次查看，能得到一致且对应的模拟数据。
- 不同节点查看时，数据能区分开。
- 从页面交互到弹窗展示的链路是完整的。
- 后端与前端的职责边界清晰，注入层不承担大段 UI 逻辑。

### 10.5 开发环境验收

- 能本地启动 `Komari` 开发环境用于联调。
- 能本地启动独立后端服务。
- 能本地启动独立 Web 前端或其资源承载能力。
- 能验证 `Komari` 自定义 header 注入链路。
- 能在本地完整复现“节点详情页 -> 按钮 -> 弹窗 -> 后端接口”的流程。

## 11. 我理解的后续演进方向

虽然当前只做阶段 1，但从文档可以比较清楚地推测后续方向：

### 11.1 从模拟查询演进到真实上报

阶段 1 的服务端“按节点返回状态”接口，后续可以把模拟数据源替换成：

- 节点上报结果
- 缓存中的最近一次记录
- 数据库中的最新记录

### 11.2 从当前状态演进到历史记录

项目名中包含 `history`，这说明后续很可能不仅仅展示“当前状态”，还会涉及：

- 历史趋势
- 最近一次检测结果
- 每日记录
- 状态变化追踪

但这些内容不在阶段 1 范围内。

### 11.3 从简单展示演进到状态判断和告警

未来不仅可以展示原始数据，还可能进一步扩展为：

- 风险状态高亮
- 异常提示
- 变差提醒
- 解锁能力变化提示

不过这些也不属于当前最小链路目标。

## 12. 我理解的当前最重要设计原则

结合现有资料，我认为阶段 1 应遵循以下原则：

### 12.1 先跑通链路，不先做重型系统

阶段 1 明确不需要数据库，不需要真实上报，不需要复杂调度。  
因此最重要的是把“节点识别 -> 接口查询 -> 页面展示”跑通。

### 12.2 数据结构尽量向未来靠拢

虽然现在返回模拟数据，但字段组织不应过于随意。  
最好让模拟数据已经能代表未来真实数据的大致形态。

### 12.3 尽量沿着 Komari 的方式集成

技术栈和实现方式都应尽量参考官方，以便后续持续集成，不要做成短期看似快、长期难维护的旁路方案。

### 12.4 明确区分“当前阶段必须做”和“后续再做”

如果边界不清，阶段 1 很容易失控，提前卷入：

- 上报协议
- 数据库存储
- 鉴权方案
- 历史查询
- 告警逻辑

这些都会让阶段 1 偏离目标。

## 13. 我目前的理解结论

综合来看，我目前对项目的理解是：

这是一个围绕 `Komari` 节点体系扩展出来的 IP 质量展示能力。  
最终目标是让节点侧基于 `IPQuality` 产出 IP 质量信息，并在 `Komari` 中查看。  
本项目本身有独立的 Web 前端和独立后端，既可以独立配置和展示，也可以把展示能力嵌入到 `Komari`。  
你的核心设想不是让用户长期单独打开这个页面，而是像 `Grafana` 一类系统那样，先在本服务中配置好展示内容，再由本服务生成一段可复制的 `Komari header` 配置，用户将其填入 `Komari` 后，即可在 `Komari` 内部使用这套能力。

因此，当前阶段 1 的最小闭环应理解为：

1. 独立服务具备基础配置页和展示承载能力。
2. 独立后端服务通过配置连接 `Komari`，获取节点。
3. 本服务可以生成适合填入 `Komari` 的 header 内容。
4. `Komari` 通过自定义 header 注入少量 JavaScript。
5. 节点详情页出现入口按钮。
6. 点击后调用独立服务，并展示与节点 UUID 对应、且结构尽量贴近 `references/IPQuality/` 的模拟数据。

也就是说，阶段 1 不是在做完整产品，而是在验证整个集成方向成立：

- 节点可识别
- UUID 可关联
- 页面可注入
- 接口可调用
- 数据可展示
- 独立后端与 `Komari` 的连接方式成立
- “轻注入、重后端承载”的前端集成方式成立

只要这条链路成立，后续再把模拟数据替换成真实上报，再逐步引入缓存、数据库、历史记录和告警逻辑，就会顺很多。

## 14. 下一步应落地的内容

基于目前已经确认的方向，我理解后续应落地的内容会包括：

1. 独立后端的基础目录结构与配置系统，至少包含 `Komari` 地址和认证配置。
2. 面向节点状态查询的主流风格 API 路径设计。
3. 面向 `Komari` 自定义 header 的最小注入脚本方案。
4. 独立 Web 前端中的配置页、展示页与资源承载方式。
5. 由独立服务承载的弹窗页面、样式资源或前端资源组织方式。
6. “一键复制 Komari header” 的产品流程设计。
7. 参考 `references/IPQuality/` 提炼一版阶段 1 可用的数据模型。
8. 本地开发环境整理，包括 `Komari` 联调环境、独立后端运行方式、独立前端运行方式、注入调试方式和基础说明文档。

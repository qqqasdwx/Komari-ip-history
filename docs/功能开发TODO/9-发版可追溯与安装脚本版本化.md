# 第九步：发版可追溯与安装脚本版本化

## 目标

收口正式发版前的版本可追溯问题，确保用户部署某个固定版本镜像时，页面生成的节点接入命令也使用与该版本匹配的安装脚本，避免服务端版本和节点侧脚本来自不同代码版本。

## TODO

- [ ] 应用能明确知道当前运行版本，用于生成接入脚本地址和展示运行信息。
- [ ] 开发环境、`latest` 镜像和正式 tag 镜像能区分安装脚本来源。
- [ ] 正式版本生成的接入命令不再固定指向默认分支 `master`。
- [ ] 正式版本生成的接入命令应指向当前版本对应的安装脚本，或指向当前服务端提供的同版本安装脚本。
- [ ] 开发环境仍然可以方便地使用本地脚本替换方案做真实 Debian 验收。
- [ ] 页面上生成的接入命令能让用户看出当前脚本来源，避免误以为是另一个版本。
- [ ] 安装配置接口返回的脚本地址和页面展示保持一致。
- [ ] 节点侧真实安装流程继续支持首次安装、重复执行、目标 IP 更新、时区更新和立即执行。
- [ ] 发版说明补充固定版本镜像的推荐使用方式和升级注意事项。
- [ ] 双环境验收脚本覆盖版本化安装命令，不再只断言 `master` 地址。
- [ ] 固定 Debian 真实上报验收继续从页面生成命令并真实执行。

## 不包含

- 不在本步实际发布生产版本。
- 不修改真实 GitHub Release 或远端 tag。
- 不把本地双环境改造成生产部署方案。
- 不实现复杂的 beta、rc 或多发布通道体系。
- 不要求 GitHub Actions 自动跑完整 E2E 门禁。

## 验收清单

- [ ] 开发环境生成的接入命令仍可被固定 Debian 容器真实执行。
- [ ] 模拟正式版本时，页面生成的安装脚本地址不再是 `master`。
- [ ] 安装配置 JSON、页面接入命令、节点侧脚本实际下载地址三者一致。
- [ ] 重新运行安装命令不会破坏已有 cron、token、目标 IP 或时区配置。
- [ ] `verify-react-preview-nodes.mjs` 中关于安装脚本地址的断言已更新为版本化逻辑。
- [ ] `run-acceptance-e2e.sh` 能完整通过。
- [ ] `run-fixed-debian-ui-flow.sh` 能完整通过。
- [ ] 发版说明不再把安装脚本地址固定 `master` 列为未闭环限制。

## 用户体验变更

- 节点设置页生成的接入命令会体现当前版本的安装脚本来源。
- 用户使用固定版本镜像部署时，复制到节点上的安装命令与该镜像版本保持一致。
- 发版说明会更明确地区分 `latest` 快速试用和固定版本部署。

## 核心逻辑变更

- 安装脚本地址生成逻辑从固定 `master` 收口为版本感知。
- 服务端运行版本会参与安装配置生成。
- 自动化验收不再把默认分支地址当作正确结果，而是验证“当前运行版本和安装脚本来源一致”。

## 自动验收计划

- `go test ./...`
- `cd web && npx tsc --noEmit`
- `cd web && npm run build`
- `git diff --check`
- `docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/run-acceptance-e2e.sh`
- `bash deploy/dev/reporter/run-fixed-debian-ui-flow.sh`

## 手工验收建议

- 在本地开发环境打开节点设置页，确认接入命令仍可复制和执行。
- 模拟正式版本配置后打开节点设置页，确认安装脚本地址不再引用 `master`。
- 用固定 Debian 容器执行页面生成的接入命令，确认页面最终能看到真实上报数据。
- 查看发版说明，确认固定版本部署和升级注意事项能指导用户操作。

# 第十步：正式发版前验收与发布准备

## 目标

在 0-9 步功能完成后，做一次面向正式版本的总验收和发布准备，确认代码、镜像、安装脚本、文档和升级路径都具备打首个正式版本 tag 的条件。

## 状态

计划中。

## 不包含

- 不直接打正式 tag。
- 不创建 GitHub Release。
- 不修改 GitHub、GHCR、Cloudflare、VPS 或其它外部生产资源。
- 不变更功能范围，除非验收发现必须修复的发布阻断问题。
- 不新增大型功能，只做发版前修复、文档补齐和验收补强。

## 验收清单

- [ ] `go test ./...` 通过。
- [ ] `cd web && npx tsc --noEmit && npm run build` 通过。
- [ ] `git diff --check` 通过。
- [ ] `docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/run-acceptance-e2e.sh` 通过。
- [ ] `bash deploy/dev/reporter/run-fixed-debian-ui-flow.sh` 通过。
- [ ] 本地模拟正式版本镜像可以启动并通过 health check。
- [ ] 模拟正式版本镜像的 runtime 接口返回候选版本号和 commit。
- [ ] 模拟正式版本镜像的接入命令不引用默认分支 `master`。
- [ ] 真实 Debian 容器使用页面命令替换本地脚本后能上报真实数据。
- [ ] 默认主题和 PurCarte 下，已接入、未接入、空数据、游客允许、游客拒绝和登录桥接流程都可用。
- [ ] 发布说明能够指导用户选择固定版本镜像、升级和重新执行节点接入脚本。

## 用户介入点

- 需要用户确认候选版本号。
- 需要用户确认是否进入真实发版动作。
- 需要用户确认 release notes 中的面向用户变更描述。

## 自动验收计划

- `go test ./...`
- `cd web && npx tsc --noEmit && npm run build`
- `git diff --check`
- `docker compose -f compose.dev.yml up -d --build`
- `docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/run-acceptance-e2e.sh`
- `bash deploy/dev/reporter/run-fixed-debian-ui-flow.sh`
- 本地 `docker build` 注入候选 `IPQ_VERSION` 和 `IPQ_COMMIT` 后启动镜像做 smoke test。

## 手工验收建议

- 打开节点设置页，确认接入命令下方的脚本来源文案能被普通用户理解。
- 在 Komari 默认主题和 PurCarte 主题中分别打开 IP 质量入口，确认注入页适配当前主题。
- 阅读发版说明，确认固定版本部署和升级说明足够清晰。

# 协作与验收规则

## 基本原则

- 代码审查、测试、静态检查、必要的本地验证由 Codex 负责。
- 用户不负责审查代码实现细节。
- 只有涉及用户体验、页面交互、业务流程或需要人工确认的验收口径时，才需要用户介入。
- 每个功能分支应尽量只覆盖对应 TODO 文件中的功能，不夹带后续功能。
- 如发现实现风险、测试缺口或阻塞问题，Codex 应先自行处理；只有需要产品取舍时才交给用户判断。

## 需要告知用户的内容

当一个功能分支完成或需要用户验收时，Codex 需要明确说明：

- 哪些页面的交互逻辑发生变化。
- 哪些核心业务逻辑发生变化。
- 用户需要如何测试。
- 哪些场景已经由自动测试覆盖。
- 哪些场景仍建议用户手工验收。

## 不需要用户介入的内容

- 代码风格、重构方式、函数拆分、迁移实现细节。
- 单元测试怎么写。
- 静态检查、格式化、构建验证。
- 为修复实现问题所需的局部代码调整。
- 不影响体验和业务语义的内部实现修正。

## Codex 每步完成前必须自行确认

- 已做必要 code review。
- 已运行与改动范围匹配的测试。
- 已运行必要的静态检查或构建。
- 已确认没有无关改动混入。
- 已更新对应 TODO 状态或验收记录。

## Komari 注入页主题兼容

- 凡是改动 Komari 注入脚本、注入入口、弹窗、iframe 嵌入页或嵌入态样式，都必须把“是否适配当前 Komari 主题”作为验收范围。
- 注入页不能假设透明背景会继承 Komari 父页面。iframe 是独立文档，必须根据宿主主题上下文自行渲染或承接背景、卡片、边框、文字、按钮和状态样式，而不是只避免某个特定背景色。PurCarte 这类毛玻璃主题要保留卡片间隙的玻璃底效果；跨源 iframe 在深色透明场景会露出浏览器白画布时，必须用宿主传入的主题 canvas 承接，不能露白。
- loader 需要识别并传递足够的宿主主题上下文，至少包括默认主题/PurCarte、light/dark/system 的实际结果、主题色/强调色、卡片色、文字色、边框色和 PurCarte 玻璃参数。
- loader 和 iframe 嵌入页之间使用 `komari_theme_protocol=1` 作为主题协议标记。新增主题字段时必须同时更新 loader token map、React 嵌入解析 map、CSS 变量消费和 Playwright 协议断言，不能临时散落添加 URL 参数或样式覆盖。
- 自动测试和截图验收需要覆盖：默认主题 light、默认主题 dark、默认主题不同主题色、PurCarte light、PurCarte dark、PurCarte 不同卡片/颜色配置、已接入有数据、已接入无数据、未接入“去接入”、游客允许查看、游客禁止查看、IPQ 未登录桥接页。
- 验收不能只看 computed style；需要 Playwright 截图和必要的视觉/像素检查，确认嵌入内容在当前主题下整体一致：背景、文字对比度、卡片层级、边框、按钮、标签页和空/错/桥接状态都要符合宿主主题风格。

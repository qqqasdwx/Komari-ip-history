# 协作与验收规则

## 基本原则

- 代码审查、测试、静态检查、必要的本地验证由 Codex 负责。
- 用户不负责审查代码实现细节。
- 只有涉及用户体验、页面交互、业务流程或需要人工确认的验收口径时，才需要用户介入。
- 每个功能分支应尽量只覆盖对应 TODO 文件中的功能，不夹带后续功能。
- 如发现实现风险、测试缺口或阻塞问题，Codex 应先自行处理；只有需要产品取舍时才交给用户判断。

## 需要告知用户的内容

当一个功能分支完成或需要用户验收时，Codex 需要明确说明：

- 哪些页面的交互逻辑发生变化。
- 哪些核心业务逻辑发生变化。
- 用户需要如何测试。
- 哪些场景已经由自动测试覆盖。
- 哪些场景仍建议用户手工验收。

## 不需要用户介入的内容

- 代码风格、重构方式、函数拆分、迁移实现细节。
- 单元测试怎么写。
- 静态检查、格式化、构建验证。
- 为修复实现问题所需的局部代码调整。
- 不影响体验和业务语义的内部实现修正。

## Codex 每步完成前必须自行确认

- 已做必要 code review。
- 已运行与改动范围匹配的测试。
- 已运行必要的静态检查或构建。
- 已确认没有无关改动混入。
- 已更新对应 TODO 状态或验收记录。

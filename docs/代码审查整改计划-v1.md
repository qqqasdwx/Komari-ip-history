# 代码审查整改计划 v1

> 生成日期：2026-04-27
> 来源：`.omx/plans/ralplan-code-review-remediation-v1.md`
> 状态：Ralplan 共识已通过（Architect：CLEAR，Critic：APPROVE）
> 说明：本文是给项目维护者阅读的中文版整改计划，不包含源码实现。

## 1. 整改目标

本轮整改的目标是解决当前全项目 Code Review 中发现的阻塞级可靠性与维护性问题，同时保持改动范围可控、可测试、可回滚。

已经确认的基础验证命令此前可通过：

```bash
go test ./...
go vet ./...
npm --prefix web run build
(cd web && npx tsc --noEmit)
git diff --check
```

但 Code Review 仍指出以下需要整改的问题：

1. **通知规则存在孤儿引用风险**
   删除节点或目标 IP 后，通知规则里的关联范围可能残留，导致后台通知规则列表接口失败。

2. **JavaScript 通知渠道缺少同步执行上限**
   管理员配置的 JS 脚本如果包含死循环或长时间同步运算，可能卡住通知发送流程。

3. **Reporter 安装脚本在无本地候选 IP 时过早退出**
   即使服务端配置了手动目标 IP，本地扫描不到 candidate IP 时脚本也可能不会请求 `/plan`。

4. **运行时数据目录未完整忽略**
   双环境 DB、主题等运行时数据可能作为未跟踪文件出现在工作区。

5. **少量前端异步错误处理较脆弱**
   本轮仅处理与上述整改直接相关的部分，不做大规模 UI 重构。

## 2. 本轮不做什么

为避免整改范围失控，本轮明确不做以下事项：

- 不重做整套通知模型。
- 不拆分 `App.tsx`、`nodes.go`、`admin.go` 等大文件。
- 不重新讨论已确认“按设计接受”的项目权衡。
- 不做大规模前端架构重构。
- 不以数据库外键作为唯一修复手段。
- 不质疑前端安装命令默认依赖 GitHub raw main；这是刻意设计，目的是让用户能在 GitHub 上直观看到脚本内容。

## 3. 采用方案

采用 **方案 B：显式防御式修复**。

核心思路：

- 删除节点 / 目标 IP 时，显式清理相关通知规则范围。
- 应用启动或数据库打开时，修复历史遗留的孤儿通知范围数据。
- 通知规则读取路径保持一定容错，避免单条坏数据导致整个列表接口 500。
- JavaScript 通知执行增加硬超时 / 中断机制。
- 安装脚本在 candidate IP 为空时仍然请求服务端 `/plan`。
- 用回归测试覆盖每个阻塞问题。

不选择其它方案的原因：

| 方案 | 结论 | 原因 |
| --- | --- | --- |
| 只在删除路径做最小清理 | 不采用 | 无法修复历史遗留孤儿数据，读接口仍可能失败。 |
| 数据库外键 / schema 优先重构 | 暂不采用 | 范围较大，迁移风险高，不适合本轮阻塞问题修复。 |
| 大规模前后端拆分 | 暂不采用 | 不是当前阻塞点，会扩大验证成本。 |

## 4. 通知规则清理语义

本轮必须采用确定性策略，避免执行时临时猜测。

### 4.1 删除目标 IP 时

当一个 target 被删除：

1. 删除所有引用该 target 的 `NotificationRuleTargetScope`。
2. 如果某个非 `AllTargets` 的 node scope 因此没有任何 target scope，则删除该 node scope。
3. 如果某个非 `AllNodes` 的 notification rule 因此没有任何 node scope，则自动删除该 rule。

### 4.2 删除节点时

当一个 node 被删除：

1. 删除该节点下相关 notification target scope。
2. 删除引用该 node 的 notification node scope。
3. 如果某个非 `AllNodes` 的 notification rule 因此没有任何 node scope，则自动删除该 rule。

### 4.3 读取通知规则时

读取或列出通知规则时：

- 缺失 node / target 不应导致整个接口返回 500。
- 启动修复和删除路径清理是主修复手段。
- 读取路径可以防御性跳过异常子引用。
- 但不能把无效的空规则伪装成有效规则继续保留。

## 5. 分阶段整改计划

### 阶段 0：先补回归测试

先用测试锁住现有问题，避免修复过程中引入回归。

建议新增或调整：

- `internal/service/notification_test.go`
- `internal/service/nodes_test.go`
- `internal/service/nodes_install_test.go`
- `internal/service/notification_send_test.go`
- `internal/database/database_test.go`
- `internal/httpx/handlers/report_admin_test.go` 或等价后台 handler 测试

必须覆盖：

1. 删除 target 后通知 target scope 被清理。
2. 删除最后一个 target scope 后，无效的非 `AllNodes` rule 被删除。
3. 删除 node 后相关通知 scope / rule 被清理。
4. 含历史孤儿 scope 的数据库启动后会被修复，通知规则列表不再失败。
5. 同步死循环 JS 会在限定时间内中断并返回可识别错误。
6. `candidate_ips: []` 时，服务端 `/plan` 仍能返回启用的手动目标。
7. 生成安装脚本不会在请求 `/plan` 前因为空 candidate IP 直接退出。
8. `deploy/install.sh` 的运行时脚本块与生成安装脚本行为保持一致。

### 阶段 1：通知孤儿引用修复

改动范围：

- `internal/service/notification.go`
- `internal/service/nodes.go`
- `internal/database/database.go`
- 必要时补充小型 helper 文件

实施内容：

1. 增加通知规则范围清理 helper。
2. 在 `DeleteNodeTarget` 中删除 target 前调用清理逻辑。
3. 在 `DeleteNode` 中删除 node / node targets 前调用清理逻辑。
4. 在数据库 open / migration 后增加启动修复：
   - 删除 target 不存在的 target scope。
   - 删除 node 不存在的 node scope。
   - 删除非 `AllTargets` 且已无 target scope 的 node scope。
   - 删除非 `AllNodes` 且已无 node scope 的 rule。
5. 加固通知规则详情加载逻辑，避免孤儿子记录导致整页失败。

### 阶段 2：JavaScript 通知执行超时

改动范围：

- `internal/service/notification_send.go`
- `internal/service/notification_send_test.go`

实施内容：

1. 为 goja runtime 同步执行增加硬 deadline / interrupt。
2. 覆盖以下执行入口：
   - `vm.RunString(script)`
   - 同步 `sendEvent(event)`
   - 同步 `sendMessage(message, title)`
3. 保留当前 Promise 等待超时机制。
4. 正常完成时清理 timer，避免 goroutine / timer 泄漏。
5. 测试中断错误类型，不强依赖非常精确的耗时。

### 阶段 3：Reporter installer 空候选 IP 修复

改动范围：

- `internal/service/nodes.go`
- `deploy/install.sh`
- `internal/service/nodes_install_test.go`

实施内容：

1. `buildNodeInstallScript` 中取消“发现 0 个 candidate IP 就提前退出”的逻辑。
2. 无 candidate IP 时仍构造 `candidate_ips: []` 请求 `/plan`。
3. 只有 `/plan` 返回 0 个 approved targets 后，脚本才 no-op 退出。
4. `deploy/install.sh` 生成的运行时脚本块必须同步上述行为。
5. 保持 legacy inline `--target-ip` 模式兼容，不在本轮强行改变其语义。

### 阶段 4：运行时数据忽略

改动范围：

- `.gitignore`

实施内容：

1. `.gitignore` 忽略当前运行时数据目录。优先考虑：

   ```gitignore
   data/
   ```

   如果 `data/` 下存在需要纳入版本管理的文件，则改为精确忽略：

   ```gitignore
   data/ipq-default/
   data/ipq-purcarte/
   data/komari-default/
   data/komari-purcarte/
   __pycache__/
   *.pyc
   ```

2. 不在本阶段做 `App.tsx` 大拆分。

## 6. 验收标准

整改完成后必须满足：

1. 删除被通知规则引用的 target 或 node 后，`GET /api/v1/admin/notification/rules` 返回 200。
2. 非 `AllNodes` 的通知规则如果失去所有 node scope，会被删除，而不是保留为无效空规则。
3. 历史数据库中的孤儿通知 scope 会在启动 / open 时修复，或至少不会导致列表接口失败。
4. 同步死循环 / 长时间 JavaScript 通知脚本会被超时中断，不会卡死进程。
5. 生成安装脚本和 `deploy/install.sh` 都有证据证明：candidate IP 为空时仍会请求 `/plan`。
6. 服务端手动启用的目标 IP 在 `candidate_ips: []` 时仍能被批准安装。
7. 安装脚本只在 `/plan` 返回 0 个 approved targets 后才 no-op 退出。
8. 当前双环境运行时数据不再作为 untracked 文件出现。
9. 全量验证命令全部通过。

## 8. 风险与规避

| 风险 | 影响 | 规避方式 |
| --- | --- | --- |
| 通知清理误删用户仍需要的规则 | 中 | 只删除当前写入校验已无法表达的无效规则：非 `AllNodes` 且无 node scope。 |
| 读取路径容错掩盖数据问题 | 中 | 删除路径和启动修复作为权威修复；测试覆盖；不保留无效空规则。 |
| JS 中断影响正常脚本 | 中 | 保留正常同步 / Promise 脚本测试，只在超时后 interrupt。 |
| installer 修改破坏 legacy inline 模式 | 低到中 | 保持 legacy `--target-ip` 语义，增加兼容性证据。 |

## 9. 推荐执行顺序

1. 先补回归测试。
2. 实现通知清理与启动修复。
3. 实现 JS 通知执行超时。
4. 修复生成安装脚本与 `deploy/install.sh` 的空候选 IP `/plan` 流程。
5. 更新 `.gitignore`。
6. 跑完整验证。
7. 做最终 code review。

## 10. 可选执行方式

### 顺序执行

适合低协调成本、单人负责到底：

```text
$ralph execute .omx/plans/ralplan-code-review-remediation-v1.md
```

### 并行执行

适合希望更快推进，但需要严格控制文件所有权：

```text
$team 5 "Implement .omx/plans/ralplan-code-review-remediation-v1.md. Use lanes: executor A notification cleanup/startup repair excluding internal/service/nodes.go until serialized handoff; executor B JS timeout; executor C installer/.gitignore and owns internal/service/nodes.go generated-installer edits; test-engineer regression/parity evidence; verifier final evidence."
```

并行时要特别注意：`internal/service/nodes.go` 只能由一个 lane 串行持有，避免通知删除 hook 和 installer 修改产生冲突。

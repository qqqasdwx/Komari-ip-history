# 待确认事项

本文档记录当前仍未最终拍板、但会影响后续实现方式的问题。

---

## 1. 页内弹窗的具体实现路径

当前已经确认：

- 产品上优先争取页内弹窗
- 工程上保留独立页面兜底

但还没最终拍板的是：页内弹窗到底怎么实现。

当前可选方向大致有：

1. 轻量弹层 + 跨域请求数据 + 注入端自己渲染
2. 弹层内嵌远程页面
3. 轻量弹层只显示摘要，并提供“打开完整页面”按钮

这个问题需要后续根据真实浏览器行为和实现复杂度再确定。

---

## 2. 弹窗与独立页的降级策略

虽然独立页已经确定是兜底方案，但还需要明确：

- 是用户点击按钮后先尝试弹窗，失败再跳转
- 还是由配置决定“本实例默认用弹窗 / 默认用独立页”
- 还是弹窗只承载轻量内容，完整详情固定走独立页

当前方向倾向于：

- 先优先做弹窗
- 但不要让系统依赖弹窗才能成立

---

## 3. Header 中公开地址的最终配置方式

当前已确认：

- Header 默认不能再依赖同源 `/ipq/` 子路径前提
- 应支持基于本服务公开地址生成绝对地址

但还没完全定死的是：

- 是否必须强制用户填写 `IPQ_PUBLIC_BASE_URL`
- 如果没填，系统是否允许退回到“当前访问地址”
- 系统配置页里这块文案和交互如何设计

---

## 4. 节点详情页中节点信息的稳定提取方式

当前主接入方案依赖：

- 从 `Komari` 节点详情页获取节点 `uuid`
- 尽量获取节点名称

但这件事仍然属于实现敏感点，需要继续确认：

- 页面结构变化时是否稳定
- 不同版本 `Komari` 是否一致
- 是否需要准备多套选择器或兜底逻辑

---

## 5. 开发环境最终收口方式

当前方向已经调整为：

- Debian 原生日常开发
- Docker 用于联调 / E2E / CI

但文档和脚本层面还没有完全收口，后续仍需要确认：

- 是否补统一的本地开发脚本
- 是否补 `Makefile`
- 是否保留现有容器工作区脚本

---

## 6. 当前不再作为主待确认项的内容

以下方向目前不再作为当前主产品问题继续推进：

- 同域 `/ipq/` 子路径是否作为默认部署方式
- 是否继续扩展“历史变化 / 变化视图 / 趋势分析”

这些内容目前都不应重新回到主优先级。

# 待确认事项

本文档只记录**当前仍未拍板**、并且会直接影响后续实现方式的问题。

已经在 `docs/注入弹窗鉴权定稿.md` 中拍板的内容，不再回到本文件。

---

## 1. Header 中公开地址的最终配置方式

当前已确认：

- Header 默认不能再依赖同源 `/ipq/` 子路径前提
- 应支持基于本服务公开地址生成绝对地址

但还没完全定死的是：

- 是否必须强制用户填写 `IPQ_PUBLIC_BASE_URL`
- 如果没填，系统是否允许退回到“当前访问地址”
- 接入配置页里这块文案和交互如何设计

---

## 2. 节点信息的稳定提取方式

当前主接入方案依赖：

- 从 `Komari` 节点详情页获取节点 `uuid`
- 尽量获取节点名称

但这件事仍然属于实现敏感点，需要继续确认：

- 页面结构变化时是否稳定
- 不同版本 `Komari` 是否一致
- 是否需要准备多套选择器或兜底逻辑

---

## 3. 游客只读开关的后台入口与提示文案

当前已确认：

- 游客只读是**全局开关**
- 默认关闭
- 只影响 `Komari` 注入弹窗
- 关闭时游客点击只弹 `toast`

但还没完全定死的是：

- 这个开关最终放在哪个设置页
- `toast` 文案最终怎么写
- 是否需要在后台页补更明确的安全说明

---

## 4. 开发环境最终收口方式

当前方向已经调整为：

- Debian 原生日常开发
- Docker 用于联调 / E2E / CI

但文档和脚本层面还没有完全收口，后续仍需要确认：

- 是否补统一的本地开发脚本
- 是否补 `Makefile`
- 是否保留现有容器工作区脚本

---

## 5. 当前不再作为主待确认项的内容

以下内容本轮已经拍板，不再作为主待确认项：

- 注入弹窗的游客 / 管理员权限边界
- 是否继续依赖跨站 iframe 内管理员 cookie
- 是否通过 DOM 判断 `Komari` 管理员身份
- 游客链路是否复用管理员接口
- 详情接口是否继续默认带历史数据

以下方向目前也不再作为当前主产品问题继续推进：

- 同域 `/ipq/` 子路径是否作为默认部署方式
- 是否继续扩展“历史变化 / 变化视图 / 趋势分析”

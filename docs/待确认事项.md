# 待确认事项

本文档记录当前仍未最终拍板、需要在后续设计或实现前继续确认的问题。

## 1. 页面内弹窗可行性

- 当前倾向优先使用页面内弹窗，这是首选交互方案。
- 但尚未验证 `Komari` 当前页面结构、样式环境和注入脚本兼容性是否足够稳定。
- 如果页面内弹窗实现成本过高或兼容性差，需要退回独立页面方案。

需要后续确认的点：

- `Komari` 节点详情页是否存在稳定的挂载点。
- 页面路由切换时，注入脚本能否稳定感知当前节点变化。
- 弹窗样式是否能在不污染 `Komari` 页面样式的前提下稳定渲染。
- 弹窗内是否适合再承载“查看历史”入口。

## 2. 历史入口的最终位置

- 已确认历史能力后续要做成独立页面。
- 但阶段 1 中“历史入口”放在哪里尚未最终确定。

当前暂定规则：

- 若页内弹窗可行，优先考虑把“查看历史”入口放在弹窗内。
- 若页内弹窗不可行，则按独立页面主流程重新设计入口位置。

## 3. 当前状态摘要的视觉风格

- 节点列表中会显示“当前状态摘要”，但该摘要的具体视觉形态尚未最终确定。
- 目前已确认本服务不负责评估 IP 质量，只负责承载和展示结果，因此摘要样式不应过早写死。

尚未确定的点：

- 列表中摘要是标签块、单行文本，还是更轻的结构。
- 是否需要单独设计“无数据 / N/A / 有数据”的摘要状态。
- 是否需要在阶段 1 就做“变化过 / 未变化”的摘要提示。

## 4. 动态字段展示的具体交互

- 已确认阶段 1 不预先固定所有字段，展示配置应尽量动态。
- 已确认阶段 1 只做全局字段开关，不做按节点单独配置。
- 但“字段开关”的具体交互还没有完全定下来。

尚未确定的点：

- 字段开关页面是否按分组展示，例如风险、黑名单、流媒体、AI 服务、基础信息。
- 动态字段的显示名称、顺序、描述如何生成。
- 如果后续检测结果字段发生变化，配置页面是自动出现新字段，还是需要用户手动刷新/重新载入。

## 5. 注入脚本获取节点上下文的稳定性

- 目前已确认按钮点击时只向本服务发送 `uuid` 和节点名称。
- 但仍需要验证这些字段能否从 `Komari` 当前页面稳定获取。

尚未确定的点：

- `uuid` 在节点详情页前端上下文中的具体获取方式。
- 节点名称在页面中的稳定来源。
- 路由切换后是否需要重新解析页面上下文。

## 6. 独立前端与嵌入视图的边界

- 已确认本服务既有独立前端，又要嵌入 `Komari`。
- 但两者之间哪些页面共享、哪些页面单独存在，还未完全定死。

尚未确定的点：

- 当前状态详情在弹窗中展示的内容，是否与独立前端节点详情页复用同一套前端资源。
- “历史页面”是否完全独立于弹窗逻辑。
- 登录后默认节点列表页与 `Komari` 内查看节点详情之间的跳转关系如何设计。

## 7. 开发模式与生产模式的功能差异边界

- 已确认开发模式通过环境变量开启，默认生产模式。
- 已确认开发模式可为节点自动分配模拟结果。
- 但两种模式的行为边界还未彻底列清。

尚未确定的点：

- 生产模式下是否完全禁用模拟数据生成。
- 生产模式下如果没有真实数据，前端具体展示成什么样最合适。
- 开发模式下是否需要在页面中有明确标记，避免误解为真实数据。

## 8. “IP 质量变化”功能进入阶段 2 还是 1.5

- 你已明确说明这个项目的核心目的之一是记录 IP 质量变化。
- 但当前也明确要求阶段 1 不要把这部分做复杂。

尚未确定的点：

- 阶段 1 结束后，下一步是先补真实数据链路，还是先补变化记录能力。
- “变化”的定义是按原始 JSON 差异、按关键字段差异，还是按用户选择关注字段差异。
- “变化”展示最终是列表提示、详情页高亮，还是独立事件流。

# Debian 原生开发环境迁移方案

> 注意：本文件是从“Docker 唯一开发方式”转向“双轨开发方式”时的专题方案文档。  
> 其中关于 `/ipq/` 子路径、旧环境变量和早期联调假设的内容，已不完全代表当前默认口径。  
> 当前有效口径请优先查看：
> - `docs/开发环境.md`
> - `docs/文档索引.md`

## 结论

建议采用“双轨制”：

- **Debian 原生开发**：作为日常开发主方案
- **Docker Compose**：作为联调、E2E、CI、演示环境保留

也就是说：

- 平时改 Go、前端、样式、接口：直接在 Debian 上跑
- 需要验证 Komari 注入、反代、子路径、完整链路：再用 Docker

---

## 为什么现在不必再把 Docker 当成唯一开发环境

之前在 Windows 上强依赖 Docker，主要是因为：

- 路径和权限问题多
- Go / Node 环境不统一
- 本机构建和脚本执行容易出问题
- 用容器做统一入口更省事

迁到 Debian 之后，这些问题大多会明显减轻：

- Go、Node、npm、Playwright 原生跑通常更自然
- 文件权限、脚本执行、软链接、监听行为都更稳定
- 不需要每次 `docker exec` 进去开发，反馈速度更快
- 排查问题更直接

所以继续把：

- `go build`
- `go test`
- `npm ci`
- `npm run dev`
- `npm run build`

全部强行塞进容器里，收益已经没有之前那么大了。

---

## 为什么又不建议完全删掉 Docker

这个项目不是单纯的本地单服务开发，而是涉及：

- Komari
- 本服务
- 反向代理
- `/ipq/` 子路径
- Header 注入
- 同源行为

这些内容仍然很适合用 Docker 固化成一套标准联调环境。

Docker 继续保留，主要价值在于：

1. **Komari 联调**
2. **统一入口反代**
3. **注入链路验证**
4. **Playwright / E2E**
5. **CI 可复现**
6. **一键演示环境**

所以结论不是“删 Docker”，而是“降低 Docker 的日常开发占比”。

---

## 建议的新开发模式

## 模式 A：Debian 原生日常开发

作为主推荐方案。

用途：

- 改页面
- 改接口
- 调样式
- 改渲染逻辑
- 跑构建和测试
- 快速迭代

建议方式：

- 后端：本机直接运行
- 前端：本机直接运行 Vite
- 数据库：本地 SQLite
- Komari：按需本机跑，或者继续放在容器里跑

---

## 模式 B：Docker 联调 / 验证环境

作为保留方案。

用途：

- 验证 `/ipq/` 子路径
- 验证 Komari 注入
- 验证统一入口反代
- 跑 E2E
- 跑 CI
- 做演示环境

建议继续保留：

- `compose.dev.yml`
- 现有 Playwright 流程
- GitHub Actions 对容器环境的依赖

---

## 推荐调整目标

目标不是推翻现有开发环境，而是把它从：

> Docker 是唯一开发方式

改成：

> Debian 原生是日常开发主方式，Docker 是联调和验证方式

---

## 需要做的改造

## 1. 文档改造

当前文档偏向“所有开发都在容器里完成”。

建议调整为两部分：

### A. Debian 原生开发

写清楚：

- 需要安装哪些依赖
- 怎么启动后端
- 怎么启动前端
- 怎么跑测试
- 怎么跑构建
- SQLite 路径怎么配置

### B. Docker 联调环境

写清楚：

- 什么情况下用 Docker
- 怎么启动 Komari + proxy + 联调环境
- 怎么跑 Playwright
- 怎么做完整链路验证

---

## 2. 后端原生启动能力

目标：

- 不进容器也能直接启动 Go 服务

建议补齐：

- 本地开发环境变量示例
- 本地默认 SQLite 路径
- 本地启动脚本或 Make 命令

建议默认配置：

- `IPQ_APP_ENV=development`
- `IPQ_BASE_PATH=/ipq`
- `IPQ_DB_PATH=./data/ipq/ipq-dev.db`
- `IPQ_LISTEN=127.0.0.1:8090`

---

## 3. 前端原生启动能力

目标：

- 本机直接跑 Vite 开发服务器

建议补齐：

- `npm ci`
- `npm run dev`
- 前端开发环境变量说明
- `VITE_BASE_PATH=/ipq/`

建议默认监听：

- `127.0.0.1:5173`

---

## 4. 联调方式分层

建议把联调分成两档：

### 轻量联调

用于日常开发：

- 前端本机跑
- 后端本机跑
- 直接调接口
- 主要解决页面和业务逻辑开发

### 完整联调

用于验证集成：

- 启动 Docker Compose
- 带上 Komari、proxy、子路径、注入链路
- 跑完整浏览器流程

这样就不需要每次改一个样式都开整套容器。

---

## 5. Docker 的新定位

Docker 不再承担“唯一开发入口”角色，而是主要承担：

- Komari 联调
- 子路径与反代验证
- 注入链路验证
- E2E
- CI
- 演示环境

也就是说，以下操作不再强制要求进容器：

- `go build`
- `go test`
- `npm ci`
- `npm run build`
- `npm run dev`

---

## 6. 脚本层优化建议

为了减少切换成本，建议补一组原生开发脚本，例如：

- `scripts/dev-backend.sh`
- `scripts/dev-frontend.sh`
- `scripts/build.sh`
- `scripts/test.sh`

如果愿意，也可以补一个 `Makefile`，例如：

- `make dev-backend`
- `make dev-frontend`
- `make build`
- `make test`
- `make e2e`

这样以后开发、验证、回归命令会更统一。

---

## 7. 数据目录建议

建议原生开发继续沿用当前目录语义：

- `./data/ipq/`：本服务 SQLite 数据
- `./data/komari/`：Komari 数据（如果仍需要）

这样可以保证：

- 原生模式和 Docker 模式目录结构一致
- 文档更统一
- 数据迁移和排查更简单

## 建议实施步骤

## 第一步：先不删 Docker

只调整定位，不破坏现有可用链路。

保留：

- `compose.dev.yml`
- 现有 CI
- 现有 Playwright 验证方式

---

## 第二步：补齐 Debian 原生启动能力

目标：

- 后端本地能跑
- 前端本地能跑
- 本地 build / test 能跑

产出：

- 本地 env 示例
- 启动脚本
- 开发文档

---

## 第三步：补充轻量联调说明

目标：

- 页面开发不必每次拉整套容器
- 只在需要时进入完整联调环境

产出：

- 本机开发方式说明
- Komari / proxy 何时需要启动的说明

---

## 第四步：保留 Docker 用于完整验证

目标：

- CI 不受影响
- E2E 不受影响
- Komari 注入链路仍有标准验证环境

---

## 验收标准

如果按这个方案落地，我认为验收标准应为：

### 原生开发

1. Debian 上可直接执行：
   - `go build ./cmd/ipq`
   - `go test ./...`
   - `cd web && npm ci && npm run build`
2. 后端可本地直接启动
3. 前端可本地直接启动
4. SQLite 可正常创建和读写

### Docker 保留

1. `compose.dev.yml` 仍可正常启动
2. Komari 注入链路不回归
3. Playwright 仍可运行
4. GitHub Actions 无需大改

---

## 风险点

主要风险有三个：

### 1. 子路径 `/ipq/`

原生直跑时，和完整反代环境可能表现不完全一致。

所以子路径相关问题，仍然要靠 Docker 联调验证。

### 2. Komari 注入链路

这条链路天然更适合在完整环境里测。

不建议完全脱离 Compose。

### 3. 文档分叉

如果原生和 Docker 两套说明都写得很散，后面会越来越难看懂。

所以文档必须明确主次：

- 日常开发：原生优先
- 完整验证：Docker

---

## 最终建议

我建议的最终决策是：

- **保留 Docker**
- **不再把 Docker 当成唯一开发方式**
- **把 Debian 原生开发改成主推荐方案**
- **把 Docker 调整为联调 / 验证 / CI 专用方案**

这样改动最小，但收益最大。

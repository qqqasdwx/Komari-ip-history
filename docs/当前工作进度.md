## 3.6 最新嵌入链路验证结论

本轮已经额外做了一次“跨域独立部署”方向的最小本地复现实验，验证结果很明确：

### 已验证结论

1. **当前 loader 的跨域状态接口请求会被 CORS 拦截**
   - loader 当前会先请求：
     - `/api/v1/embed/nodes/:uuid/status`
   - 在跨域场景下，如果后端不返回 `Access-Control-Allow-Origin` 等 CORS 头，浏览器会直接拦截

2. **当前 iframe 本身不是第一阻塞点**
   - 单独直接打开跨域 iframe 时，页面本身可以正常加载
   - 当前真正先失败的是“弹窗打开前的状态 fetch / register fetch”

### 这次验证的意义

这说明当前链路里最先需要解决的，不是“iframe 能不能跨域显示”，而是：

- loader 是否继续依赖跨域 `fetch`
- 如果继续依赖，就必须补 CORS 策略
- 如果不想引入 CORS 复杂度，就要调整注入侧交互路径

### 当前结论

因此目前的工程判断可以进一步收口为：

- **页内弹窗目标仍然保留**
- **iframe 方案暂时不必立刻推翻**
- **但当前 loader 的跨域 API 访问方式不能直接照现状进入真实独立部署场景**

## 3.11 已新增应用域内 `connect` 接入页

本轮又推进了一个对主链路很关键的改动：

- 新增 React 路由：
  - `/#/connect?uuid=...&name=...`
- loader 也已改成：
  - 在无法直接确认状态或需要接入时，优先打开 `connect` 页
- 这个页面的职责是：
  - 在**应用域内**完成节点注册
  - 成功后跳转到对应节点详情页
  - 支持 `embed=1`，可直接用于弹窗 iframe

同时还补了登录回跳逻辑：

- 未登录访问 `connect` 时，会先跳到：
  - `/#/login?redirect=...`
- 登录成功后会自动回到原本的 `connect` 目标
- 再继续完成注册和跳转

这一步的意义很大：

- 注入端不再必须先完成跨域 `register`
- loader 在跨域环境下即使拿不到状态，也仍然可以继续把用户送进一条可工作的主链路
- “接入动作后移到应用域”已经不只是文档想法，而是有了真实页面落点

本轮已确认：

- 已登录状态下直接访问 `connect`：
  - 可完成注册
  - 可跳到对应节点详情
- 未登录状态下访问 `connect`：
  - 会先跳登录
  - 登录后可正确回到 `connect`

# 当前工作进度

更新时间：2026-04-02

## 1. 当前状态概览

当前仓库不是从零开始，而是已经具备一套可运行的主链路：

- Go 后端
- React 前端
- 管理员登录
- 节点列表 / 节点详情
- Header 生成
- `embed=1` 弹窗详情页
- Docker 联调环境

但需要注意：

- 代码里仍混有一部分旧方向实现
- 文档和代码直到本轮前还没有完全对齐
- 当前真正需要推进的，已经从“样式整理”转成了“鉴权和接口边界收口”

---

## 2. 当前已经确认的方向

目前已经拍板的关键结论是：

1. 默认按**跨域独立部署**设计
2. `/ipq/` 子路径只保留兼容能力，不再是默认前提
3. 前端主线是 **React**
4. 产品体验上仍然优先争取**页内弹窗**
5. 工程上必须保留**独立页面兜底**
6. 当前重点是**当前 IP 质量结果**
7. “历史变化 / 变化视图 / 趋势分析”不再作为当前主产品方向
8. 开发环境采用：
   - Debian 原生日常开发
   - Docker 联调 / E2E / CI

另外，本轮又补齐了一个新的架构级结论：

- 注入弹窗鉴权以 `docs/注入弹窗鉴权定稿.md` 为准

---

## 3. 当前代码已经具备的能力

### 3.1 前端与部署口径

- 默认前端入口已经切到 React
- 默认部署入口已经切到根路径 `/`
- Header 生成支持独立部署地址
- Compose / CI / Playwright 的默认应用入口也已经从 `/ipq/` 收口到根路径

### 3.2 当前结果展示

- 节点详情页和 `embed=1` 详情页都已经可用
- 详情页整体展示已按 IPQuality 风格收口
- 缺失字段可显示 `N/A`
- 弹窗嵌入页样式也已做过一轮压缩和细修

### 3.3 注入链路基础能力

- loader 可注入节点详情页按钮
- loader 可打开页内弹窗
- `iframe` 可加载 `embed=1` 详情页
- 独立页登录和 `connect` 过渡页能力都已经存在

---

## 4. 当前实现与最新方向之间的主要偏差

### 4.1 鉴权模型仍然是旧实现导向

虽然弹窗骨架已经存在，但当前代码主链路还明显带着旧假设：

- 试图在嵌入链路里完成更多登录动作
- `connect` 页承担了过多统一入口职责
- 游客与管理员链路没有真正拆开

而当前最新结论已经变成：

- 游客只读走匿名公开接口
- 管理员链路走独立页登录 / 接入 / 回跳
- 不再把“跨站 iframe 中复用管理员 cookie”作为主方案

### 4.2 详情接口仍然背着历史数据

当前产品方向已经明确：

- 详情只看当前状态
- 历史走独立接口

但代码侧这件事还没真正拆开。

### 4.3 尚未接入 Komari 身份判断

当前产品已经明确：

- 不依赖 DOM 猜管理员身份
- 以 `Komari` 接口结果为准

但代码还没按这个方向真正接进去。

### 4.4 游客只读链路还没落地

当前还没有：

- 游客只读全局开关
- `/public/...` 匿名只读接口
- 基于 `Komari` 节点公开状态的游客分流
- “管理员未开放该功能”的 toast 分支

### 4.5 loader 性能问题仍在

之前审查里提到的 loader 全局扫描 / `MutationObserver` 偏重问题，当前还没处理。

---

## 5. 当前对 `connect` 页的定位

代码里现有的 `/#/connect?uuid=...&name=...` 仍然有价值，但它的定位已经变化：

- 它不再应被视为游客和管理员统一共用的唯一主入口
- 更合理的定位是：
  - 管理员登录 / 接入 / 回跳链路中的过渡页

也就是说：

- 这条能力可以保留
- 但其产品地位已经从“总主链路”下降为“管理员支线能力”

---

## 6. 当前更合理的下一步

文档整理到这里已经基本够用，后面重点应转回实现。

更合理的顺序是：

1. 先接入 `Komari` 身份判断接口
2. 再拆：
   - 管理员接口
   - 游客匿名只读接口
3. 再把详情接口和历史接口分开
4. 最后把 loader / `connect` / 回跳链路按新的状态机收口

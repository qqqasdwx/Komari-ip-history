# Komari 集成方式重定稿

## 1. 这份文档的目的

这份文档用于固化当前项目的集成前提。

它纠正的不是“要不要页内弹窗”，而是之前那个已经不成立的假设：

- 默认同域
- 默认子路径
- 默认跨域问题可以先不管

当前项目真正要成立的前提是：

> 用户只需要部署 IPQ 服务，复制一段 header 到 `Komari`，就能开始使用。

因此本项目默认应按：

> **跨域独立部署 + Header 入口桥接**

来设计。

---

## 2. 新的默认前提

### 2.1 本服务独立部署

- 不要求与 `Komari` 同域
- 不要求与 `Komari` 同机
- 不要求用户额外配置统一入口反代
- `/ipq/` 子路径只保留兼容能力，不再是默认前提

### 2.2 Header 只做桥接

Header 注入代码职责应尽量收缩，只做：

1. 在 `Komari` 节点详情页识别挂载位置
2. 从当前页面提取节点 `uuid` 和节点名称
3. 插入按钮
4. 根据当前身份和节点状态分流

不让 header 默认承担：

- 复杂鉴权闭环
- 大量展示逻辑
- 管理后台能力

---

## 3. 集成目标

新的集成目标应当是：

1. 用户部署本服务
2. 用户在本服务后台中生成一段 `Komari header`
3. 用户把 header 粘贴到 `Komari`
4. `Komari` 节点详情页出现入口按钮
5. 用户点击按钮后：
   - 优先在 `Komari` 当前页打开弹窗
   - 必要时跳到 IPQ 独立页完成登录 / 接入 / 回跳

重点在于：

- `Komari` 是入口宿主
- IPQ 是业务承载者
- 两者默认独立部署

---

## 4. 地址生成方式

### 4.1 默认使用绝对地址

Header 中引用本服务资源时，应默认基于本服务公开地址生成绝对 URL，例如：

- `https://ipq.example.com/...`

而不是默认依赖：

- 当前站点同源地址
- 相对路径
- `/ipq/...`

### 4.2 本服务需要明确公开访问地址

系统配置中应明确存在：

- 本服务公开访问基地址

例如：

- `IPQ_PUBLIC_BASE_URL`

这个地址用于：

- 生成 header
- 生成跳转链接
- 生成弹窗和回跳地址

---

## 5. 接入与查看流程重定稿

建议把注入链路明确拆成两条：

### 5.1 游客链路

1. 用户在 `Komari` 节点详情页点击按钮
2. header 通过 `Komari` 接口判断当前身份是游客
3. 再判断：
   - IPQ 游客只读全局开关是否开启
   - 当前节点在 `Komari` 中是否对游客公开
4. 满足条件时：
   - 直接打开页内弹窗
   - 弹窗里走匿名只读详情接口
5. 不满足条件时：
   - 不打开弹窗
   - 直接给出提示

### 5.2 管理员链路

1. 用户在 `Komari` 节点详情页点击按钮
2. header 通过 `Komari` 接口判断当前身份是管理员
3. 若节点已接入且 IPQ 已有有效登录态：
   - 直接打开完整详情弹窗
4. 若节点未接入，或 IPQ 尚未登录：
   - 跳转到 IPQ 独立页面
   - 在独立页完成登录 / 接入
   - 登录成功后继续之前的操作
   - 接入成功后优先回到 `Komari` 弹窗详情

这个流程的关键点是：

- 入口来自 `Komari`
- 业务闭环主要在本服务完成
- 不再把“跨站 iframe 中复用管理员 cookie”当成主方案

---

## 6. 页内弹窗与独立页的关系

### 6.1 产品目标

本项目的产品目标仍然是：

- `Komari` 页面内弹窗查看 IP 质量

因为这更符合项目初衷：

- 用户不想长期单独开一个新页面
- 主要体验仍然希望发生在 `Komari` 内

### 6.2 工程策略

更准确的表述应当是：

> **产品上优先做页内弹窗，工程上保留独立页兜底。**

但这里的“兜底”不再只是“弹窗失败了再打开页面”，还包括：

- 管理员未登录时，独立页负责登录
- 节点未接入时，独立页负责接入
- 完成后再优先回到弹窗

### 6.3 当前不再采用的思路

当前不再把以下方案当主实现：

- 在跨站 iframe 中长期复用管理员 cookie
- 依赖同域 `/ipq/` 子路径前提
- 依赖 DOM 猜测 `Komari` 管理员身份

---

## 7. 跨域下的设计原则

### 7.1 不让注入脚本承担复杂鉴权

默认不应依赖在 `Komari` 页面里直接完成复杂跨域业务流程。

更合理的默认模型是：

- `Komari` 负责提供入口和身份环境
- IPQ 负责业务与展示

### 7.2 管理员登录只属于 IPQ 自身域名

更合理的默认模型是：

- `Komari` 有自己的登录态
- IPQ 有自己的登录态
- 两者独立
- 管理员高风险动作放到 IPQ 独立页完成

### 7.3 游客链路允许只读公开

游客只读不复用管理员能力。

它应该满足：

- 默认关闭
- 全局开关控制
- 跟随 `Komari` 节点是否公开
- 只返回当前结果所需最小数据

---

## 8. 对现有产品设计的影响

如果按这份重定稿执行，则现有产品设计需要做以下修正：

### 8.1 仍然成立的部分

- 本服务是独立服务
- 不修改 `Komari` 官方源码
- 通过 header 注入实现入口接入
- 单管理员模型
- 节点由用户主动从 `Komari` 页面添加
- 展示重点是“当前 IP 质量结果”

### 8.2 需要修正的部分

- “默认同域部署”
- “默认 `/ipq/` 子路径部署”
- “跨站 iframe 中直接完成管理员登录”
- “游客与管理员共用同一条详情接口”

建议统一改成：

- 默认跨域独立部署
- 默认绝对地址生成 header
- 游客链路与管理员链路分开
- 页内弹窗优先，独立页兜底

---

## 9. 阶段 1 推荐主链路

建议阶段 1 的主链路收口为：

1. 部署本服务
2. 配置本服务公开地址
3. 复制 header
4. 粘贴到 `Komari`
5. 在 `Komari` 节点详情页看到入口按钮
6. 点击按钮后，先通过 `Komari` 接口判断当前身份
7. 若为游客：
   - 根据游客只读全局开关和节点公开状态决定：
     - 打开匿名只读弹窗
     - 或直接提示未开放
8. 若为管理员：
   - 已登录且已接入：直接打开完整详情弹窗
   - 未登录或未接入：跳独立页完成登录 / 接入，再优先回到弹窗

这条链路应当在“跨域、独立部署、无需统一入口”的前提下成立。

---

## 10. 最终结论

本项目后续应以以下原则为准：

### 核心原则

- **默认独立部署**
- **默认跨域场景成立**
- **Header 只做入口桥接**
- **页内弹窗优先，独立页面兜底**
- **管理员高风险动作不在跨站 iframe 中完成**

### 实现原则

- 默认绝对地址生成 header
- 默认按游客链路 / 管理员链路分流
- 游客只读接口与管理员接口分离
- 管理员登录 / 接入走独立页并支持回跳

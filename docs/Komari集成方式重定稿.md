# Komari 集成方式重定稿

## 1. 这份文档的目的

这份文档用于纠正之前讨论中一个已经被证明不成立的前提：

- 之前默认把本服务设计成与 `Komari` **同域 / 子路径部署**
- 例如：
  - `https://example.com/` -> `Komari`
  - `https://example.com/ipq/` -> 本服务

这个前提虽然在技术上可行，但它会额外要求用户：

- 具备统一入口反代能力
- 额外配置 `Nginx / Caddy / Traefik`
- 处理路径分流
- 把两个原本独立的服务强行绑到同一个入口

这与本项目原始设想不一致。

本项目真正希望达成的使用方式应当是：

> 用户只需要找一台机器部署本服务，拿到一个可访问地址，在 `Komari` 中粘贴 header，即可开始使用。

因此，本项目的默认架构需要从：

> **同域嵌入式集成**

调整为：

> **跨域独立部署 + Header 入口桥接**

---

## 2. 新的默认前提

从现在开始，默认按以下前提设计：

### 2.1 本服务是独立部署的

- 本服务部署在独立地址
- 不要求与 `Komari` 同域
- 不要求与 `Komari` 同一台机器
- 不要求与 `Komari` 共用同一个反向代理入口

用户可以自由选择：

- 独立域名
- 子域名
- 独立端口
- 独立服务器

例如以下形式都应视为正常部署方式：

- `https://ipq.example.com`
- `https://my-ipq.example.net`
- `http://1.2.3.4:8090`

---

### 2.2 不再把“同域部署”作为默认要求

之前“同域部署、暂不考虑跨域问题”的假设，不再作为默认要求。

原因是浏览器判断是否跨域，看的是完整 `origin`：

- 协议
- 主机名
- 端口

只要其中任一项不同，就是跨域。

因此即便是：

- `komari.example.com`
- `ipq.example.com`

它们仍然是跨域，不应该继续把“主域名相同”误认为“没有跨域问题”。

---

### 2.3 不再把 `/ipq/` 子路径作为主部署模型

`/ipq/` 子路径方案不是错误方案，但它不再是本项目默认主方案。

原因是它会让用户多做很多部署层工作，而这不是本项目想给用户增加的门槛。

从现在开始：

- **默认主方案**：独立地址部署
- **可选兼容方案**：同域子路径部署

也就是说：

- 我们仍然可以支持 `/ipq/`
- 但不能再把它当成唯一前提去设计系统

---

## 3. 新的集成目标

新的集成目标应当是：

1. 用户部署本服务
2. 用户登录本服务后台
3. 用户在本服务中复制一段 `Komari header`
4. 用户把 header 粘贴进 `Komari`
5. `Komari` 页面中出现“添加 IP 质量检测 / 查看 IP 质量”入口
6. 点击入口后，跳转或打开本服务页面完成后续动作

重点在于：

- `Komari` 只是入口宿主
- 本服务才是业务承载者
- 两者默认独立部署
- 不要求用户额外做统一入口整合

---

## 4. Header 的职责重定义

Header 注入代码的职责应当尽量收缩，只承担桥接作用。

建议职责范围如下：

1. 在 `Komari` 节点详情页识别合适挂载位置
2. 从当前页面提取节点 `uuid` 和节点名称
3. 插入按钮或入口
4. 点击后跳转到本服务

不建议让 header 默认承担以下职责：

- 直接在 `Komari` 页面内完成复杂业务流程
- 直接承担大规模数据展示
- 直接承担复杂鉴权逻辑
- 强依赖与本服务同源
- 依赖 `/ipq/` 子路径存在

也就是说，header 应尽量薄，只做：

> **识别当前节点 -> 提供入口 -> 把用户带到本服务**

---

## 5. 接入流程重定稿

建议把节点接入流程明确为：

1. 用户在 `Komari` 节点详情页点击 `添加 IP 质量检测`
2. header 从当前页面拿到节点 `uuid` 和节点名称
3. 浏览器打开本服务地址，例如：
   - `/connect?uuid=...&name=...`
4. 如果用户尚未登录本服务，则先在本服务域名下完成登录
5. 登录后，由本服务完成节点接入
6. 节点接入按幂等处理

这个流程的关键点是：

- 入口来自 `Komari`
- 业务完成在本服务
- 不要求 `Komari` 与本服务同源
- 不要求从 `Komari` 页面直接跨域调用本服务 API

---

## 6. 查看详情的方式重定稿

本项目的产品目标仍然应优先追求：

- `Komari` 页面内弹窗查看 IP 质量

因为这更符合项目初衷：

- 用户不想长期单独开一个新页面
- 希望主要体验仍然发生在 `Komari` 内
- Header 注入的价值，本来就是尽量减少来回跳转

因此这次纠偏并不是要否定“页内弹窗”本身，而是要修正另外一个错误前提：

- **页内弹窗不能再建立在“默认同域、默认无跨域问题、默认子路径部署”这些前提上**

更准确的表述应当是：

> **产品上优先做页内弹窗，工程上保留独立页兜底。**

---

### 6.1 推荐策略

建议阶段 1 按以下策略实现：

#### 产品默认目标

- 点击 `查看 IP 质量`
- 优先在 `Komari` 当前页面打开弹窗
- 在弹窗中展示当前 IP 质量核心内容

#### 工程兜底目标

- 如果弹窗方案在跨域、登录态、浏览器兼容性上不稳定
- 则自动或明确降级为打开本服务独立页面

例如：

- `/nodes/:uuid`

也就是说：

- **弹窗是优先体验**
- **独立页是稳定兜底**

而不是把两者理解为互相排斥。

---

### 6.2 为什么不能只押注弹窗

如果本服务和 `Komari` 默认是跨域独立部署，那么页内弹窗在实现时容易遇到：

- iframe 限制
- `X-Frame-Options`
- `CSP`
- 第三方 Cookie
- 跨域通信
- 宿主页面样式与交互冲突

所以不能把系统做成：

- 只有弹窗能用
- 一旦弹窗链路不稳，整个产品就不可用

更合理的工程思路是：

- 优先验证并争取做成弹窗
- 但从架构上确保“就算弹窗失败，产品主链路依然成立”

---

### 6.3 阶段 1 的实现优先级

我建议阶段 1 的优先级写成：

#### 第一优先级：验证页内弹窗是否可行

先验证：

- 注入按钮能否稳定打开弹层
- 弹层能否稳定挂载
- 跨域情况下能否稳定获取和展示最小必要内容
- 登录态是否可处理

#### 第二优先级：让弹窗承载“当前 IP 质量核心内容”

如果验证可行，则弹窗承载：

- 当前状态总览
- 重点结果
- 必要按钮

#### 第三优先级：保留独立详情页兜底

如果弹窗表现不稳定，或者只适合承载轻量内容，则：

- 弹窗中保留“打开完整页面”
- 或直接退回独立页面方案

---

### 6.4 结论

因此本项目在“查看详情方式”上的最终结论应当是：

- **产品目标：页内弹窗优先**
- **工程策略：独立页兜底**
- **不能再把“同域 / 子路径”当成页内弹窗成立的前提**
- **也不能因为跨域存在，就直接放弃弹窗目标**

---

## 7. 地址生成方式重定稿

既然默认按独立部署设计，那么 header 生成方式也必须调整。

### 7.1 默认使用绝对地址

Header 中引用本服务资源时，应当默认基于本服务自己的公开地址生成绝对 URL。

例如：

- `https://ipq.example.com/...`

而不是默认依赖：

- 相对当前页面路径
- 当前站点同源地址
- `/ipq/...`

---

### 7.2 本服务需要明确“公开访问地址”配置

系统配置中应当明确存在一个概念：

- 本服务公开访问基地址

例如：

- `IPQ_PUBLIC_BASE_URL`

这个地址用于：

- 生成 header
- 生成跳转链接
- 生成可能需要暴露给节点或用户的外部访问地址

---

## 8. 跨域下的设计原则

既然默认按跨域部署设计，那么系统设计应遵守以下原则：

### 8.1 尽量避免从注入脚本直接跨域调业务 API

不是说绝对不能做，而是默认不应依赖这条路径。

因为一旦依赖它，就会立刻引出：

- CORS
- 认证凭据传递
- 预检请求
- Cookie 策略
- 浏览器策略差异

因此默认设计应优先选择：

- 打开本服务页面
- 由本服务页面内完成业务动作

而不是：

- 在 `Komari` 页面内直接向本服务 API 发复杂跨域请求

---

### 8.2 登录态只属于本服务自身域名

本服务的登录态应只在本服务自己的域名下成立。

不应继续默认假设：

- `Komari` 页面和本服务共享同一套 Cookie
- 同站点下天然没有登录态问题

更合理的默认模型是：

- `Komari` 有自己的登录态
- 本服务有自己的登录态
- 两者独立

---

### 8.3 页面内增强能力的定位

“页内弹窗”本身仍然是产品优先目标，不应被简单降级成“不重要的增强项”。

但需要区分两件事：

- **产品优先体验**：页内弹窗
- **工程唯一依赖**：不能只依赖页内弹窗

也就是说：

- 页内弹窗要优先尝试
- 但不能让系统因为弹窗不稳定而整体失效

因此更准确的原则是：

- 页内弹窗优先
- 独立页面兜底
- 不把页内展示能力当成产品成立的唯一前提

---

## 9. 对现有产品设计的影响

如果按这份重定稿执行，则现有产品设计需要做以下修正：

### 9.1 保留的部分

以下方向仍然成立：

- 本服务是独立服务
- 不修改 `Komari` 官方源码
- 通过 header 注入实现入口接入
- 单管理员模型
- 节点由用户主动从 `Komari` 页面添加
- 节点详情以本服务为主承载
- 展示重点是“当前 IP 质量结果”

---

### 9.2 需要修正的部分

以下前提需要修正：

- “默认同域部署”
- “阶段 1 不考虑跨域问题”
- “默认相对路径生成 header”
- “默认 `/ipq/` 子路径部署”
- “页内弹窗必须建立在同域子路径前提上”

建议改成：

- 默认跨域独立部署
- 默认绝对地址生成 header
- 页内弹窗仍然优先争取实现
- 独立页面作为稳定兜底

---

## 10. 对开发环境的影响

这次纠偏也意味着：

- 开发环境不应再把“统一入口反代 + `/ipq/` 子路径”当成唯一模型
- Docker 联调环境仍然可以保留
- 但它应当只是“兼容一种部署方式”，而不是“产品默认形态”

更合理的开发思路是：

### 默认开发思路

- 本服务按独立地址运行
- Komari 按独立地址运行
- 以跨域真实场景思考产品行为

### 可选联调思路

- 保留同域反代环境
- 用于验证兼容性
- 但不再把它当成唯一主路径

---

## 11. 阶段 1 推荐主链路

我建议阶段 1 的主链路收口为：

1. 部署本服务
2. 登录本服务
3. 配置本服务公开地址
4. 复制 header
5. 粘贴到 `Komari`
6. 在 `Komari` 节点详情页看到入口按钮
7. 点击按钮后，优先在当前页面打开弹窗
8. 如果弹窗链路不可行或不稳定，则打开本服务页面完成：
   - 登录（如果未登录）
   - 节点接入
   - 当前 IP 质量查看

这条链路应当在“跨域、独立部署、无需统一入口”的前提下成立。

其中：

- **理想体验**：页内弹窗
- **稳定兜底**：独立页面

---

## 12. 最终结论

本项目后续应以以下原则为准：

### 核心原则

- **默认独立部署**
- **默认跨域场景成立**
- **Header 只做入口桥接**
- **业务闭环在本服务完成**
- **独立页面优先于页内嵌入**

### 部署原则

- 不要求用户配置同域子路径
- 不要求用户额外搭统一入口反代
- 不要求本服务与 `Komari` 绑在同一域名下

### 实现原则

- 默认绝对地址生成 header
- 默认跳转到本服务页面
- 页内增强能力后置

---

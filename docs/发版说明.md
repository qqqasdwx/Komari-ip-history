# 发版说明

本项目采用两条发布线：

- `latest`
  用于开发联调和快速试用
- `v0.y.z`
  用于正式版本

## 版本规则

当前按务实版 SemVer 执行。

### PATCH

用于：

- bug 修复
- 安全修复
- 样式修正
- 兼容性修正
- 不改变主要使用方式的小改动

### MINOR

用于：

- 新增用户可见功能
- 接入流程变化
- 历史、快照、注入体验的结构性变化
- API 结构变化
- 有升级影响的数据模型变更

### MAJOR

用于：

- 不兼容的 API 或行为变更
- 需要明确升级迁移说明的大版本发布

当前首个稳定版本采用 `v1.0.0`。

## 当前版本状态（截至 2026-04-17）

当前分支已经不再只是“第一阶段链路修正”，而是提前合入了若干后续能力，包括：

- cron / timezone / `CRON_TZ` 链路
- 独立节点创建
- `Komari` 绑定 / 解绑
- 自动探查计划接口
- 目标 IP 启停
- 通知系统（Telegram / Javascript / Webhook）
- API Key 与对外只读 API

因此后续发版时，建议不要再把它描述成“仅修复 Komari 跳转问题”的小版本，而应按实际用户可见能力评估版本号。

## 当前发布产物

当前以 Docker 镜像为主：

- `ghcr.io/qqqasdwx/komari-ip-history:latest`
- `ghcr.io/qqqasdwx/komari-ip-history:v0.y.z`

正式部署建议固定到具体版本，不要长期跟随 `latest`。

## 当前这批改动的建议版本等级

如果以上已实现能力一起进入正式版本，建议至少按 **MINOR** 处理，而不是 PATCH。

原因：

- 有明显新增用户可见功能
- 有数据模型变更
- 有新的管理页与新接口
- 有新的 public API 与 API Key 能力

## 正式发版流程

推荐流程：

1. 在本地完成验证
2. 确认 `master` 已包含本次发布代码
3. 创建 annotated tag
4. 推送 tag
5. 等待 GitHub Actions 构建并发布对应镜像
6. 创建同名 GitHub Release

示例命令：

```bash
git checkout master
git pull --ff-only origin master
git tag -a v0.1.0 -m "Release v0.1.0"
git push origin v0.1.0
```

## GitHub Release

建议 Release 内容包含：

- 本版主要变更
- 升级注意事项
- 已知限制

可以在 GitHub 自动生成的 release notes 基础上补充说明。

## 当前限制

节点安装命令已切换为当前服务端直出 install script，不再固定依赖仓库默认分支脚本。

当前剩余限制更集中在：

- `IPQ Node + Komari Binding` 的底层主实体仍未完全拆表重构
- public API 虽已补日志与限流，但长期稳定开放前仍建议继续收口错误语义与兼容约束

## 当前已知发布风险（截至 2026-04-17）

除安装脚本来源问题外，当前分支还有几项“可以继续开发验证，但不建议在文案上宣称完全收口”的点：

1. **节点模型底层仍在兼容旧结构**
   - `Node` 仍同时承担主实体与旧 `Komari` 标识兼容职责

2. **public API 仍建议按“可用但继续观察”发布**
   - 已有 API Key、限流、访问日志
   - 但若要长期稳定开放，仍建议继续收口错误语义与兼容边界

## 当前建议的 Release Notes 写法

如果近期要发一个可用版本，建议 Release Notes 采用下面这种口径：

- **本版新增**
  - 节点上报时区配置与 `CRON_TZ`
  - 未接入节点的新标签页配置链路
  - 独立节点与 `Komari` 绑定能力
  - 自动探查计划接口与目标 IP 启停
  - 通知通道 / 规则 / 测试发送 / 投递日志
  - API Key 与只读 public API

- **本版仍有限制**
  - 节点模型底层仍在兼容旧结构
  - public API 虽已补日志与限流，但仍建议继续观察兼容语义

## 当前不做的事

当前不包含：

- 自动跑 E2E 的 GitHub Actions 发布门禁
- 多平台二进制附件
- 复杂的 `beta` / `rc` 发布体系
- 独立维护发布分支

# 发版说明

本文档用于固化当前项目的版本发布口径，避免上下文丢失后重新讨论。

---

## 1. 当前发版策略

- 默认分支持续发布 `latest` 镜像，用于开发联调和快速试用。
- 正式版本通过 Git tag 发布，版本号采用 `v0.y.z` 形式。
- 正式版本发布时，同时创建 GitHub Release。
- 当前交付物以 Docker 镜像为主，不额外提供二进制附件。
- 浏览器端整链路验证继续在本地手工完成，不再依赖 GitHub Actions 跑 E2E。

---

## 2. 版本线说明

当前保留两条版本线：

- `latest`
  - 跟随默认分支
  - 允许不稳定
  - 主要用于开发、自测、快速联调

- `v0.y.z`
  - 代表正式版本
  - 面向实际部署
  - 用户部署时应优先固定到具体版本，而不是长期跟随 `latest`

当前暂不引入预发布通道，不使用：

- `beta`
- `rc`
- `nightly`

如果后面确实出现“大改动需要提前试用”的场景，再补：

- `v0.6.0-rc.1`
- `v0.6.0-beta.1`

---

## 3. 版本号规则

当前按务实版 SemVer 执行。

### 3.1 起始版本

- 第一版正式版本从 `v0.1.0` 开始。
- 当前阶段不直接使用 `v1.0.0`。

### 3.2 何时升 PATCH

以下情况升 `PATCH`，例如 `v0.1.0 -> v0.1.1`：

- bug 修复
- 安全修复
- 样式修正
- 兼容性修正
- 不改变主要使用方式的小型实现调整

### 3.3 何时升 MINOR

以下情况升 `MINOR`，例如 `v0.1.0 -> v0.2.0`：

- 新增用户可感知功能
- 接入流程发生变化
- 历史 / 快照 / 注入体验有结构性变化
- API 返回结构或字段口径发生变化
- 数据模型或数据库迁移对部署升级有影响

### 3.4 何时升 MAJOR

当前阶段暂不启用 `MAJOR`。

等以下内容都进入稳定期后，再考虑 `v1.0.0`：

- 注入模式稳定
- 接入命令稳定
- 多 IP 行为稳定
- 历史与快照能力稳定
- 升级路径稳定

---

## 4. 当前仓库的发布基础

当前仓库已经具备这些基础：

- 默认分支推送会发布 `latest`
- 默认分支推送会发布 commit sha 镜像
- `v*` tag 会触发 Docker 镜像构建与发布

也就是说，正式发版当前不需要重做整套 Docker 发布流程。

---

## 5. 正式发版流程

正式发版按这个顺序执行：

1. 在本地完成手工验证
2. 确认 `master` 已合入本次正式版本的代码
3. 创建 annotated tag
4. 推送 tag
5. 等待 GitHub Actions 产出对应 tag 镜像
6. 创建同名 GitHub Release
7. 在 Release 中写清本版变更和升级注意事项

推荐命令：

```bash
git checkout master
git pull --ff-only origin master
git tag -a v0.1.0 -m "Release v0.1.0"
git push origin v0.1.0
```

---

## 6. GitHub Release 口径

GitHub Release 采用轻量方案：

- 标题直接使用版本号，例如 `v0.1.0`
- 内容先使用 GitHub 自动生成的 release notes
- 维护者再补一段简短说明，概括：
  - 这版主要改了什么
  - 是否有升级注意事项
  - 是否存在已知限制

建议 Release 描述保持这个结构：

```md
## 版本说明

一句话说明这版的重点。

## 升级注意事项

- 如有数据库迁移、接入命令变化、配置项变化，写在这里

## 已知限制

- 如有未闭环风险，写在这里

## What's Changed

使用 GitHub 自动生成内容
```

---

## 7. 用户侧部署建议

README 中需要同时保留两种镜像说明：

- `latest`
  - 给快速试用和联调

- 固定版本 tag
  - 给正式部署
  - 例如 `ghcr.io/qqqasdwx/komari-ip-history:v0.1.0`

生产环境默认建议使用固定版本 tag，不建议长期直接跟随 `latest`。

---

## 8. 当前未闭环风险

当前正式发版还有一个关键风险没有解决：

- 节点接入命令里的安装脚本地址仍固定指向默认分支 `master`

这会导致一个问题：

- 即使用户部署的是 `v0.1.0` 镜像
- 只要安装脚本仍从 `master` 拉取
- 节点端实际执行的脚本就可能来自未来版本

这会破坏正式版本最基本的能力：

- 版本固定
- 问题回滚
- 线上行为可追溯

因此，在第一版正式发布前，必须完成下面其中一种收口：

- 接入命令改为指向 tag 对应脚本地址
- 或者接入命令改为由当前服务端生成并输出与当前版本匹配的安装脚本地址

在这个问题收口之前：

- 可以继续发布 `latest`
- 也可以继续做本地验证
- 但不应把 tag 版本视为真正闭环的正式版本

---

## 9. 当前不做的事

当前暂不做：

- GitHub Actions 自动跑 E2E
- 多平台二进制附件发布
- 自动生成完整 changelog 文件
- 复杂的 beta / rc 发布体系
- 每个版本单独维护一套专门分支

当前先把最小可用的正式发版路径走通即可。

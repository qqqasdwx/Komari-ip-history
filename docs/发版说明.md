# 发版说明

本项目采用两条发布线：

- `latest`
  用于开发联调和快速试用
- `v0.y.z`
  用于正式版本

## 版本规则

当前按务实版 SemVer 执行。

### PATCH

用于：

- bug 修复
- 安全修复
- 样式修正
- 兼容性修正
- 不改变主要使用方式的小改动

### MINOR

用于：

- 新增用户可见功能
- 接入流程变化
- 历史、快照、注入体验的结构性变化
- API 结构变化
- 有升级影响的数据模型变更

### MAJOR

用于：

- 不兼容的 API 或行为变更
- 需要明确升级迁移说明的大版本发布

当前首个稳定版本采用 `v1.0.0`。

## 当前发布产物

当前以 Docker 镜像为主：

- `ghcr.io/qqqasdwx/komari-ip-history:latest`
- `ghcr.io/qqqasdwx/komari-ip-history:v0.y.z`

正式部署建议固定到具体版本，不要长期跟随 `latest`。

## 正式发版流程

推荐流程：

1. 在本地完成验证
2. 确认 `master` 已包含本次发布代码
3. 创建 annotated tag
4. 推送 tag
5. 等待 GitHub Actions 构建并发布对应镜像
6. 创建同名 GitHub Release

示例命令：

```bash
git checkout master
git pull --ff-only origin master
git tag -a v0.1.0 -m "Release v0.1.0"
git push origin v0.1.0
```

## GitHub Release

建议 Release 内容包含：

- 本版主要变更
- 升级注意事项
- 已知限制

可以在 GitHub 自动生成的 release notes 基础上补充说明。

## 当前限制

当前正式发版还有一个未闭环点：

- 节点安装命令中的安装脚本地址仍固定指向默认分支 `master`

这意味着：

- 即使用户部署的是固定版本镜像
- 节点侧安装脚本仍可能来自更新的默认分支内容

在这个问题收口之前：

- `latest` 可以继续使用
- 版本 tag 可以继续发布
- 但严格意义上的“完全可追溯正式版本”仍不完整

建议后续收口方式二选一：

- 安装命令指向 tag 对应脚本
- 或由当前服务端生成与当前版本匹配的安装脚本地址

## 当前不做的事

当前不包含：

- 自动跑 E2E 的 GitHub Actions 发布门禁
- 多平台二进制附件
- 复杂的 `beta` / `rc` 发布体系
- 独立维护发布分支

# 集成与行为说明

本文档面向部署者和贡献者，说明项目当前稳定的集成方式和主要功能行为。

## 当前同步说明（截至 2026-04-17）

本文件描述“当前稳定行为”，因此这里优先同步已经实际进入代码的能力与限制。若与较早的方案讨论文档存在冲突，以本文、`用户操作手册.md` 和当前代码行为为准。

## 项目定位

`Komari-ip-history` 是一个独立部署的 IP 质量服务，用来把节点的 IP 质量结果接入 `Komari`。

它不修改 `Komari` 官方源码，而是通过 `Komari` 的 `custom_head` 注入一个入口按钮，再由本服务提供：

- 后台管理页面
- 节点结果页面
- 历史变化页面
- 快照对比页面
- 节点侧上报配置和安装脚本

## 默认集成方式

当前默认按下面的方式设计：

- IPQ 作为独立服务部署
- `Komari` 通过 `custom_head` 注入入口
- 注入脚本只负责桥接、识别节点、挂载按钮和跳转/弹窗
- 复杂展示和业务逻辑都在 IPQ 自身页面完成

当前不把下面这些前提当成默认要求：

- 与 `Komari` 同域
- 与 `Komari` 同机
- 默认走 `/ipq/` 子路径

## 查看流程

### 管理员

`Komari` 管理员点击入口后：

- 如果节点已接入且 IPQ 已有登录态，优先直接打开页内弹窗查看当前结果
- 如果节点未接入：
  - 允许先自动注册节点壳子
  - 然后在**新标签页**打开 IPQ 后台
  - 停在节点列表页并自动打开对应节点的上报设置
- 当前设计目标是不再把接入 / 配置流程继续嵌进 `Komari` 弹窗
- 配置完成后，用户自行回到 `Komari` 再次点击查看

### 游客

游客链路受全局“游客只读”开关控制。

- 默认关闭
- 开启后，游客可以查看当前节点的匿名只读结果
- 游客不能接入节点，也不能进入管理后台

## 节点与目标 IP

当前系统里的“节点”已经不再只等同于 `Komari` 节点。

现阶段同时支持：

- 从 `Komari` 入口自动注册的节点
- 在 IPQ 后台独立创建的节点
- 后续手动绑定到 `Komari` 的节点

一个节点下可以配置多个目标 IP。

当前行为：

- 每个目标 IP 单独保存当前结果
- 每个目标 IP 单独累积历史快照
- 结果页、历史页、快照对比页都支持按目标 IP 查看
- 节点侧安装命令会按当前目标 IP 顺序依次探测并上报
- 每个目标 IP 当前还有两层额外语义：
  - `source`
    - `manual`
    - `discovered`
  - `enabled`
    - 是否继续参与上报

## 历史与快照

历史能力拆成两个页面：

- 历史主页面
- 快照对比页面

### 历史主页面

历史主页面的主视角是“字段变化列表”，不是“快照列表”。

支持：

- 按时间范围筛选
- 按目标 IP 筛选
- 按字段筛选
- 分页

### 快照对比页面

快照对比页面用于比较同一目标 IP 的两个历史时间点。

当前行为：

- 顶部使用双游标时间轴选择两个时间点
- 页面下方并排展示两个完整快照
- 变化字段会高亮
- 支持收藏单条历史快照

## 历史保留策略

历史继续按完整快照存储，不改成 diff-only。

当前保留策略：

- 全局配置项：`retention_days`
- `-1` 表示永久保留
- `1` 及以上表示保留最近 N 天
- 已收藏快照不会被自动清理
- 取消收藏后重新受全局保留策略约束

保留策略只作用于历史快照，不影响：

- 节点本身
- 目标 IP 本身
- 当前结果

## cron / timezone 语义

当前实现已经打通：

- 页面预览
- 配置保存
- install config
- 安装脚本
- 节点侧 cron

当前行为是：

- 页面会明确告诉用户：
  - 当前 cron 按哪个时区解析
- 当前版本默认使用**用户当前浏览器时区**
- 页面也支持手动输入 IANA 时区名进行覆盖
- 节点安装后的 cron 文件会写入：
  - `CRON_TZ=<timezone>`

## 自动探查与计划接口

当前节点侧不再只依赖“服务端预置固定目标 IP 列表”。

已经进入实现的链路包括：

- 节点侧枚举候选 IP
- 调用 `POST /api/v1/report/nodes/:uuid/plan`
- 服务端返回：
  - `approved_targets`
  - `schedule_cron`
  - `timezone`
  - `run_immediately`

当前服务端会：

- 自动把新发现目标 IP 持久化为 `discovered`
- 支持对单个目标 IP 做启用 / 停用

当前实现已经支持：

- 启用的手动目标 IP 与自动发现目标 IP 并存
- 手动目标 IP 不会因为不在本次 `candidate_ips` 中而从计划中丢失

## 公开访问地址

项目默认按当前访问地址生成：

- 注入脚本地址
- 页面跳转地址
- 节点上报地址

如果你的部署前面有反向代理，应正确传递：

- `Host`
- `X-Forwarded-Proto`

如果你希望统一使用固定外部地址，可设置：

- `IPQ_PUBLIC_BASE_URL`

## 通知系统

当前后台已提供通知设置页和后端接口。

当前能力包括：

- sender provider 定义
- channel 管理
- 规则管理
- 测试发送
- 投递日志

当前 sender 类型：

- `telegram`
- `javascript`
- `webhook`

当前已知限制：

- 事件模型还未完全补齐到方案文档中的完整粒度
- Javascript sender 已能跑基础流程，但异步 `fetch/xhr` 运行时还需要继续加固

## 对外只读 API

当前已经提供 public API：

- `/api/public/v1/nodes`
- `/api/public/v1/nodes/:uuid`
- `/api/public/v1/nodes/:uuid/targets`
- `/api/public/v1/nodes/:uuid/targets/:targetID`
- `/api/public/v1/nodes/:uuid/history`
- `/api/public/v1/nodes/:uuid/history/events`

认证方式：

- `X-IPQ-API-Key`

后台同时已提供 API Key 管理页面与接口。

当前已知限制：

- 已有访问日志
- 已有限流
- 若对外承诺长期稳定兼容，还需要后续补更明确的错误语义与版本约束

## 安装链路当前限制

当前用户在页面复制的安装命令，实际链路是：

1. 直接请求当前服务端的 `install-script` 接口
2. 由当前服务端生成并返回 install script
3. script 再向当前服务端获取本次计划与 reporter 相关配置

这意味着：

- 节点最终使用的 reporter 配置来自当前服务端
- 安装引导脚本本身也来自当前服务端
- 若要发正式版本，重点应转向验证当前服务端生成脚本与目标版本行为一致

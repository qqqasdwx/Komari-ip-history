# 前端迁移计划

> 注意：本文件是前端迁移过程中的专题计划文档。  
> 当前仍有参考价值，但不直接代表最新产品边界；产品边界以 `docs/已确认事项.md` 为准。

本文档用于说明：是否要把当前前端从“原生 TypeScript + 手写 DOM”迁移到与 `Komari` 更接近的前端技术栈，以及如果迁移，应该怎么迁、先迁什么、每一步如何验收。

## 1. 当前结论

我建议迁移。

原因不是“为了追新”，而是当前项目已经明显进入前端信息架构和界面一致性阶段，继续在现有前端实现上叠加，会越来越难维护：

- 当前前端使用原生 `TypeScript + Vite`，通过字符串拼接 DOM 和手动绑定事件完成页面渲染。
- 这种方式在功能很少时足够轻，但现在项目已经进入：
  - 后台左侧菜单拆分
  - 节点列表/详情/历史页多页面联动
  - 当前结果页结构化渲染
  - 设置页拆分与路由组织
  - 后续移动端与响应式适配
- 这些工作继续在“手写 DOM”上做，修改成本会持续上升，回归风险也会越来越高。
- `Komari` 官方前端使用的是现代前端栈，当前已经确认：
  - `React`
  - `react-router-dom`
  - `Vite`
  - `Tailwind CSS`
  - `@radix-ui/themes`
- 本项目如果继续强调“风格尽量贴近 Komari”，那么技术栈也尽量靠近它，会更容易对齐页面结构、布局方式和组件抽象。

因此我的判断是：

- 迁移有成本。
- 但现在迁，成本仍然可控。
- 再往后拖，前端逻辑继续堆积后，迁移只会更贵。

## 2. 迁移目标

本次迁移不是“重做产品”，而是“重建前端承载方式”。

迁移后的目标：

1. 前端技术栈与 `Komari` 接近。
2. 页面路由、壳子、菜单、通用组件改成组件化组织。
3. 当前已有主流程不丢失：
   - 登录
   - 节点列表
   - 节点详情
   - 历史页
   - 变化视图
   - 接入配置
   - 变化规则
   - 管理员设置
4. 保持现有后端 API 不大改，优先只替换前端实现。
5. 迁移过程中始终保证可运行、可回归验证，而不是一次性推倒。

不属于本次迁移目标的内容：

- 不在迁移阶段同时大规模扩功能。
- 不在迁移阶段同时重做后端接口语义。
- 不以“完全复制 Komari 全部 UI”为目标。
- 不要求第一版迁移完成时就做到完整移动端优化，但要为后续响应式适配打好基础。

## 3. 目标技术栈

建议采用以下前端栈：

- `React`
- `TypeScript`
- `Vite`
- `react-router-dom`
- `Tailwind CSS`
- `@radix-ui/themes`

可选补充：

- `clsx` / `class-variance-authority`
  - 用于整理通用组件变体
- `sonner`
  - 用于 toast，和 `Komari` 更接近

暂不建议一开始就引入复杂状态管理库，例如：

- `redux`
- `zustand`
- `mobx`

原因：

- 当前状态规模还没大到必须引入全局状态库。
- 先用 `React` 组件状态 + 少量 context 就足够。
- 先把页面重构完成，再决定是否需要状态管理升级。

## 4. 迁移原则

### 4.1 原则一：前后端解耦，优先只换前端壳

当前 Go 后端已经能提供基础 API，迁移时不应同时重写后端。

这意味着：

- 先保持 `/ipq/api/v1/...` 接口兼容。
- 前端优先重写页面组织和渲染逻辑。
- 只有当某些前端体验明显受阻时，才反推后端 API 微调。

### 4.2 原则二：渐进迁移，不一次性替换所有页面

如果一次性把全部页面推倒重来，风险太高。

更合理的方式是：

- 先迁“壳子”
- 再迁“主路径”
- 最后迁“高级设置和低频页”

### 4.3 原则三：迁移顺序服从产品优先级

当前项目最重要的是：

1. 节点当前 IP 质量结果
2. 历史变化
3. 接入配置

因此迁移顺序也必须围绕这三件事，而不是先迁最容易的页面。

### 4.4 原则四：每一阶段都要能验收

每迁完一部分，都必须满足：

- 能构建
- 能跑
- 核心页面能点
- Playwright 主流程不回归，或同步更新脚本后重新通过

## 5. 推荐迁移顺序

## 5.1 阶段 A：建立新的 React 前端骨架

目标：

- 在现有仓库内建立新的 React 前端基础结构。
- 完成路由、布局、主题、通用组件能力。
- 不要求此阶段就完成所有业务页。

预计内容：

- 创建新的 React 入口
- 配置 `Vite + React + Tailwind + Radix`
- 建立基础目录结构
- 建立后台通用壳子：
  - 左侧菜单
  - 顶部工具栏
  - 页面标题区
  - 通用按钮、卡片、表格行、代码框

验收标准：

- 能正常启动前端
- 左侧菜单路由跳转正常
- 登录页和后台壳子可显示
- 可以保留当前旧页面作为过渡占位

## 5.2 阶段 B：迁移节点列表页

目标：

- 先迁最常用的入口页。
- 把节点列表页改成 React 组件渲染。

预计内容：

- 节点列表请求逻辑迁移
- 搜索输入迁移
- 单行列表表格迁移
- 空状态迁移

验收标准：

- `/ipq/#/nodes` 页面可正常打开
- 搜索能用
- 空状态能用
- 列表点击能进入详情页

## 5.3 阶段 C：迁移节点详情页

目标：

- 迁移当前项目最核心页面。

预计内容：

- 当前 IP 质量结果区迁移
- 最近变化摘要迁移
- 历史变化入口迁移
- 节点上报设置折叠区迁移

这一步最关键，因为它决定后续的展示模板。

验收标准：

- 当前结果区渲染完整
- 风险评分、类型、流媒体、邮局等结构化展示不回归
- 返回按钮、顶部操作按钮正常
- 与现有 API 对接无误

## 5.4 阶段 D：迁移历史页和变化视图

目标：

- 把“当前结果 + 历史变化”链路补完整。

预计内容：

- 历史记录列表迁移
- 历史记录对比展示迁移
- 变化视图筛选迁移

验收标准：

- 历史页可正常进入
- 变化视图可正常进入
- 字段级变化与重点/辅助规则继续生效

## 5.5 阶段 E：迁移接入配置与系统设置页

目标：

- 把后台其余页面迁移完成。

预计内容：

- 接入配置页迁移
- 变化规则页迁移
- 管理员设置页迁移
- 代码高亮和固定高度代码框迁移

验收标准：

- 左侧 4 个设置页全部可用
- 保存逻辑不回归
- 接入配置页可复制 loader / inline 代码

## 5.6 阶段 F：删除旧实现

目标：

- 当新前端覆盖主路径后，清理旧的手写 DOM 实现，避免双轨长期并存。

预计内容：

- 删除旧页面渲染逻辑
- 删除旧样式中已废弃部分
- 更新文档
- 更新 Playwright 校验口径

验收标准：

- 仓库内只保留一种前端实现
- 没有“新旧两套页面长期共存”的维护负担

## 6. 建议目录结构

建议迁移后前端逐步整理为以下结构：

```text
web/
  src/
    app/
      router.tsx
      providers.tsx
    components/
      layout/
      ui/
      node/
      change/
      settings/
    pages/
      login/
      nodes/
      node-detail/
      node-history/
      node-changes/
      settings/
    lib/
      api.ts
      format.ts
      result.ts
    styles/
      globals.css
      theme.css
```

目录意图：

- `components/layout`
  - 后台壳子、顶部栏、左侧菜单、页面头
- `components/ui`
  - 通用按钮、卡片、输入框、代码框、状态标签
- `components/node`
  - 节点列表行、详情页结果区、上报设置区
- `components/change`
  - 历史变化、变化卡片、对比项
- `components/settings`
  - 接入配置、规则配置

## 7. 数据与状态组织建议

当前前端数据主要分几类：

- 当前登录状态
- 节点列表
- 单节点详情
- 变化优先级规则

建议迁移后先按下面方式组织：

- `AuthContext`
  - 当前管理员登录状态、登出
- `SettingsContext` 或页面级 hooks
  - 变化规则
- 页面内局部状态
  - 搜索词、筛选项、当前 record 选择等

暂不建议：

- 一开始就把所有页面状态集中到一个大 store

原因：

- 现阶段页面数量不算夸张
- 先完成迁移比提前设计复杂状态架构更重要

## 8. 风险与应对

## 8.1 风险：迁移期间双轨并存导致混乱

问题：

- 新旧前端同时存在时，路由和样式容易混乱。

应对：

- 明确阶段边界
- 每迁完一页就切换该页到新实现
- 不长期让同一路由挂两套页面

## 8.2 风险：旧 Playwright 校验脚本大量失效

问题：

- DOM 结构变化后，脚本可能大量断裂。

应对：

- 在每个迁移阶段同步维护 E2E
- 优先保留稳定的 `data-*` 标识，而不是让脚本依赖样式类名

## 8.3 风险：为了迁移而改坏现有产品行为

问题：

- 技术重构很容易让用户体验回退。

应对：

- 迁移时以“行为保持一致”为原则
- 核心按钮、路由、提示文案在迁移前后尽量一致

## 8.4 风险：过早引入太多库

问题：

- 一口气引入过多库，会把迁移变成依赖治理工程。

应对：

- 第一阶段只引入最必要的栈：
  - React
  - Router
  - Tailwind
  - Radix

## 9. 迁移前的准备工作

正式开始迁移前，建议先做这些准备：

1. 冻结旧前端的大范围视觉改动  
   原因：避免一边迁移一边继续在旧实现上深挖 UI。

2. 盘点现有主路径  
   明确哪些页面必须先迁，哪些可以后迁。

3. 盘点现有可复用逻辑  
   例如：
   - API 调用封装
   - 结果结构化解析
   - 变化比较逻辑
   - 文案格式化逻辑

4. 稳定 E2E 基线  
   在迁移前保留一份当前可通过的 Playwright 基线。

## 10. 建议的实际执行顺序

如果现在就开始迁移，我建议这样做：

1. 在 `web/` 中建立 React 新壳子，但先不删除旧实现。
2. 先迁登录页和后台布局。
3. 再迁节点列表页。
4. 再迁节点详情页。
5. 节点详情稳定后，再迁历史页和变化视图。
6. 最后迁设置页。
7. 全部主路径稳定后，删除旧实现。

这个顺序的好处是：

- 始终围绕用户最关心的主路径推进
- 详情页尽早进入新体系
- 最低频的设置页放后面，不阻塞主路径

## 11. 验收标准

迁移计划完成后，应满足以下结果：

1. 前端主路径全部运行在 React 组件体系下。
2. 页面结构和风格明显更接近 `Komari`。
3. 当前结果页、历史页、变化页、设置页都不再依赖手写 DOM 拼接。
4. Playwright 主流程重新通过。
5. 后续要做响应式适配、视觉统一、页面继续收口时，改动成本显著下降。

## 12. 当前建议

我建议：

- 这份计划可以作为后续前端重构的正式起点。
- 如果你确认迁移，就不要再在旧前端上做大规模结构性改动。
- 旧前端只做必要修补。
- 新工作重点转到“建立 React 新壳子 + 逐页迁移”。

换句话说，当前最合理的后续动作不是继续堆旧页面，而是：

1. 确认迁移计划；
2. 起新的 React 前端骨架；
3. 从节点列表和节点详情开始迁移。

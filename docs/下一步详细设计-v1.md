# 下一步详细设计 v1

本文档在 [下一步正式方案-v1.md](/root/Komari-ip-history/docs/下一步正式方案-v1.md) 基础上继续下钻，给出：

- 数据模型草案
- 页面结构草案
- 分阶段任务清单

当前目标不是直接定死所有实现细节，而是把下一轮开发拆成可执行的模块。

## 1. 数据模型草案

## 1.1 节点主实体升级

当前 `Node` 仍然直接内嵌了 `KomariNodeUUID`，这不利于未来支持“独立节点 + 可选绑定”。

建议拆成两层：

### `ipq_nodes`

表示 IPQ 自己的主实体节点。

建议字段：

- `id`
- `name`
- `has_data`
- `current_summary`
- `current_result_updated_at`
- `reporter_token`
- `install_token`
- `reporter_schedule_cron`
- `reporter_timezone`
- `reporter_run_immediately`
- `reporting_enabled`
- `created_at`
- `updated_at`

说明：

- `reporter_timezone`：新增，用于保存 cron 解析时区
- `reporting_enabled`：新增，用于节点级总开关

### `komari_bindings`

表示一个 IPQ 节点和 `Komari` 节点之间的绑定关系。

建议字段：

- `id`
- `ipq_node_id`
- `komari_node_uuid`
- `komari_node_name`
- `binding_source`
  - `manual`
  - `from_komari`
- `created_at`
- `updated_at`

约束：

- `ipq_node_id` 唯一
- `komari_node_uuid` 唯一

这样就能保证：

- 一个 IPQ 节点最多绑定一个 `Komari`
- 一个 `Komari` 节点最多绑定一个 IPQ 节点

---

## 1.2 目标 IP 模型升级

当前 `NodeTarget` 需要补充“来源”和“启停”语义。

建议新增字段：

- `source`
  - `manual`
  - `discovered`
- `enabled`
  - `true/false`
- `last_seen_at`
  - 节点最近一次报告该 IP 的时间

完整建议字段：

- `id`
- `node_id`
- `target_ip`
- `source`
- `enabled`
- `sort_order`
- `has_data`
- `current_summary`
- `current_result_json`
- `current_result_updated_at`
- `last_seen_at`
- `created_at`
- `updated_at`

说明：

- `enabled=false` 表示节点侧不应再继续探查这个目标 IP
- `last_seen_at` 用来区分“当前自动发现列表里还活跃的 IP”和“历史发现过但近期没再出现的 IP”

---

## 1.3 自动探查计划模型

因为节点需要“先和服务端确认，再探查”，建议新增一层计划接口，但不一定需要单独建表。

第一版可以先不建独立表，而是由服务端动态计算计划。

如果后续要审计或回放，再考虑持久化：

### 可选的 `reporting_plans`（暂不强制）

字段示意：

- `id`
- `node_id`
- `candidate_ips_json`
- `approved_ips_json`
- `generated_at`

第一版建议：

- 先做动态计划接口
- 不先建库表

---

## 1.4 通知相关模型

通知体系建议拆成两层：

### `notification_channels`

表示可复用的发送器配置。

建议字段：

- `id`
- `name`
- `type`
  - `telegram`
  - `javascript`
  - `webhook`
- `enabled`
- `config_json`
- `created_at`
- `updated_at`

说明：

- `config_json` 用于对齐 `Komari` 的 sender 配置方式
- 例如 Telegram、Javascript、Webhook 都放 JSON

### `notification_rules`

表示实际订阅规则。

建议字段：

- `id`
- `node_id`
- `target_id`
- `field_id`
- `channel_id`
- `enabled`
- `created_at`
- `updated_at`

约束建议：

- 唯一键：`node_id + target_id + field_id + channel_id`

### `notification_deliveries`

表示投递日志。

建议字段：

- `id`
- `rule_id`
- `history_entry_id`
- `status`
  - `success`
  - `failed`
- `response_summary`
- `created_at`

第一版价值：

- 方便调试 Telegram / Javascript / Webhook
- 方便用户排查“为什么没收到”

## 2. 页面结构草案

## 2.1 节点列表页

继续保留当前入口，但能力会扩展：

### 现有能力保留

- 节点列表
- 搜索
- `report_config` 弹窗

### 新增能力

- “新建节点”按钮
- 节点支持两种来源标签：
  - `Komari 绑定`
  - `独立节点`

### 未接入节点从 Komari 进入时

继续采用：

- `/#/nodes?report_config=<uuid>`

但这里的 `report_config` 应该改成：

- 优先按 `komari_node_uuid` 找绑定节点
- 找不到则按“未完成接入壳子节点”打开配置弹窗

---

## 2.2 节点配置弹窗

这个弹窗会成为下一阶段最核心的管理入口。

建议拆成 4 个区块：

### 区块 A：节点基础信息

- 节点名称
- 节点来源
- 是否绑定 `Komari`
- 绑定的 `Komari` 节点信息

### 区块 B：上报设置

- 启用 / 停止节点级上报
- Cron
- 时区
- 安装后立即执行

### 区块 C：目标 IP

- 当前目标 IP 列表
- 区分：
  - 手动
  - 自动发现
- 支持：
  - 单个目标 IP 启停
  - 手动补充 IP
  - 删除手动 IP

### 区块 D：通知

- 入口按钮：
  - 打开通知设置页

---

## 2.3 通知设置页

整体风格和功能尽量对齐 `Komari`。

建议页面结构：

### 区块 A：总开关

- 是否启用通知系统

### 区块 B：消息模板

- 模板编辑器

### 区块 C：发送方式

- Sender 下拉：
  - `Telegram`
  - `Javascript`
  - `Webhook`

### 区块 D：发送器配置

根据 Sender 动态展示配置项。

#### Telegram

- `bot_token`
- `chat_id`
- `message_thread_id`
- `endpoint`

#### Javascript

- `script`

说明文案建议直接贴近 `Komari`：

- 需要实现 `sendMessage(message, title)`
- 可选实现 `sendEvent(event)`

#### Webhook

- `url`
- `method`
- `content_type`
- `headers`
- `body`
- `username`
- `password`

### 区块 E：测试发送

- 发送测试消息

### 区块 F：字段变化订阅规则

这是我们新增的核心区块：

- 规则列表
- 新建规则
- 选择节点
- 选择目标 IP
- 选择字段
- 选择发送器
- 启停规则

## 2.4 节点创建 / 绑定页面

建议新增一个轻量节点创建流程：

### 新建 IPQ 节点

支持：

- 只创建 IPQ 节点
- 创建后再决定是否绑定 `Komari`

### 绑定 Komari 节点

支持：

- 从一个下拉框中选择待绑定 `Komari` 节点
- 系统自动推荐可能匹配项

建议不做：

- 静默自动绑定

## 3. 后端接口草案

## 3.1 节点与绑定

建议新增：

- `POST /api/v1/nodes`
- `PUT /api/v1/nodes/:id`
- `POST /api/v1/nodes/:id/komari-binding`
- `DELETE /api/v1/nodes/:id/komari-binding`

### 兼容现有方案

保留：

- `POST /api/v1/embed/nodes/register`

但内部语义调整为：

- 注册或更新一个“待接入的 Komari 节点壳子”
- 不自动代表已完成接入

---

## 3.2 自动探查计划

建议新增：

- `POST /api/v1/report/nodes/:uuid/plan`

请求体建议：

- `candidate_ips`
- `agent_version`
- `hostname`
- `interface_summary`（可选）

返回体建议：

- `approved_targets`
- `timezone`
- `schedule_cron`
- `run_immediately`

### 使用方式

节点侧流程变成：

1. 枚举候选 IP
2. 调 `plan`
3. 按返回结果探查
4. 逐个上报

---

## 3.3 目标 IP 启停

建议新增：

- `POST /api/v1/nodes/:uuid/targets/:targetID/enable`

也可以做成：

- `PUT /api/v1/nodes/:uuid/targets/:targetID`

但从 UI 语义上看，显式启停接口更清楚。

## 5. 我建议你下一步优先拍板的两个实现问题

> 更新：截至 2026-04-17，这两个问题里，第 5.2 已经部分进入实现（计划接口已返回 `schedule_cron / timezone / run_immediately`），但仍有两个更紧急的工程性问题需要先修。

### 当前优先级更高的两个修正项

#### A. 自动探查计划接口要真正支持“手动 + 自动并存”

- 不要因为本次 `candidate_ips` 里没有某个手动目标 IP，就把它从计划里静默移除
- 否则会直接违背阶段 3 的核心目标

#### B. Javascript sender 的异步运行时需要加固

- 当前能力已可用来验证基本流程
- 但若要按“Komari sender 兼容”对外宣称可用，还需要先把异步 `fetch/xhr` 执行模型收稳

到这里还剩两个最值得先确认的实现问题：

### 5.1 Javascript sender 的执行环境

虽然方向已经定成“按 Komari 来”，但我们仍要决定：

- 是完全照 `Komari` 用同类 JS runtime
- 还是做一个更轻的兼容版

### 5.2 自动探查计划接口的返回内容

也就是：

- 服务端只返回“允许探查哪些 IP”
- 还是顺便返回：
  - cron
  - timezone
  - run_immediately
  - 其他 reporter 配置

我倾向于后者，一次拿全。

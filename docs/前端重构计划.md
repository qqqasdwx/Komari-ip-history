# 前端重构计划

## 目标与边界

本次重构目标是把现有前端整理成可维护的 React SPA，并逐步引入 Tailwind CSS v4 与 shadcn/ui 风格的基础组件。重构不迁移到 Next.js，不引入必需的 Node.js 运行时，不改变 Go 后端托管静态资源的部署模型。

必须保留的边界：

- 继续使用 Vite 构建，产物输出到根目录 `public/`。
- 继续使用 `HashRouter`，现有 `#/...` 路由保持兼容。
- 保持 `IPQ_BASE_PATH` 与 `VITE_BASE_PATH` 行为不变。
- 不改变 `/api/v1`、`/embed`、`/public` 相关接口契约。
- 不删除 Playwright 依赖的选择器与数据属性，例如 `data-node-row`、`data-node-uuid`、`data-node-name`、`data-node-report-config`、`data-detail-report`、`data-history-change-row`。
- 前端验收以 Playwright 为主，手动检查作为补充，不用“只构建通过”代替页面验收。

## UI 策略

采用「shadcn/ui primitives + Tailwind 自定义组合」。

执行原则：

- 常见交互组件、可访问性要求较高的组件，优先使用 shadcn/ui，例如 Button、Input、Label、Dialog、Tabs、Dropdown、Tooltip、Toast、Select。
- Tailwind 用于布局、间距、响应式、字体层级、页面组合和业务视觉设计。
- 不要机械地把所有视觉块都套成 shadcn/ui 组件。
- 业务卡片、Dashboard 区块、空状态、介绍区块、页面结构，根据实际信息架构自由组合。
- 不要从零手写复杂交互组件；如果 shadcn/ui 或成熟 React 库已经能稳定解决，就优先使用成熟方案。
- 不引入 Ant Design、MUI、Bootstrap、Chakra 等其他 UI 框架，除非明确要求。
- 图标统一优先使用 `lucide-react`；现有 Radix icons 可以在重构过程中逐步替换，最后再移除依赖。
- 界面风格保持现代、简洁、克制、有层次，避免过度装饰。
- shadcn/ui 组件按需引入，不批量生成；每引入一个组件，都应能说明它解决的交互、可访问性或一致性问题。

## 验收文档与截图规范

每个重构阶段完成后，都必须更新验收文档。建议新建或持续维护 `docs/前端重构验收记录.md`，记录阶段、提交、测试命令、截图路径、行为变化和遗留问题。

截图要求：

- 修改前先用 Playwright 对受影响页面截图，作为 baseline。
- 修改后再用 Playwright 对同一页面、同一视口截图，作为 after。
- 截图至少覆盖桌面视口 `1440x1200`；涉及响应式布局时增加移动视口 `390x844`。
- 截图文件建议放在 `web/playwright-output/frontend-refactor/<阶段>/`。
- 每完成一项计划，都要在验收文档中配上改前和改后截图路径，并说明具体改了什么。
- 截图生成后必须查看图片内容，确认页面已加载完成、数据正常、关键控件可见；不能用加载中、空白页、错误页或骨架屏截图作为验收材料。
- 如果截图依赖种子数据，先运行开发环境种子脚本，避免因为没有节点数据导致误判页面正常。

Playwright 验收要求：

- 优先扩展现有 `web/playwright/verify-react-preview-nodes.mjs` 和 `web/playwright/verify-embed-auth-flows.mjs`，不要另起一套不可复用的临时脚本。
- 每个主要页面至少校验一个稳定可见文本或 ARIA role，再截图。
- 截图前必须等待关键选择器，例如节点页等待 `data-node-row` 或空状态，详情页等待 `data-detail-report`，历史页等待 `data-history-change-row` 或明确空状态。
- 对弹窗、Toast、嵌入 iframe、公开只读页等状态型 UI，要分别截图并记录触发步骤。

## 测试数据要求

重构前必须构建稳定、可重复的测试数据。不能只依赖当前数据库里的偶然数据，也不能只验证“有一个节点”的 happy path。

测试数据至少覆盖：

- 无节点：节点列表空状态。
- 有节点但无 IP 目标：上报设置提示先添加目标 IP。
- 单 IP 目标且无历史：详情页当前结果、历史入口和空历史状态。
- 单 IP 目标且有多条历史：历史变化列表、分页、时间筛选。
- 多 IP 目标：目标切换、不同目标的当前结果和历史记录。
- 多条快照：快照对比页可以选择不同时间点并展示差异。
- 收藏快照：收藏状态展示，且不被历史清理误判。
- 游客公开访问开启和关闭：公开详情页、游客禁用提示。
- 管理员未登录：受保护页面跳转登录并带 `redirect`。
- 接口错误或空响应：加载态、错误态、重试或提示文案。
- 移动端数据密集场景：长节点名、多目标、多历史行不应撑破布局。

数据准备方式：

- 优先复用并扩展 `deploy/dev/workspace/seed-dev-nodes.sh` 和 `seed-dev-nodes.py`。
- Playwright 可以在运行前通过 API 补充本阶段需要的数据，但必须可重复执行，重跑不会产生不可控脏数据。
- 每个测试节点使用稳定命名，例如 `开发种子-空节点`、`开发种子-单IP历史`、`开发种子-多IP历史`、`开发种子-多快照对比`。
- 验收文档必须记录本阶段依赖了哪些测试数据，以及如何重新生成。

## 当前页面清单

| 路由 | 页面 | 当前关键行为 |
| --- | --- | --- |
| `#/login` | 登录页 | 支持用户名密码登录，支持登录后按 `redirect` 返回原页面。 |

| `#/nodes/:uuid` | 节点详情 | 展示当前 IP 质量报告、目标 IP 切换、历史入口，嵌入模式下隐藏管理入口。 |

| `#/nodes/:uuid/compare` | 快照对比 | 支持选择时间范围、时间线、两份报告并排比较。 |

| `#/connect` | 接入桥接页 | 用于 Komari 注入链路和授权桥接。 |

| `#/settings/history-retention` | 历史保留 | 配置历史保留策略。 |

## 分步计划

### 1. 建立基线

目标：记录当前行为，保证后续每一步都能比较回归风险。

查看页面：`#/login`、`#/nodes`、`#/nodes/:uuid`、`#/nodes/:uuid/history`、`#/nodes/:uuid/compare`、`#/settings/integration`。

主要改动：不改 UI，只补充必要的验证说明或临时截图输出目录说明。

行为变化：无。

验证方式：

- `go test ./...`
- `cd web && npm run build`
- 构建并记录基线测试数据，覆盖空节点、有节点、多历史、多快照等场景。
- 使用 Playwright 截取本阶段改动前 baseline 图。
- `docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/run-e2e.sh`
- 查看 Playwright 输出截图，确认不是加载中、空白页或错误页。
- 更新 `docs/前端重构验收记录.md`。

### 2. 引入 UI 基础设施

目标：加入 shadcn/ui 所需的基础结构，但先不大规模替换页面。

查看页面：`#/login`、`#/nodes`、`#/settings/integration`。

主要改动：

- 新增 `web/src/react-preview/components/ui/`。
- 新增 `cn` 工具函数，例如 `web/src/react-preview/lib/utils.ts`。
- 整理 Tailwind v4 主题变量、圆角、边框、背景、前景色、危险色和静音色。
- 先引入低风险组件：`Button`、`Input`、`Label`、`Card`、`Badge`。

行为变化：按钮、输入框和卡片的视觉样式可以变化；表单提交、接口请求、路由跳转不变。

验证方式：登录、节点搜索、设置页保存必须仍可操作。用 Playwright 分别保存改前和改后截图，并在验收文档中说明基础组件替换范围。

### 3. 拆分 `App.tsx`

目标：把接近全量页面逻辑的 `App.tsx` 拆成页面、布局、组件和 hooks，先追求结构清晰，不追求视觉重做。

查看页面：所有路由。

主要改动：

- 新增 `pages/`，拆出 `LoginPage`、`NodesPage`、`NodeDetailPage`、`NodeHistoryPage`、`NodeHistoryComparePage`、`PublicNodeDetailPage`、`settings/*`。
- 新增 `components/layout/`，拆出 `AppShell`、侧边栏、顶部栏、页面标题。
- 新增 `components/node/`，拆出节点行、目标切换、上报配置弹窗。
- 新增 `components/history/`，拆出筛选条、分页、变化列表、对比时间线。
- 保持现有 `lib/api.ts`、`lib/history.ts`、`lib/report.tsx` 等领域逻辑位置稳定。

行为变化：无。拆分期间不得改接口 payload、query 参数、日期转换逻辑和权限判断。

验证方式：每拆一组页面执行 `cd web && npm run build`，并运行相关 Playwright 检查确认截图与拆分前行为一致。

### 4. 重构应用框架与导航

目标：让后台布局更稳定，支持桌面和移动端，同时保留现有信息架构。

查看页面：`#/nodes`、`#/nodes/:uuid`、`#/settings/integration`、`#/settings/history-retention`、`#/settings/user`。

主要改动：

- 使用统一的 `AppShell` 承载侧边栏、顶部操作区和内容区。
- 当前路由高亮逻辑保持在导航组件内。
- 移动端把侧边栏改成可打开的导航层或顶部菜单。
- 页面标题、返回按钮、顶部操作按钮统一样式。

行为变化：移动端导航交互可以改变；桌面端路由、登出、返回、跳转行为不变。

验证方式：用 Playwright 截取桌面宽度 `1440px` 和移动宽度 `390px` 的前后对比图，人工查看导航不是遮挡、错位或加载中状态。

### 5. 替换表单、弹窗和反馈组件

目标：优先改最容易复用且风险可控的交互控件。

查看页面：`#/login`、`#/nodes` 的“上报设置”、`#/settings/integration`、`#/settings/history-retention`、`#/settings/user`。

主要改动：

- 登录表单切换到统一 `Input`、`Label`、`Button`。
- “上报设置”切换到统一 `Dialog`、`Tabs` 或分组卡片。
- Toast 组件统一为固定位置、自动关闭、成功和错误两种语义。
- 设置页表单统一加载态、保存态、错误态。

行为变化：弹窗可增加 Esc 关闭和焦点管理；Toast 样式可以变化，自动关闭时间保持接近当前 `2600ms`。

验证方式：登录失败、登录成功、添加 IP、删除 IP、保存设置、接口错误提示都要用 Playwright 或手动触发覆盖；弹窗和 Toast 需要单独截图并写入验收记录。

### 6. 重构节点列表与详情

目标：提升核心数据浏览页面的可读性和响应式表现。

查看页面：`#/nodes`、`#/nodes/:uuid`、`#/public/nodes/:uuid`。

主要改动：

- 节点列表改成清晰的表格或卡片列表。
- 搜索框、空状态、加载态和错误态统一。
- 节点详情保留 `CurrentReportView` 的字段展示逻辑，但外围卡片、目标切换、历史入口重新组织。
- 公开详情页复用详情展示组件，但隐藏后台管理操作。
- 确认空节点、无目标节点、单目标节点、多目标节点都能正常展示。

行为变化：允许视觉层次、排序展示和空状态文案优化；搜索触发方式、行点击、上报设置按钮阻止行点击的行为不变。

验证方式：运行 `verify-react-preview-nodes.mjs`，重点确认节点搜索、弹窗、详情、嵌入模式。对节点列表、详情页、公开详情页分别保存前后截图并人工查看。

### 7. 重构历史与对比页面

目标：降低历史筛选和快照对比的认知成本。

查看页面：`#/nodes/:uuid/history`、`#/nodes/:uuid/compare`。

主要改动：

- 历史筛选条改成更明确的时间范围区域。
- 预设日期按钮、`datetime-local` 输入、应用筛选按钮统一视觉。
- 历史变化列表使用稳定的行组件，保留 `data-history-change-row`。
- 快照对比页强化左右报告、差异行、时间线和空状态。
- 确认无历史、多条历史、多条快照、无差异和有差异场景都能正常展示。

行为变化：筛选控件布局可以改变；URL query、ISO 时间转换、分页语义和“快照对比”跳转不变。

验证方式：用 Playwright 设置开始和结束时间，确认筛选后仍展示变化行并能跳转对比页。历史页和对比页都要保存前后截图，并确认截图中不是加载态。

### 8. 处理嵌入与公开访问场景

目标：确保 Komari 注入链路和游客访问不被后台 UI 重构破坏。

查看页面：`#/public/nodes/:uuid`、`#/nodes/:uuid?embed=1`、`#/connect`，以及 Komari 节点页弹窗。

主要改动：

- 嵌入模式使用独立轻量 shell，避免后台侧边栏和顶部栏进入 iframe。
- 保留透明背景主题兼容，例如 `data-ipq-embed-theme="purcarte"`。
- 公开节点只读展示复用详情组件，但严格隐藏上报配置和用户设置入口。
- 接入桥接页只做样式整理，不改授权流。

行为变化：嵌入页视觉可以更紧凑；游客关闭时 toast、管理员登录返回 Komari 弹窗、公开读开关行为不变。

验证方式：运行 `verify-embed-auth-flows.mjs` 或 `run-e2e.sh`。游客禁用 Toast、游客公开弹窗、管理员登录返回弹窗都要截图并写入验收记录。

### 9. 清理旧样式与重复代码

目标：让新组件体系成为唯一维护入口。

查看页面：所有页面。

主要改动：

- 删除不再使用的 CSS class。
- 合并重复的加载态、错误态、空状态组件。
- 移除未使用依赖和 dead code。
- 保留 `report.tsx`、`display-fields.tsx` 中复杂字段渲染逻辑的稳定性，除非单独测试覆盖。

行为变化：无。

验证方式：`cd web && npm run build`，确认 TypeScript 和 Vite 构建通过；再跑 Playwright 核心流程并查看最终截图，避免清理样式导致页面不可见。

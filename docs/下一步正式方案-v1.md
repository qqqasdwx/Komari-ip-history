## 1. 最终收敛后的关键约束

### 1.1 通知系统

- 通知规则粒度：`节点 + 目标 IP`
- 通知页整体能力：**尽量贴近 Komari，Komari 有的我们都有**
- 字段选择方式：单独做“通知订阅页”
- 所有变化都通知：
  - `A -> B`
  - `N/A -> B`
  - `A -> N/A`
  - tone 变化但文本不变
- Telegram 第一版：`Bot Token + Chat ID`
- Javascript 第一版：按 `Komari` 的 Javascript sender 模式实现
- 如需普通 Webhook，也按 `Komari` 的 sender 一并支持
- 不做通知去重：变化一次发一次

### 1.2 Komari 弹窗职责

- `Komari` 弹窗只能承载“查看当前 IP 质量”
- 后台页面、接入页面、配置页面都不能嵌进 `Komari` 弹窗
- 未接入节点点击图标时：
  - 允许自动注册节点壳子
  - 不自动视为已接入
  - 新标签页打开 IPQ 后台
  - 停在节点列表页并自动打开对应节点配置弹窗
- 配置完成后不自动 reopen `Komari` 弹窗

### 1.3 多 IP 自动探查

- 自动探查为主
- 手动模式继续保留
- 自动与手动允许并存
- 自动发现的目标 IP 要持久化到服务端
- “关闭上报”粒度是单个目标 IP
- 节点不应“探完再被服务端拒绝”，而是：
  - **先和服务端确认，再决定探查哪些 IP**

### 1.4 Cron 与时区

- 执行时区不是固定北京时区
- 执行时区以**用户保存配置时的浏览器时区**为准
- 页面只需要明确：
  - 当前 cron 按哪个时区解析
- 不需要展示节点本地时区

### 1.5 对外 API

- 第一版只做只读 API
- 操作能力继续通过页面完成
- 认证方式：`API Key`
- 对象模型：
  - 节点
  - 目标 IP
  - 历史变化作为附属资源

### 1.6 节点模型

- 未来以 `IPQ 节点` 为主实体
- `Komari` 只是可选绑定来源
- 一个 IPQ 节点最多绑定一个 `Komari` 节点
- 保留现有“从 Komari 直接接入”的方案
- 后台支持手动绑定
- 从 `Komari` 入口进入时可以自动推荐，但不静默直接绑定

## 2. 方案总览

我建议把下一阶段分成 4 个子系统一起设计，但按顺序落地：

1. **访问链路收口**
   - 先修 `Komari` 未接入节点点击行为
   - 先切清弹窗与独立页职责

2. **节点模型升级**
   - 引入 IPQ 节点主实体
   - 引入 Komari 可选绑定

3. **自动探查与调度升级**
   - 节点侧改为“先拿计划，再探查”
   - 服务端接管目标 IP 的启停和发现策略

4. **通知与开放 API**
   - 基于统一变化事件做通知
   - 基于统一对象模型做对外 API

## 3. 访问链路方案

## 3.1 新的职责边界

### Komari 弹窗只做什么

只做：

- 当前 IP 质量查看

不做：

- 登录
- 接入
- 配置
- 节点编辑
- 目标 IP 管理
- 通知设置

### 独立页面做什么

统一承载：

- 登录
- 接入
- 配置
- 节点管理
- 目标 IP 管理
- 通知设置

## 3.2 未接入节点点击流程

目标流程：

1. 用户在 `Komari` 节点页点击图标
2. Header 将 `uuid + 节点名` 上报给 IPQ
3. 服务端如果没有该节点，则创建一个“未完成接入”的节点壳子
4. Header 不打开弹窗
5. 直接新标签页打开：
   - `/#/nodes?report_config=<uuid>`
6. IPQ 节点列表页自动打开该节点的“上报设置”弹窗
7. 用户完成配置后，页面提示：
   - “配置完成，请返回 Komari 重新查看”

## 3.3 需要改的地方

代码层面至少要改：

- Loader 里的未接入跳转策略
- `ConnectPage` 的职责
- embed 模式下的后台路由容忍度
- 节点列表弹窗的打开逻辑

## 4. 节点模型升级方案

## 4.1 新主实体

建议引入：

- `ipq_nodes`
- `komari_bindings`

### `ipq_nodes`

负责表示：

- 我们自己的一条逻辑节点

### `komari_bindings`

负责表示：

- 这条 IPQ 节点是否绑定到了某个 `Komari` 节点

建议关系：

- 一个 IPQ 节点
- 零或一条 Komari 绑定

## 4.2 两种创建方式

### 独立创建

在 IPQ 后台新建节点：

- 可不绑定 `Komari`
- 也可后续绑定一个 `Komari`

### 从 Komari 进入

从 `Komari` 点图标进入时：

- 自动上报 `Komari` 节点信息
- 后台自动推荐一个绑定对象
- 用户确认后完成绑定/接入

不做静默绑定。

## 5. 自动探查与多 IP 方案

## 5.1 目标

从“服务端先维护一串目标 IP，再让节点逐个探查”改成：

- 节点先枚举候选 IP
- 服务端下发探查计划
- 节点只探查被允许的 IP
- 服务端持久化这些目标 IP 与结果

## 5.2 核心变化：先拿计划，再探查

这一条是你新补充后最关键的设计约束。

建议新增一个“计划获取”接口：

- 节点先把：
  - 节点身份
  - 本次枚举到的候选 IP
  - 版本信息
  - 主机信息（可选）
  发给服务端

- 服务端返回：
  - 哪些 IP 需要探查
  - 哪些 IP 已关闭上报
  - 哪些 IP 需要新建为自动发现目标
  - 本次调度配置

这样可以避免：

- 节点白跑一堆不该探的 IP
- 服务端在“关闭上报”后还被动接收结果

## 5.3 目标 IP 模型

每个目标 IP 建议增加：

- `source`
  - `manual`
  - `discovered`

- `enabled`
  - `true/false`

### 行为

- `manual`
  - 后台人工添加

- `discovered`
  - 节点枚举上报后由服务端创建

关闭上报时：

- 服务端在“计划获取”阶段就不再把这个 IP 下发给节点

### 当前实现备注（截至 2026-04-17）

当前代码已经具备：

- `POST /api/v1/report/nodes/:uuid/plan`
- `NodeTarget.source`
- `NodeTarget.enabled`
- `NodeTarget.last_seen_at`
- 单目标 IP 启停 UI

但这里仍有一条需要文档明确保留为“待修正”：

- 计划接口当前只会返回本次 `candidate_ips` 中命中的目标 IP
- 这意味着“手动添加但不在本机枚举结果里的目标 IP”会被漏掉
- 因此现状还**不满足**“手动模式继续保留，自动与手动允许并存”这条方案约束

## 5.4 候选 IP 枚举策略

因为你已经明确“不按公网/私网简单裁剪”，第一版建议采用：

- 排除：
  - loopback
  - link-local
  - down 接口
  - 明显无效虚拟地址

- 保留：
  - RFC1918 私网地址
  - NAT 机器上的出口地址
  - 其他可能作为实际探查出口的绑定地址

这里需要我们在节点侧做一层 IP 过滤器，不直接照搬 `IPQuality`。

## 5.5 与手动模式并存

第一版明确支持：

- 自动发现 IP
- 手动补充 IP

两类目标 IP 在 UI 上建议可区分：

- 手动
- 自动发现

## 6. Cron 与时区方案

## 6.1 执行语义

现在的正确理解应当是：

> Cron 按用户保存配置时的浏览器时区解析，而不是按节点本地时区，也不是固定按北京时区。

这意味着保存配置时，除了 cron 表达式，还要同时保存：

- `timezone`

例如：

- `Asia/Shanghai`
- `America/Los_Angeles`

来源建议直接取：

- `Intl.DateTimeFormat().resolvedOptions().timeZone`

## 6.2 节点执行

节点端最终不要依赖机器默认时区，而是显式设置：

- `CRON_TZ=<saved_timezone>`

这样用户填“每天 12 点”时，节点就会按保存时的那个时区执行。

## 6.3 页面显示

页面建议显示：

- 当前 cron 表达式
- 当前解析时区
- 最近 10 次执行时间

执行时间可以按浏览器当前时区友好显示，但必须同时注明：

- “按 `<timezone>` 解析”

不能只展示一个本地时间，不然用户还是会误解。

## 7. 通知系统方案

## 7.1 事件层

通知系统应建立在统一变化事件之上。

建议统一事件字段：

- `node_id`
- `node_name`
- `komari_node_uuid`
- `target_id`
- `target_ip`
- `field_id`
- `field_label`
- `group_path`
- `previous_value`
- `current_value`
- `previous_recorded_at`
- `recorded_at`
- `history_entry_id`
- `detail_url`
- `compare_url`

## 7.2 规则层

建议 `notification_rule` 至少包含：

- `node_id`
- `target_id`
- `field_id`
- `channel_type`
- `channel_config_id`
- `enabled`

## 7.3 页面整体形态

通知设置页整体建议直接对齐 `Komari` 的通知页结构。

也就是说，第一版尽量具备这些部分：

- 通知总开关
- 消息模板
- 发送方式选择
- 发送器配置表单
- 测试发送
- 我们自己的字段变化订阅规则

其中：

- “发送器配置”这一层尽量复用 `Komari` 的概念和命名
- “字段变化订阅”是我们在其上新增的一层业务规则

## 7.4 Telegram

第一版仅支持：

- `bot_token`
- `chat_id`

建议字段尽量贴近 `Komari`：

- `bot_token`
- `chat_id`
- `message_thread_id`
- `endpoint`

## 7.5 Javascript

你已经明确不要单独发明“Hook 脚本”概念，而是直接按 `Komari` 的 `Javascript` sender 来。

也就是：

- 用户填写一段 Javascript 代码
- 代码实现：
  - `sendMessage(message, title)`
  - 可选 `sendEvent(event)`
- 可用能力尽量贴近 `Komari`
  - `fetch()`
  - `xhr()`
  - `console.log()`

我们这里不再把它额外定义成一个新的 Hook 系统，而是直接复用 `Komari` 现有的发送器思路。

## 7.6 Webhook

因为 `Komari` 本身也提供 `Webhook` sender，建议我们一并支持。

也就是说第一版发送器建议直接做这三种：

- `Telegram`
- `Javascript`
- `Webhook`

这样才符合：

> Komari 有的我们都有

同时也给不想写 Javascript 的用户一个直接可用的通道。

## 7.7 与 Komari 的兼容策略

通知页建议按下面的原则做：

- 名称尽量贴近 `Komari`
- 配置项命名尽量贴近 `Komari`
- Javascript sender 接口尽量贴近 `Komari`
- Webhook sender 配置项尽量贴近 `Komari`

但在此基础上增加我们自己的核心能力：

- 按字段变化订阅
- 按节点 + 目标 IP 订阅

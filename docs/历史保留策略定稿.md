# 历史保留策略定稿

更新时间：2026-04-04

本文档用于固定“历史快照保留与自动清理”的产品和技术口径。  
后续实现以本文档为准，不再临时讨论“要不要删历史、怎么删、收藏算不算例外”。

---

## 1. 目标

历史功能继续以**完整快照**为存储基础，不改成 diff-only。  
保留策略解决的是：

- 历史数据无限增长
- 快照对比页首次加载越来越慢
- SQLite 文件持续膨胀

同时不破坏已经确定的产品体验：

- 历史主页面仍然看字段变化
- 快照对比页仍然优先保持本地顺滑交互
- 关键时间点快照允许长期保留

---

## 2. 产品口径

### 2.1 保留范围

- 保留策略只作用于**历史快照**
- 不作用于：
  - 节点
  - 目标 IP
  - 当前结果 `current_result`

也就是说：

- 即使历史被清理
- 当前结果仍然保留

### 2.2 配置粒度

- 只做**全局保留策略**
- 不做：
  - 单节点保留策略
  - 单 IP 保留策略
  - 手动清理按钮

### 2.3 配置值

- 配置项为：**历史保留天数**
- `-1` 表示：**永久保留**
- `1` 及以上表示按滑动窗口保留最近 N 天历史

### 2.4 默认值

- 默认值：`-1`

原因：

- 当前已有数据模型就是长期保留
- 升级后不应默认删旧数据
- 用户主动设置后，再开始进入自动清理模式

### 2.5 收藏快照例外

- 收藏对象是**单条历史快照**
- 已收藏快照：
  - 不受历史保留策略删除
- 取消收藏后：
  - 重新受全局保留策略约束

### 2.6 自动清理方式

- 保留策略按**滑动窗口**自动生效
- 不做“保存后手动执行清理”
- 不做“立即清理”按钮

### 2.7 用户提示

页面上要明确提示：

- 当前历史占用空间是估算值
- `-1` 表示永久保留
- 永久保留会让：
  - 磁盘占用持续增长
  - 历史查询和快照对比逐渐变慢

---

## 3. 页面与交互

### 3.1 入口位置

第一版放在：

- `设置 / 历史保留`

原因：

- 历史保留属于全局系统配置
- 与“接入配置”混在一起会打断生成 Header 的主流程
- 单独成页后，设置职责更清晰

### 3.2 页面内容

在“历史保留”页提供一张独立卡片：

- 标题：`历史保留`
- 说明：历史快照自动清理规则

卡片内至少包含：

- 历史保留天数输入
- `-1` 说明文案
- 当前历史占用空间
- 最近增长速度估算
- 当前配置下的预计保留体积
- 单独保存按钮

### 3.3 表单交互

- 保留天数输入为整数
- 允许输入：
  - `-1`
  - `1` 及以上
- 不允许：
  - `0`
  - 小数
  - 非数字

### 3.4 展示文案

建议在卡片内明确提示：

- `-1` 表示永久保留
- 历史越多，快照对比加载越慢
- 收藏快照不会被自动清理

## 4. 后端实现口径

### 4.1 设置项

新增全局设置项：

- `history_retention_days`

存储方式继续使用现有 `AppSetting`。

### 4.2 读取接口

建议新增管理端接口：

- `GET /api/v1/admin/history-retention`

返回内容建议包括：

- `retention_days`
- `history_bytes`
- `recent_growth_bytes_per_day`
- `estimated_retained_bytes`
- `estimated_is_unbounded`

### 4.3 保存接口

建议新增管理端接口：

- `PUT /api/v1/admin/history-retention`

请求体：

- `retention_days`

### 4.4 清理执行时机

清理任务建议：

- 服务启动后执行一次
- 之后按固定周期执行

第一版建议周期：

- 每 `6` 小时

这样足够简单，也足够满足滑动窗口语义。

### 4.5 清理条件

当 `history_retention_days != -1` 时：

- 计算保留窗口起点 `cutoff`
- 删除所有满足下面条件的历史快照：
  - `recorded_at < cutoff`
  - `is_favorite = false`

### 4.6 不应受影响的数据

清理时不应修改：

- `node_target.current_result_json`
- `node_target.current_result_updated_at`
- 节点与目标 IP 本身

### 4.7 删除目标 IP 时

如果管理员手动删除某个目标 IP：

- 该 IP 的所有历史一并删除
- 即使其中有收藏快照，也跟着删除

原因：

- 目标 IP 被删除，说明这条探查目标本身不再存在
- 收藏快照不应凌驾于“删除目标 IP”这个显式管理动作之上

---

## 5. 容量估算口径

### 5.1 当前占用

当前历史占用按历史表本身估算：

- `sum(length(result_json) + length(summary))`

这是估算，不是 SQLite 文件的真实磁盘块占用。

### 5.2 增长速度

建议按最近 7 天历史写入量估算：

- 最近 7 天历史字节数 / 7

如果近 7 天没有足够数据，则退化为：

- 当前总量提示
- 不提供可靠趋势估算

### 5.3 预计占用

当 `retention_days != -1` 时，显示：

- 基于最近增长速度估算的预计保留体积

当 `retention_days = -1` 时，显示：

- `长期增长，无法给出上限`

---

## 6. 明确不做的内容

当前不做：

- 单节点保留策略
- 单 IP 保留策略
- 手动立即清理
- 改成只存 diff
- 为快照对比页牺牲交互体验去做按需请求

---

## 7. 与现有功能的关系

这套保留策略与当前功能的关系是：

- 历史主页面：不变，只是数据源未来会被自动清理
- 快照对比页：不变，只是长期历史量会被控制
- 收藏快照：作为全局保留策略的唯一例外

也就是说：

> 保留策略的目的是控制数据规模，  
> 不是改变当前历史与快照对比的交互模型。

## 1. 已确认的方向

### 1.1 通知系统

已确认：

- 通知规则粒度为：**节点 + 目标 IP**
- 字段选择方式：**单独做专门的字段订阅页**
- 变化定义：以下都通知
  - `A -> B`
  - `N/A -> B`
  - `A -> N/A`
  - 文本不变但 tone 变化
- Telegram 第一版采用：
  - `Bot Token + Chat ID`
- Hook 第一版采用：
  - 用户可注入自定义代码 / 自定义调用逻辑
- 不做通知去重：
  - 变化一次发一次

### 1.2 Komari 未接入节点点击图标

已确认：

- `Komari` 弹窗只负责“查看当前 IP 质量”
- 后台页面、接入页面、配置页面都不应嵌入 `Komari` 弹窗
- 未接入时允许自动注册节点壳子
- 但不自动视为“已接入完成”
- 未接入时应：
  - 新标签页打开 IPQ 后台
  - 停在节点列表页
  - 自动打开对应节点的配置弹窗
- 配置完成后：
  - 不自动 reopen `Komari` 弹窗
  - 只提示用户返回 `Komari` 重新查看

### 1.3 多 IP 自动探查

已确认：

- 自动探查为主，但保留手动模式
- 自动模式与手动模式允许并存
- 服务端自动发现的 IP 需要持久化
- “关闭上报”按钮粒度为：
  - **单个目标 IP**

对“自动枚举范围”的确认是：

- 不按“公网 / 私网”简单裁剪
- 只排除明显不是出口用途的地址
- NAT 机器上的内网地址也可能是有效探查目标

### 1.4 Cron 与时区

已确认：

- 用户填写 cron 时，希望按“北京时间语义”执行
- 不是按节点机器自身 UTC 时区解释
- 页面上不需要展示节点本地时区
- 只需要明确显示“cron 按哪个时区解析”

### 1.5 对外 API

已确认：

- 第一版只做**只读 API**
- 操作能力继续通过页面完成
- 认证方式采用：
  - `API Key`
- 对象模型采用：
  - 节点
  - 目标 IP
  - 历史变化作为附属资源

### 1.6 节点模型升级

已确认：

- 未来以 **IPQ 节点** 为主实体
- `Komari` 只是可选绑定来源
- 一个 IPQ 节点与 `Komari` 的关系为：
  - **一对零或一**
- 保留现有“从 `Komari` 直接接入”的方案
- 后台支持手动绑定
- 从 `Komari` 进来时可以做自动推荐，但不静默自动绑定

## 2. 草案方案

## 2.1 通知系统草案

### 数据模型

建议新增以下概念：

- `notification_rule`
  - 绑定到 `node_id + target_id`
  - 指定订阅字段
  - 指定通知通道
  - 可启停

- `notification_channel`
  - `telegram`
  - `hook`

- `notification_delivery_log`
  - 记录每次投递结果
  - 方便排障

### 变化事件模型

通知使用统一事件模型，字段至少包括：

- `node_id`
- `node_name`
- `komari_node_uuid`
- `target_id`
- `target_ip`
- `field_id`
- `field_label`
- `group_path`
- `previous_value`
- `current_value`
- `previous_recorded_at`
- `recorded_at`
- `history_entry_id`
- `detail_url`
- `compare_url`

### Telegram

第一版仅支持：

- `bot_token`
- `chat_id`

发送内容建议采用统一模板文本，不先做复杂可视化自定义。

### Hook

因为你已经明确希望支持“注入代码”方式，建议 Hook 第一版拆成两层：

1. **标准事件 JSON**
   - 服务端固定产出

2. **用户注入脚本**
   - 用户在 Hook 配置中写自己的请求逻辑
   - 平台负责把标准事件对象传进去

这里要注意：

- 第一版就做“任意注入代码”风险会比较高
- 更稳妥的落地方式是：
  - 先做“固定 JSON + 可自定义 header + 可自定义 URL”
  - 后面再扩到“脚本注入”

但如果你要一步到位支持注入代码，建议明确这是“高级模式”。

### 页面

建议在节点详情或上报设置中增加：

- `通知` 入口

进入后管理：

- 通知规则列表
- 新建规则
- 选择目标 IP
- 选择字段
- 选择 Telegram 或 Hook

## 2.2 Komari 未接入节点点击行为草案

### 新的职责边界

统一收口为：

- `Komari` 弹窗：
  - 只显示当前 IP 质量查看页面

- 独立页面：
  - 登录
  - 接入
  - 配置
  - 目标 IP 管理
  - 通知设置

### 新交互

未接入节点点击图标时：

1. Header 上报 `Komari` 节点 UUID 和名称
2. 服务端若没有对应节点，则创建一个“未完成接入”的节点壳子
3. 不打开 `embed` 弹窗
4. 直接新标签页打开：
   - `/#/nodes?report_config=<komari_uuid>`
5. 节点列表页打开该节点的配置弹窗
6. 用户完成配置后，页面提示：
   - “配置完成，请返回 Komari 重新查看”

### 代码层影响

这会要求我们把当前 `embed` 路由里的“继续登录 / 继续接入 / 自动回跳”逻辑拆掉。

要点是：

- `embed` 页面不再承担后台行为
- `ConnectPage` 的职责需要收缩
- `NodeDetailPage` 的 embed 模式只允许查看

## 2.3 多 IP 自动探查草案

### 目标

把“目标 IP 来源”从单一的服务端手动维护，改成：

- 自动探查为主
- 手动维护为兼容

### 核心模型

建议把目标 IP 分成两类：

- `manual`
  - 后台手动添加

- `discovered`
  - 节点自动枚举上报后由服务端持久化

并为每个目标 IP 增加：

- `enabled`
  - 用于“关闭上报”

### 节点侧探查器

由于上游 `IPQuality` 只明确支持：

- 指定网卡
- 指定出口 IP

所以我们需要在节点侧自己做一层包装：

1. 枚举候选 IP
2. 逐个调用 `IPQuality`
3. 把结果逐个上报给服务端

### 自动枚举建议

不能简单按“是否公网”过滤。

更合理的第一版规则是：

- 排除：
  - loopback
  - 明显无效接口
  - link-local
  - down 状态接口
- 不因为是 RFC1918 私网地址就直接排除
- 允许 NAT 机器上的出口内网地址被探查

### 服务端行为

服务端收到一个新的目标 IP 时：

- 如果节点下还没有这个 IP，则自动持久化为 `discovered`
- 如果已有同 IP 目标，则更新当前结果和历史
- 如果该 IP 被手动关闭上报，则忽略新的同目标上报或做显式拒绝

### 并存模式

第一版建议明确支持：

- 自动发现 IP
- 同时继续允许管理员手动补充目标 IP

不要一开始就强制迁移或移除手动模式。

## 2.4 Cron 与时区草案

### 你想要的实际语义

你的目标不是“节点按自己的本地时区执行”，而是：

> 用户按北京时间填写，节点就必须按北京时间执行。

这意味着：

- cron 解释时区应该是一个**明确指定的时区**
- 不能依赖节点系统时区

### 我建议的落地方式

> 这部分旧草案已被后续确认覆盖。

当前已确认并进入实现的方向是：

- 不固定为 `Asia/Shanghai`
- 而是按**用户保存配置时的浏览器时区**解释 cron
- 页面明确写“当前 cron 按哪个时区解析”
- 节点侧实际 cron 配置写入相同时区的：
  - `CRON_TZ=<timezone>`

### 页面显示

页面建议同时展示：

- 规则说明：
  - `当前 Cron 按 Asia/Shanghai 解析`
- 下一次执行时间：
  - 浏览器本地显示
  - 但附带说明其源时区是 `Asia/Shanghai`

这样用户看到的是：

- 自己易读的本地显示
- 但不会误以为系统按浏览器时区执行

### 当前实现备注

当前页面已经：

- 显示“当前规则按哪个时区解析”
- 预览最近 10 次执行时间
- 安装脚本写出 `CRON_TZ`

但仍有一个待补齐项：

- 目前没有手动覆盖时区的编辑控件
- 当前版本实际上是“按浏览器时区显示并保存”

## 2.6 节点模型升级草案

### 新主实体

未来以 `IPQ Node` 为主实体。

建议拆成：

- `ipq_nodes`
- `komari_bindings`

### 关系

- 一个 `IPQ Node`
- 可选绑定一个 `Komari Node`

### 支持两种创建方式

#### 方式 A：独立创建

- 在 IPQ 后台新建节点
- 可选绑定 `Komari`

#### 方式 B：从 Komari 直接进入

- 点击图标
- 自动带入 `Komari` 节点信息
- 在 IPQ 后台完成接入

### 绑定方式

建议：

- 后台支持手动绑定
- 从 `Komari` 入口进入时，系统自动推荐可绑定对象
- 但不静默直接绑定

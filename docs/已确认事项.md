# 已确认事项

本文档记录当前已经拍板的产品与技术结论。  
后续如有新增结论，优先更新本文件。

> 架构级口径以 `docs/Komari集成方式重定稿.md` 为准。  
> 注入弹窗鉴权口径以 `docs/注入弹窗鉴权定稿.md` 为准。  
> 历史功能口径以 `docs/历史视图定稿.md` 为准。  
> 文档阅读顺序见 `docs/文档索引.md`。

---

## 1. 项目形态

- 本项目是**独立服务**，不是只靠注入脚本。
- 独立服务包含：
  - 后端
  - 独立 Web 前端
- 独立前端用于：
  - 登录
  - 节点列表 / 节点详情
  - 系统配置
  - 生成可复制的 `Komari header`

---

## 2. 与 Komari 的关系

- 不修改 `Komari` 官方源码。
- 通过 `Komari` 自定义 header 注入少量 JavaScript 实现集成。
- 注入代码尽量少，只做：
  - 当前页面识别
  - 按钮或入口插入
  - 节点信息提取
  - 打开弹窗或跳转到本服务
- 复杂展示逻辑尽量由本服务承载，不写死在注入代码中。

---

## 3. 部署与集成口径

- 默认按**独立部署**设计。
- 不要求与 `Komari` 同域。
- 不要求用户额外配置统一入口反代才能使用。
- `/ipq/` 子路径可以兼容，但不再作为默认前提。
- Header 生成应支持基于本服务**公开访问地址**输出绝对地址。

---

## 4. Header 生成方式

- 本服务需要提供“一键复制 Komari header”的功能。
- 同时提供两种可复制内容：
  - 短 loader 版
  - 完整内联版
- 默认推荐短 loader 版。
- 需要明确提示用户：完整内联版更新后需要重新复制。
- Header 默认应根据本服务公开地址生成，不再默认依赖同源相对路径。

---

## 5. 节点接入方式

- 不以“自动拉取 Komari 全量节点列表”作为当前主方案。
- 不使用 `Komari API Key` 作为主接入方案。
- 不长期保存 `Komari` 管理员账号密码用于同步节点列表。
- 改为在 `Komari` 节点详情页中注入按钮，由用户主动接入当前节点。
- 按钮点击时，从当前页面获取节点 `uuid` 和可用的节点名称。
- 本服务据此创建节点记录并纳入管理。
- 节点接入按幂等处理：
  - 已存在则不报错
  - 直接视为已接入

---

## 6. 入口按钮与交互

- 阶段 1 的入口位置放在节点详情页操作区。
- 未接入节点显示：`添加 IP 质量检测`
- 已接入节点显示：`查看 IP 质量`
- “添加”操作默认立即执行，不再额外弹确认框。

---

## 7. 展示方式

- 产品体验上，**优先争取页内弹窗**。
- 如果页内弹窗受限或不稳定，使用**独立页面兜底**。
- 当前最重要的展示内容是：
  - 当前 IP 质量总览
- 历史记录也是当前主功能之一，不再只是后台保留能力。
- 独立页面不是主体验目标，但必须存在，保证产品可用性。

---

## 8. 登录与会话

- 本服务需要登录鉴权。
- 本服务采用单管理员模型，不做普通用户体系。
- 本服务登录采用服务端 session + HttpOnly cookie。
- `Komari` 登录态与本服务登录态是两套独立会话。
- 管理员如果在 `Komari` 中点击入口时尚未登录本服务：
  - 允许跳转到本服务独立登录页
  - 登录后继续在本服务中完成后续动作
- 游客链路不复用本服务后台登录。

---

## 9. 管理员初始化

- 默认管理员账号：`admin`
- 默认管理员密码：`admin`
- 首次登录后不强制修改密码
- 管理员用户名和密码都允许修改

---

## 10. 数据库与存储

- 数据库使用 `SQLite`。
- 需要保存：
  - 管理员账号
  - 节点记录
  - 节点下目标 IP
  - 每个目标 IP 的最近一次结果
  - 每个目标 IP 的历史
  - 系统配置
- 节点主记录以 `komari_node_uuid` 作为唯一键。
- 节点接入后允许暂时没有任何目标 IP。
- 同一节点下目标 IP 必须唯一。
- 目标 IP 支持排序。
- 手动删除目标 IP 时，该 IP 的最近一次结果与历史一并删除。

---

## 11. 数据模型方向

- 数据模型参考 `references/IPQuality/`。
- 阶段 1 允许使用模拟数据打通展示链路。
- 真实使用阶段如果没有真实数据，应显示 `N/A`，不能用模拟值伪装成真实结果。
- 检测结果存储以原始 JSON 为主，不强行把所有字段拆成固定数据库列。
- 只抽取少量稳定字段作为索引字段，其余详细内容从 JSON 动态展示。

---

## 12. 展示重点

- 当前产品重点包括：
  - 当前 IP 质量结果
  - 按目标 IP 查看历史
- 页面不应充满与用户无关的内部元数据。
- 后台整体风格尽量贴近 `Komari`。
- 节点结果区尽量贴近 `IPQuality` 原始报告风格。
- “变化规则 / 趋势分析”不再作为当前主产品方向，但历史记录本身属于主功能。
- 历史主页面的主视角应是**字段变化列表**，不是快照列表。

---

## 13. 独立前端结构

- 本服务登录后默认进入节点列表页。
- 节点列表只显示已从 `Komari` 主动接入的节点。
- 节点列表至少支持按“节点名称 + UUID”搜索。
- 节点列表不再显示节点摘要，也不显示 IP 数量。
- 节点详情页按“目标 IP 标签页”组织：
  - 默认显示排序后的第一个目标 IP
  - 支持添加 IP
  - 支持删除 IP
  - 支持调整 IP 顺序
- 节点详情页需要同时提供：
  - 当前选中 IP 的最近一次结果
- 历史不再在详情页里做摘要主视图。
- 完整历史进入独立历史页查看。
- 独立历史页需要支持：
  - 按目标 IP 切换
  - 字段变化列表
  - 时间范围筛选
  - 字段下拉筛选
- 快照对比需要独立页面承载：
  - 顶部双滑块时间轴
  - 左右双栏快照
  - 差异高亮
- 没有任何目标 IP 时，详情页显示空状态并提示先添加 IP。
- 独立前端需要保留系统配置页，用于：
  - 生成 header
  - 展示运行模式与公开地址
  - 历史保留策略
  - 用户改名改密

---

## 14. API 方向

- 后端接口采用 REST 风格。
- 接口带版本前缀，使用 `/api/v1/...`。
- 注入侧可以先调用轻量状态接口判断节点是“添加”还是“查看”。
- 管理员接口与游客只读接口需要分离。
- 详情数据只返回：
  - 节点基础信息
  - 目标 IP 列表
  - 当前选中 IP 的最近一次结果
- 历史数据由独立接口提供，并按目标 IP 查询。
- 上报接口采用：
  - 节点级 reporter token
  - 每次单 IP 单次上报
  - 请求中必须带 `target_ip`
- `target_ip` 不在节点当前配置列表里时，后端直接拒绝。

---

## 15. 开发环境

- 当前推荐采用双轨开发方式：
  - Debian 原生日常开发
  - Docker 联调 / E2E / CI
- Docker 继续保留，但不再作为唯一开发方式。
- 开发环境仍应尽量做到首次启动即可使用。
- 默认管理员账号维持 `admin/admin`。

---

## 16. 当前默认原则

- 先跑通最小闭环。
- 尽量减少对 `Komari` 高权限接口的依赖。
- 能通过当前页面拿到的信息，就不要额外向 `Komari` 请求高权限数据。
- 能在本服务完成的逻辑，就不要塞进注入脚本。
- 产品上优先争取页内弹窗，工程上必须保留独立页兜底。

---

## 17. 注入弹窗鉴权原则

- `Komari` 注入弹窗链路分为：
  - 游客只读链路
  - 管理员完整链路
- 当前身份判断不依赖 DOM，统一以 `Komari` 自身接口结果为准。
- 游客只读全局开关默认关闭。
- 游客只读只影响 `Komari` 注入弹窗，不影响 IPQ 独立页访问规则。
- 游客只读开启后：
  - 只允许看当前结果
  - 但仍需跟随 `Komari` 节点是否对游客公开
- 游客链路不触发节点接入，不进入后台管理。
- 管理员链路仍使用 IPQ 自身登录体系。
- 管理员若未登录 IPQ，应跳独立页登录；登录成功后继续之前的操作。
- 未接入节点只允许管理员接入。
- 详情接口与历史接口分离：
  - 详情只返回当前状态
  - 历史走独立接口

## 18. 多 IP 与历史原则

- 一个节点下可以配置多个手动维护的目标 IP。
- 这些目标 IP 不是自动发现，而是管理员手动添加。
- 节点端执行的是一条统一命令，但命令对应当前节点下的完整 IP 集合。
- 实际上报按目标 IP 单独进行：
  - 一个 IP 一次上报
  - 一个 IP 一条最近结果
  - 一个 IP 一组历史
- 不做心跳，不做在线状态判断。
- 所谓“当前结果”，指的是该目标 IP 最近一次成功上报的结果。
- 历史继续保存完整快照，不改成只存 diff。
- 历史清理采用全局保留策略：
  - 默认值为 `-1`
  - `-1` 表示永久保留
  - 非收藏快照按全局窗口自动清理
- 快照可被管理员收藏：
  - 收藏后的快照不受历史保留策略删除
  - 取消收藏后重新受全局策略约束

## 19. 历史展示原则

- 历史变化按字段独立记录：
  - 一次上报里如果有 20 个字段变化，就显示 20 条变化记录
- 历史主页面默认按时间倒序展示全部字段变化
- 不传字段筛选参数时，表示查询全部字段
- 字段筛选不使用自由文本搜索，使用展示名下拉选择
- 历史变化行至少包含：
  - 变化时间
  - 细版分组路径
  - 字段名
  - 旧值（旧时间）
  - 新值（新时间）
- 历史页面与快照对比页严禁直接显示技术字段路径
- 所有字段名、值翻译、颜色和 badge 样式必须与结果展示页统一
- `A -> N/A` 也要记录，但应标记为“可能缺失”

# 已确认事项

本文档记录当前已经拍板的产品与技术结论。  
后续如有新增结论，优先更新本文件。

> 架构级口径以 `docs/Komari集成方式重定稿.md` 为准。  
> 文档阅读顺序见 `docs/文档索引.md`。

---

## 1. 项目形态

- 本项目是**独立服务**，不是只靠注入脚本。
- 独立服务包含：
  - 后端
  - 独立 Web 前端
- 独立前端用于：
  - 登录
  - 节点列表 / 节点详情
  - 系统配置
  - 生成可复制的 `Komari header`

---

## 2. 与 Komari 的关系

- 不修改 `Komari` 官方源码。
- 通过 `Komari` 自定义 header 注入少量 JavaScript 实现集成。
- 注入代码尽量少，只做：
  - 当前页面识别
  - 按钮或入口插入
  - 节点信息提取
  - 打开弹窗或跳转到本服务
- 复杂展示逻辑尽量由本服务承载，不写死在注入代码中。

---

## 3. 部署与集成口径

- 默认按**独立部署**设计。
- 不要求与 `Komari` 同域。
- 不要求用户额外配置统一入口反代才能使用。
- `/ipq/` 子路径可以兼容，但不再作为默认前提。
- Header 生成应支持基于本服务**公开访问地址**输出绝对地址。

---

## 4. Header 生成方式

- 本服务需要提供“一键复制 Komari header”的功能。
- 同时提供两种可复制内容：
  - 短 loader 版
  - 完整内联版
- 默认推荐短 loader 版。
- 需要明确提示用户：完整内联版更新后需要重新复制。
- Header 默认应根据本服务公开地址生成，不再默认依赖同源相对路径。

---

## 5. 节点接入方式

- 不以“自动拉取 Komari 全量节点列表”作为当前主方案。
- 不使用 `Komari API Key` 作为主接入方案。
- 不长期保存 `Komari` 管理员账号密码用于同步节点列表。
- 改为在 `Komari` 节点详情页中注入按钮，由用户主动接入当前节点。
- 按钮点击时，从当前页面获取节点 `uuid` 和可用的节点名称。
- 本服务据此创建节点记录并纳入管理。
- 节点接入按幂等处理：
  - 已存在则不报错
  - 直接视为已接入

---

## 6. 入口按钮与交互

- 阶段 1 的入口位置放在节点详情页操作区。
- 未接入节点显示：`添加 IP 质量检测`
- 已接入节点显示：`查看 IP 质量`
- “添加”操作默认立即执行，不再额外弹确认框。

---

## 7. 展示方式

- 产品体验上，**优先争取页内弹窗**。
- 如果页内弹窗受限或不稳定，使用**独立页面兜底**。
- 当前最重要的展示内容是：
  - 当前 IP 质量总览
- 独立页面不是主体验目标，但必须存在，保证产品可用性。

---

## 8. 登录与会话

- 本服务需要登录鉴权。
- 本服务采用单管理员模型，不做普通用户体系。
- 本服务登录采用服务端 session + HttpOnly cookie。
- `Komari` 登录态与本服务登录态是两套独立会话。
- 如果用户在 `Komari` 中点击入口时尚未登录本服务：
  - 允许跳转到本服务登录页
  - 登录后继续在本服务中完成后续动作

---

## 9. 管理员初始化

- 默认管理员账号：`admin`
- 默认管理员密码：`admin`
- 首次登录后不强制修改密码
- 管理员用户名和密码都允许修改

---

## 10. 数据库与存储

- 数据库使用 `SQLite`。
- 需要保存：
  - 管理员账号
  - 节点记录
  - 当前结果
  - 展示配置
- 节点主记录以 `komari_node_uuid` 作为唯一键。
- 节点接入时阶段 1 只要求至少保存：
  - `uuid`
  - 节点名称

---

## 11. 数据模型方向

- 数据模型参考 `references/IPQuality/`。
- 阶段 1 允许使用模拟数据打通展示链路。
- 真实使用阶段如果没有真实数据，应显示 `N/A`，不能用模拟值伪装成真实结果。
- 检测结果存储以原始 JSON 为主，不强行把所有字段拆成固定数据库列。
- 只抽取少量稳定字段作为索引字段，其余详细内容从 JSON 动态展示。

---

## 12. 展示重点

- 当前产品重点是**当前 IP 质量结果**。
- 页面不应充满与用户无关的内部元数据。
- 后台整体风格尽量贴近 `Komari`。
- 节点结果区尽量贴近 `IPQuality` 原始报告风格。
- “历史变化 / 变化视图 / 趋势分析”不再作为当前主产品方向。

---

## 13. 独立前端结构

- 本服务登录后默认进入节点列表页。
- 节点列表只显示已从 `Komari` 主动接入的节点。
- 节点列表至少支持按“节点名称 + UUID”搜索。
- 节点详情页以“当前结果”作为主体。
- 独立前端需要保留系统配置页，用于：
  - 生成 header
  - 展示运行模式与公开地址
  - 管理员改名改密

---

## 14. API 方向

- 后端接口采用 REST 风格。
- 接口带版本前缀，使用 `/api/v1/...`。
- 注入侧可以先调用轻量状态接口判断节点是“添加”还是“查看”。
- 详情数据由单独节点详情接口提供。

---

## 15. 开发环境

- 当前推荐采用双轨开发方式：
  - Debian 原生日常开发
  - Docker 联调 / E2E / CI
- Docker 继续保留，但不再作为唯一开发方式。
- 开发环境仍应尽量做到首次启动即可使用。
- 默认管理员账号维持 `admin/admin`。

---

## 16. 当前默认原则

- 先跑通最小闭环。
- 尽量减少对 `Komari` 高权限接口的依赖。
- 能通过当前页面拿到的信息，就不要额外向 `Komari` 请求高权限数据。
- 能在本服务完成的逻辑，就不要塞进注入脚本。
- 产品上优先争取页内弹窗，工程上必须保留独立页兜底。

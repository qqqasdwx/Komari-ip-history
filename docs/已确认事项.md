# 已确认事项

本文档记录目前已经确认的产品与技术决策，后续继续沟通时以此为准。

## 1. 项目形态

- 本项目有独立服务，不只是注入脚本。
- 独立服务包含后端和独立 Web 前端。
- 独立 Web 前端用于配置、展示、生成可复制的 `Komari header`。
- 主要使用体验仍然希望发生在 `Komari` 内，而不是要求用户长期单独打开本服务页面。

## 2. 与 Komari 的集成方式

- 不修改 `Komari` 官方源码。
- 通过 `Komari` 的自定义 header 注入 JavaScript 实现集成。
- 注入代码应尽量少，尽量只负责桥接、挂载和调用。
- 样式和复杂展示逻辑尽量由本服务承载，不要写死在注入代码中。
- `Komari` 与本服务默认按同域部署设计，阶段 1 不考虑跨域。

## 3. Header 生成方式

- 本服务需要提供“一键复制 Komari header”的功能。
- 同时提供两种可复制内容：
- 短 loader 版
- 完整内联版
- 默认推荐短 loader 版。
- 需要明确提示用户：完整内联版后续更新时需要重新复制粘贴。
- 注入到 `Komari` 的 loader 脚本默认按相对当前站点路径生成，不默认写死完整域名。
- 默认基于当前站点同源地址和服务路径前缀生成 header。
- 生产特殊场景下可再允许手动配置绝对地址。

## 4. Komari 节点接入方式

- 不再以“自动拉取 Komari 全量节点列表”作为阶段 1 主方案。
- 不使用 `Komari API Key` 作为主接入方案。
- 不长期保存 `Komari` 管理员账号密码用于同步节点列表。
- 改为在 `Komari` 节点详情页中注入按钮，由用户主动点击将当前节点注册到本服务。
- 按钮点击时，从当前页面获取节点 `uuid`，发送给本服务。
- 本服务据此创建节点记录并纳入管理。
- 独立前端页面中的节点列表只显示已经从 `Komari` 添加过的节点，不提供手工录入节点 UUID。

## 5. 按钮与交互

- 阶段 1 的入口位置放在节点详情页操作区按钮。
- 未接入节点显示 `添加 IP 质量检测`。
- 已接入节点显示 `查看 IP 质量`。
- 点击 `添加 IP 质量检测` 后，默认立即在本服务中创建该节点记录，不再额外弹确认框。
- “添加节点”操作按幂等处理。
- 如果节点已存在，再次点击“添加”不报错，直接返回“已接入”。
- 后续产品会涉及“IP 质量变化”与“当前 IP 质量”两类能力，但阶段 1 先按当前较简单的入口和展示来做。

## 6. 展示方式

- 理想方案是页面内弹窗，用户体验最好。
- 如果实现困难，可以用独立页面作为兜底方案。
- 阶段 1 先优先尝试页面内弹窗。
- 阶段 1 的主展示内容优先是“当前 IP 质量总览”。
- “历史”能力会保留入口方向，但具体放置位置取决于页面内弹窗是否可行：
- 若页内弹窗可行，优先考虑在弹窗中放“查看历史”入口
- 若页内弹窗不可行，再按独立页面方案调整
- “历史”默认是独立页面，不与当前状态总览混在一个复杂界面里。

## 7. 本服务的登录与会话

- 本服务需要登录鉴权。
- 这是单管理员思路，不做多管理员和普通用户体系。
- 注入脚本访问本服务时，使用本服务自己的登录态。
- 如果用户在 `Komari` 中点击注入按钮时尚未登录本服务，应跳转到本服务登录页，登录后再继续当前动作。
- 用户能够拿到 header 配置的前提是已经登录本服务，因此默认接受“Komari 登录态”和“本服务登录态”是两套独立会话。
- 本服务登录采用服务端 session + HttpOnly cookie。
- 本服务管理员密码按正常安全做法处理，只存哈希，不可逆。
- 本服务管理员体系就是单管理员，不做新增管理员、角色管理和普通用户体系。
- 如果用户从 `Komari` 跳转到本服务登录页，登录成功后不自动恢复原节点动作。
- 用户自行关闭该标签页或返回 `Komari` 继续操作。

## 8. 管理员初始化

- 默认管理员账号为 `admin`。
- 默认管理员密码为 `admin`。
- 首次登录后不强制修改密码。
- 用户如有需要，可自行修改密码。

## 9. 数据库与配置

- 阶段 1 允许并采用数据库。
- 数据库使用 `SQLite`。
- 需要保存本服务自身的管理员账号、节点记录、展示配置等内容。
- 节点主记录以 `komari_node_uuid` 作为唯一键。
- 节点接入时，阶段 1 只保存 `uuid` 和节点名称，不额外保存更多来自 `Komari` 页面上的辅助信息。
- 本服务节点详情页允许删除节点接入记录，删除只作用于本服务，不影响 `Komari`。
- 开发环境中，本服务自己的 `SQLite` 数据需要持久化。
- 开发环境按“标准开发环境”建设。
- 需要整理配置样例、启动方式、开发说明，以及 GitHub Actions 所需环境变量说明。

## 10. 数据模型方向

- 阶段 1 的数据结构按未来真实数据来设计，但当前内容可以先使用模拟值。
- 数据模型参考 `references/IPQuality/`。
- 阶段 1 虽不做复杂的“变化记录”产品能力，但后续设计需要考虑“当前 IP 质量”和“IP 质量变化”两条线。
- 阶段 1 不做实时检测。
- 阶段 1 不做“立即检测”按钮。
- 阶段 1 默认只展示最近一次已保存结果。
- 开发调试阶段，节点接入后可以自动分配一份结构接近真实模型的模拟结果。
- 真实使用阶段，如果没有真实数据，则显示 `N/A`，不使用模拟值伪装成真实结果。
- “开发模式”通过环境变量开启，默认是生产模式。
- 阶段 1 的检测结果存储以原始 JSON 为主，不预先把所有字段固定成数据库列。
- 阶段 1 只抽取少量稳定字段作为索引字段，其余详细内容从原始 JSON 动态读取和展示。
- 未来节点真实上报鉴权参考 `Komari` 的实现，正式方向采用“每节点独立 token”。
- 但这不是阶段 1 的实现范围，阶段 1 只允许在数据模型中预留相关字段，不实现 UI、接口和生成逻辑。
- 阶段 1 的历史能力也只需预留结构，不要求真正沉淀多条历史记录。

## 11. 当前默认原则

- 先跑通最小闭环。
- 尽量减少对 `Komari` 的高权限依赖。
- 能通过当前页面上下文拿到的信息，就不额外向 `Komari` 请求高权限接口。
- 能通过本服务承载的逻辑，就不要把复杂逻辑写进注入代码。

## 12. 独立前端默认结构

- 本服务登录后默认进入节点列表页。
- 阶段 1 的节点列表只显示已经从 `Komari` 添加过的节点。
- 节点列表提供一个最基础的关键字搜索。
- 阶段 1 搜索字段先按“节点名称 + UUID”实现。
- 节点列表默认展示字段先按以下最小集合设计：
- 节点名称
- 节点 UUID
- 当前状态摘要
- 最近更新时间
- 节点列表默认按最近更新时间倒序排列。
- 无检测结果的节点排在有结果节点之后。
- 阶段 1 的节点列表和详情页至少区分两种状态：
- `无数据`
- `有数据`
- 以上状态划分只是阶段 1 的最小集合，后续会扩展更多状态类型。
- 点击节点列表中的节点，进入本服务自己的节点详情页。
- 本服务节点详情页阶段 1 先围绕单节点工作：
- 顶部显示节点名和 UUID
- 主体显示当前状态总览
- 提供历史入口
- 预留节点接入配置/安装说明区域
- 不做过多旁支功能
- 节点详情页中显示 UUID，显示方式为“视觉截断 + 可见完整值/可复制”。
- 阶段 1 不提供“在 Komari 中打开原节点页面”的跳转入口。

## 13. 展示配置方向

- 本服务不承担“评估 IP 质量”的职责，主要负责承载、组织和展示结果。
- 阶段 1 不强行写死“当前状态摘要”的具体视觉风格。
- 展示内容应允许由用户决定“显示什么”。
- 阶段 1 的展示配置粒度先做“字段开关”，不做完整自定义布局。
- 阶段 1 的字段开关配置是全局配置，所有节点共用。
- 阶段 1 不做按节点单独配置展示字段。
- 后续可以扩展为更复杂的展示配置能力，这属于本服务的核心价值之一。
- 页内弹窗与本服务节点详情页共享同一套“当前状态”核心详情视图。
- 弹窗只是较轻的展示外壳，独立节点详情页在此基础上再承载历史入口和更多管理动作。
- 阶段 1 的当前状态核心详情视图默认为只读，不在节点页内做字段编辑。
- 展示字段开关统一放在系统配置页管理，不在节点页内调整。
- 系统配置页中的展示字段开关应即时影响：
- 独立前端节点详情页
- `Komari` 内弹窗视图
- 历史页先不强绑定这套字段开关，后续再单独设计
- 全局展示字段开关按“字段路径”保存配置。
- 阶段 1 的字段开关只记录显示/隐藏。
- 对于后续新出现、但用户从未配置过的字段，默认显示，用户之后可手动关闭。

## 14. API 设计方向

- 阶段 1 的后端接口采用 REST 风格。
- 接口从一开始就带版本前缀，使用 `/api/v1/...`。
- 管理接口与前台数据接口从一开始分开命名空间：
- 管理接口走 `/api/v1/admin/...`
- 节点展示与注入相关接口走 `/api/v1/...` 或 `/api/v1/nodes/...`
- 注入侧会先请求一个轻量状态接口，用于判断节点是“添加”还是“查看”状态。
- 轻量状态接口默认只返回最小状态信息，不直接返回完整详情。
- 详情数据通过单独的节点详情接口获取。
- 历史数据通过单独的历史接口承载，阶段 1 可先只预留分层与入口。

## 15. 前端与交付方式

- 前端交付方式尽量贴近 `Komari`。
- 阶段 1 采用：
- 前端源码位于 `web/`
- 使用 `npm` 作为前端包管理和构建入口
- 构建产物输出到 `public/`
- 最终由 Go 服务统一提供前端静态资源和 API
- 前端总体保持轻量，不做过重的前后端拆分部署
- 阶段 1 需要支持前端本地开发模式，不仅仅是构建后运行
- 本服务默认路径前缀为 `/ipq/`。
- 生产环境允许通过环境变量覆盖该前缀。
- 阶段 1 从一开始就按“支持子路径部署”实现前端。

## 16. 开发环境方式

- 开发环境主方案采用容器优先。
- 本地主要通过 `docker compose` 启动整套环境。
- 阶段 1 先把“开发用 compose”做好，生产示例可以后补。
- 开发环境中，`Komari` 也由我们的 `compose` 一起启动，不要求手工单独启动。
- 开发环境中，`Komari` 数据需要持久化。
- 开发环境中，本服务自己的 `SQLite` 数据也需要持久化。
- 开发环境应做到首次启动即基本可用。
- 开发环境默认初始化数据至少包括：
- 本服务默认管理员 `admin/admin`
- `Komari` 开发环境默认管理员 `admin/admin`
- 开发环境应提供可直接复制的开发用 header 示例。
- 开发环境默认采用同一入口反代方案，而不是让浏览器直接访问容器名。

## 17. 系统配置页范围

- 阶段 1 的系统配置页最小集合包括：
- header 生成
- 当前运行模式信息展示（开发/生产）
- 全局展示字段开关
- 管理员账号信息修改
- 管理员用户名和密码都可以修改。
- 修改管理员用户名或密码后，当前会话立即失效，需要重新登录。
- 当前运行模式只展示，不允许在页面中切换，切换仍由环境变量控制。
- header 生成区域支持预览最终代码内容。
- 短 loader 版和完整内联版都提供可预览文本和复制按钮。
- header 生成区域明确标记“短 loader 版”为推荐方案。
- 完整内联版作为静态快照导出，后续更新需重新复制。
- 登录页和系统配置页只存在于本服务独立前端中，不嵌入 `Komari`。
- `Komari` 内只承载节点相关动作和展示，不承载后台管理面板。

## 18. 历史展示范围

- 历史页阶段 1 先做轻量方案，不先做复杂图表。
- 历史页先支持按时间倒序的历史记录列表。
- 每条历史记录至少显示：
- 检测时间
- 简短摘要
- 进入该次详情的入口
- 历史详情页或历史详情区域阶段 1 可以直接展示格式化后的 JSON 作为兜底。

## 19. 节点接入区域预留

- 节点详情页从阶段 1 开始就预留“节点接入配置/安装说明”区域。
- 阶段 1 该区域只需占位或给出基础说明，不实现真实节点上报能力。
- 该区域主要用于为后续真实节点接入和上报流程预留页面结构与信息位置。

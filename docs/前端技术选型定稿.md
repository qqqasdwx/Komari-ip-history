# 前端技术选型定稿

本文档用于确认：**本项目前端技术选型以 `references/komari-web` 为主要参考依据**。

这里的“参考”不是把 `komari-web` 的所有依赖和能力原样照搬，而是：

- 参考它的核心技术栈
- 参考它的页面组织方式
- 参考它的布局和组件思路
- 参考它的整体工程方向

同时结合本项目自身范围，做**精简选型**。

---

## 1. 当前结论

本项目前端建议采用以下技术栈：

- **React**
- **TypeScript**

## 4. 本项目建议保留的核心技术选型

## 4.1 React

建议作为本项目前端唯一主线。

原因：

- 当前仓库已经开始做 React 迁移
- 当前项目页面结构已不适合继续长期维护手写 DOM
- 后续要继续做页内弹窗、设置页、结果页收口，React 更适合组织

---

## 4.2 TypeScript

继续保留。

原因：

- 当前前端已有 TS 基础
- 节点结果数据结构动态字段较多
- 配合 React 组件会更容易管理类型边界

---

## 4.3 Vite

继续保留。

原因：

- 与 `komari-web` 一致
- 当前项目本来也在使用 Vite
- 开发体验和构建速度都合适

---

## 4.4 react-router-dom

建议作为前端路由基础。

原因：

- `komari-web` 实际主路由也是基于 `react-router-dom`
- 我们项目需要：
  - 登录页
  - 节点列表
  - 节点详情
  - 系统配置
  - 可能的兜底页
- 这套已经足够

---

## 4.5 Tailwind CSS

建议作为主要样式基础。

原因：

- 与 `komari-web` 一致
- 更适合快速收口后台布局与页面节奏
- 对后续逐步统一风格更方便

---

## 4.6 @radix-ui/themes

建议保留。

原因：

- `komari-web` 也在用
- 适合构建后台风格 UI
- 与 Tailwind 搭配不会冲突
- 对表单、按钮、Dialog 这类基础能力很合适

---

## 4.7 必要的 Radix primitives

建议按需使用，例如：

- `@radix-ui/react-dialog`
- `@radix-ui/react-dropdown-menu`
- `@radix-ui/react-checkbox`
- `@radix-ui/react-label`
- `@radix-ui/react-slot`

理由：

- 这些正好覆盖我们当前真实需要的交互组件
- 不用自己重新造基础交互轮子

---

## 4.8 sonner

建议作为 toast / 反馈提示方案。

原因：

- `komari-web` 也在用
- 轻量
- 风格和现代后台比较一致

---

## 4.9 类名辅助工具

建议保留：

- `clsx`
- `class-variance-authority`
- `tailwind-merge`

原因：

- 当前前端后续一定会有通用按钮、标签、卡片、表单样式变体
- 这些工具可以减少 className 管理混乱

---

## 5. 路由组织方式定稿

虽然 `komari-web` 装了 `vite-plugin-pages`，但从当前代码看，它的主路由实际是：

- `BrowserRouter`
- `useRoutes(routes)`
- 手写 `routes.ts`
- `React.lazy + Suspense`

因此我建议本项目也采用同样思路：

### 采用

- 手写 `routes.ts`
- `React.lazy`
- `Suspense`
- layout 组件组织页面壳子

### 不采用

- 自动文件路由作为主组织方式

原因：

- 本项目页面数量不大
- 手写路由更清晰
- 更适合当前收口阶段

---

## 6. 页面组织方式定稿

建议参考 `komari-web` 的 layout 组织思路：

- 顶层 `App`
- 独立 `routes.ts`
- 公共 layout
- 管理后台 layout
- 页面组件按路由职责拆开

对本项目而言，建议最终组织成：

- `app/` 或 `pages/`
  - 登录
  - 节点列表
  - 节点详情
  - 系统配置
- `components/`
  - 通用按钮
  - 表单
  - 页面头
  - 结果展示块
  - 弹窗相关组件
- `lib/`
  - API
  - 结果格式化
  - 类型

---

## 7. 本项目不建议一并照搬的 Komari-web 能力

`komari-web` 里有很多依赖和能力，对它有价值，但对我们当前项目并不是必须。

当前不建议一起搬的包括：

- i18n 多语言体系
- PWA
- dnd-kit
- tanstack table
- recharts
- xterm
- next-themes
- 大量上下文层拆分

原因：

- 本项目当前最需要的是收口主链路
- 不是复制一个完整的大型前端工程
- 现在如果一起引入，会增加复杂度而不是减少复杂度

### 技术选型参考

### 后端行为参考

- `references/komari`

如后续需要，可用于参考：

- 后端接口语义
- 节点相关概念
- 某些交互习惯

但不作为前端技术选型依据。

---

## 9. 对当前仓库的直接结论

基于这份选型定稿，当前仓库后续应当朝这个方向收口：

1. React 作为前端唯一主线
2. 不再继续扩写 legacy 手写 DOM 主线
3. Tailwind + Radix 作为主要 UI 基础
4. 路由采用手写 `routes.ts` 管理
5. 后台布局参考 `komari-web`
6. 节点结果展示参考 `IPQuality`

---

## 10. 最终结论

一句话总结就是：

> **前端技术选型参考 `komari-web`，但只保留其核心栈与组织方式，不照搬它的全部依赖与功能体系。**

这会是当前项目最稳妥、也最符合现阶段需求的前端方向。

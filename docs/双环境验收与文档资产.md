# 双环境验收与文档资产

本文记录本地验收环境、自动化覆盖范围和文档截图刷新方式。这里的环境只用于开发和验收，不是生产部署方案。

## 一键命令

启动并准备双环境：

```bash
bash deploy/dev/start.sh
```

查看状态：

```bash
bash deploy/dev/status.sh
```

运行完整验收：

```bash
docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/run-acceptance-e2e.sh
```

刷新文档截图资产：

```bash
docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/refresh-doc-assets.sh
```

停止环境：

```bash
bash deploy/dev/stop.sh
```

## 环境矩阵

| 环境 | Komari | IPQ | Caddy 内部地址 | 数据目录 |
| --- | --- | --- | --- | --- |
| 默认主题 | `http://127.0.0.1:8080` | `http://127.0.0.1:8090` | `http://proxy:8080` / `http://proxy:8090` | `data/komari` / `data/ipq-default` |
| PurCarte | `http://127.0.0.1:8081` | `http://127.0.0.1:8091` | `http://proxy:8081` / `http://proxy:8091` | `data/komari-purcarte` / `data/ipq-purcarte` |

`prepare-acceptance-env.sh` 会构建前端并在 workspace 容器中启动两个 IPQ 后端进程。两个进程使用同一份当前代码和静态前端构建产物，但端口、数据库和公开访问地址互相隔离。

## 固定数据

每套环境都会重建以下开发种子节点：

| 节点 | 用途 |
| --- | --- |
| `开发种子-空节点` | 验证无目标 IP、空状态和游客空弹窗 |
| `开发种子-有目标无数据` | 验证有目标但无当前结果 |
| `开发种子-多IP历史` | 验证多目标、排序、分 IP 历史 |
| `开发种子-多快照对比` | 验证多快照、收藏快照和差异对比 |

真实 Debian 上报链路仍由 `deploy/dev/reporter/run-fixed-debian-ui-flow.sh` 单独覆盖。该脚本会从页面生成接入命令，并在固定 Debian 容器中真实执行。

## 自动验收

`run-acceptance-e2e.sh` 覆盖：

| 脚本 | 默认主题 IPQ | PurCarte IPQ | 覆盖点 |
| --- | --- | --- | --- |
| `verify-real-user-onboarding.mjs` | 是 | 是 | 从 Komari 创建节点、写入接入代码、点击注入入口、配置目标 IP、查看当前结果 |
| `verify-embed-auth-flows.mjs` | 是 | 是 | 游客未开放、游客开放、管理员未登录 IPQ 的嵌入权限链路 |
| `verify-react-preview-nodes.mjs` | 是 | 否 | 节点列表、详情、历史、快照和公开详情基础回归 |
| `verify-independent-node-binding.mjs` | 是 | 否 | 独立节点、Komari 绑定和解除绑定 |
| `verify-public-api-api-key.mjs` | 是 | 否 | 只读 Public API、API Key 和访问日志 |
| `verify-notifications.mjs` | 是 | 否 | 通知发送器、规则、字段变化触发和投递记录 |

默认主题和 PurCarte 的 Komari 注入页都会覆盖浅色和深色外观。涉及注入页时，验收重点不是“没有白底”，而是嵌入内容是否适配当前主题。

## 验收结论

第八步完成后，后续功能分支的默认验收流程是：

1. 启动双环境。
2. 重建固定种子数据。
3. 运行完整验收脚本。
4. 如有用户体验变更，刷新文档截图并人工查看对应页面。
5. 只在页面交互或核心业务逻辑发生变化时让用户介入验收。

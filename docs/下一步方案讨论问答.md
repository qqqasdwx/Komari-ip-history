## 0. 我当前的总体判断

我建议下一阶段不要把 6 个需求一起并行推进，而是先分成 3 组：

### A. 先修主链路和认知冲突

优先处理：

1. `Komari` 未接入节点点击图标时的错误跳转 / 错误嵌入后台页面
2. `cron` 执行时间与页面展示时区问题

原因：

- 这两项直接影响当前主使用流程
- 不解决会让后续新功能继续建立在一个不稳定的交互基础上

### B. 再做节点上报模型升级

优先处理：

3. 多 IP 从“服务端手动维护”转向“节点自动探查 + 服务端可控”
4. 节点不再完全依赖 `Komari` 接入

原因：

- 这两项会一起影响数据模型、安装脚本、节点/目标 IP 的生命周期
- 单独做其中一个，后面大概率还要返工

### C. 最后做对外能力扩展

优先处理：

5. 消息通知
6. 对外 API

原因：

- 这两项都依赖前面“节点 / 目标 IP / 历史变化 / 启停状态”的最终形态
- 过早做容易把接口和通知语义锁死

### 当前需求理解

- 用户选择某一个字段

- Hook 暴露足够字段，让用户通过注入代码或自建 API 自行通知

### 我当前建议

建议把通知系统拆成两层：

1. **事件层**
   - 只负责判断“哪个字段发生了变化”
   - 产出标准化变化事件

2. **发送层**
   - Telegram 作为内建发送器
   - Hook 作为通用发送器

- 邮件
- 飞书

- `field_label`
- `group_path`
- `previous_value`

## 3. 多 IP 改成节点自动探查

### 当前现状

当前实现是：

- 服务端维护目标 IP 列表
- 安装脚本逐个探查这些目标 IP

而你现在希望更接近：

- 节点自己探查所有应上报的 IP
- 服务端负责接收、展示、控制是否启用上报

### 我从 `references/IPQuality` 看到的结论

上游 `IPQuality` 脚本当前明确支持：

- 指定网卡：`-i eth0`
- 指定出口 IP：`-i <ip>`

但我暂时没有看到一套现成的“自动枚举所有网卡 / 所有绑定 IP，并逐个自动上报到外部服务”的完整模式。

也就是说：

- **自动探查多 IP 这件事，仍然需要我们自己的节点侧包装逻辑来完成**

### 我当前建议

我建议把“多 IP 来源”改成三层模型：

1. **节点**
   - 负责枚举本机可上报 IP

2. **服务端**
   - 负责接收和持久化每个 IP 的结果
   - 负责给每个节点一个“启用 / 停止上报”的控制开关

3. **安装脚本 / reporter**
   - 负责把节点上的探查逻辑封装起来

这里有一个很重要的问题：

- 目标 IP 是不是还要在服务端保留“人工编辑能力”

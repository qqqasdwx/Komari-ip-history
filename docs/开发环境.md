# 开发环境

本项目推荐使用两种开发方式：

- 日常开发：本机直接运行 Go 后端和前端开发服务器
- 联调与回归：使用 Docker Compose 启动完整环境

## 当前开发范围提示（截至 2026-04-17）

当前分支不再只覆盖最初的接入链路，已经同时包含多组能力：

- `Komari` 注入与未接入节点跳转链路
- cron / timezone / `CRON_TZ`
- 独立节点与 `Komari` 绑定
- 自动探查计划接口
- 目标 IP 启停
- 通知页、通知通道、规则、测试发送、投递日志
- API Key 与 public API

因此本地开发或联调时，建议不要只看“节点详情页是否能显示结果”，而要按本次改动范围做分层验证。

## 日常开发

适用场景：

- 修改 Go 后端
- 修改前端页面和样式
- 跑本地构建和单元测试

推荐方式：

- 后端：本机直接运行
- 前端：本机直接运行 Vite
- 数据库：本地 SQLite 文件

建议关注这些环境变量：

- `IPQ_APP_ENV`
- `IPQ_LISTEN`
- `IPQ_DB_PATH`
- `IPQ_PUBLIC_BASE_URL`
- `IPQ_BASE_PATH`
- `VITE_BASE_PATH`

默认推荐：

- `IPQ_BASE_PATH=""`
- `VITE_BASE_PATH=/`

### 当前最常用本地验证命令

后端：

```bash
go test ./...
```

前端：

```bash
cd web
npm run build
```

说明：

- 当前仓库里前端已提供 `build`
- 文档层面暂不把独立 lint 命令作为必需前置，因为现有脚本未统一收口为单一 lint 入口
- 如果你只改文档，也建议至少确认被引用的功能、接口名和页面入口仍与当前代码一致

## 联调环境

适用场景：

- 与 `Komari` 联调
- 验证注入链路
- 跑 Playwright
- 做回归验证

入口文件：

- `compose.dev.yml`

联调环境中的主要服务：

- `komari`
- `workspace`
- `proxy`

默认访问入口：

- IPQ 后端：`http://127.0.0.1:8090`
- 前端开发服务器：`http://127.0.0.1:5173`

## 推荐统一入口（保证你看到的和我看到的一样）

为了避免每次手动起环境都踩坑，当前仓库提供统一预览入口：

启动：

```bash
./scripts/dev/up.sh
```

停止：

```bash
./scripts/dev/down.sh
```

查看状态：

```bash
./scripts/dev/status.sh
```

统一入口的行为是：

1. 用 Docker 启动 `Komari + proxy`
2. 在宿主机启动 IPQ 后端（`8090`）
3. 在宿主机启动 Vite 前端开发服务器（`5173`）
4. 把 `IPQ_PUBLIC_BASE_URL` 固定为 `http://127.0.0.1:5173`
   - 这样 Komari 注入出来的页面、我调试时看的页面、以及你浏览器里看的页面是同一套实时刷新前端

推荐访问方式：

- Komari：`http://127.0.0.1:8080`
- IPQ 前端（实时刷新）：`http://127.0.0.1:5173`
- IPQ 后端 health：`http://127.0.0.1:8090/api/v1/health`

说明：

- 如果你只想看“最终构建产物”，可以访问 `8090`
- 如果你想看**实时刷新改动**，请优先访问 `5173`

### 当前联调建议优先覆盖的页面 / 能力

## 开发种子节点

如果需要快速重建一组固定联调节点，在仓库根目录执行：

```bash
docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/seed-dev-nodes.sh
```

脚本会重建以下开发节点：

- `开发种子-空节点`
- `开发种子-单IP历史`
- `开发种子-多IP历史`

## 给用户看页面前：准备验收节点集

以后凡是要给用户看本地页面，先在仓库根目录执行：

```bash
./scripts/dev/prepare-review-data.sh
```

这条命令会统一做 3 件事：

1. 清空当前本地开发环境中的临时节点 / 演示节点
2. 重建一组最小验收节点集，只保留这些分类：
   - 未接入 IPQ
   - 已接入 IPQ（空节点）
   - 已接入 IPQ（无数据）
   - 已接入 IPQ（单条数据）
   - 已接入 IPQ（多条历史）
   - 真实上报节点
   - IPQ 独立节点
3. 额外启动一个 Docker 模拟节点，**真实执行仓库里的 `deploy/install.sh`**，并等待它完成至少一次真实上报

说明：

- 真实上报节点不会走服务端动态脚本生成链路，执行的是仓库中的 `deploy/install.sh`
- 真实上报容器固定名为：`ipq-review-reporter`
- 节点摘要会写到：`.tmp/dev-preview/review-data-summary.json`

## 双主题联调环境

如果需要同时查看 **Komari 默认主题** 和 **PurCarte 主题**，执行：

```bash
./scripts/dev/up-dual.sh
```

启动后会得到两套环境：

- 默认主题
  - Komari: `http://127.0.0.1:8080`
  - IPQ UI: `http://127.0.0.1:5173`
  - IPQ API: `http://127.0.0.1:8090`
- PurCarte 主题
  - Komari: `http://127.0.0.1:8081`
  - IPQ UI: `http://127.0.0.1:5174`
  - IPQ API: `http://127.0.0.1:8091`

查看状态：

```bash
./scripts/dev/status-dual.sh
```

停止双环境：

```bash
./scripts/dev/down-dual.sh
```

重启后最小恢复命令：

```bash
./scripts/dev/resume-dual.sh
```

它会：

1. 重新拉起双套本地环境
2. 检查两套环境的验收数据摘要和当前节点集是否仍一致
3. 如果发现摘要丢失或节点集不一致，就自动执行 `./scripts/dev/prepare-review-data-dual.sh`

> 说明：
> - 当前约定是 **只保留项目级恢复流程**
> - **不要**默认做 machine-level / systemd 开机自启
> - 需要恢复环境时，由仓库内脚本显式执行即可

如果需要把两套环境都整理成可直接验收的最小节点集，再执行：

```bash
./scripts/dev/prepare-review-data-dual.sh
```

这会同时为 default / PurCarte 两套环境分别：

- 清空临时节点
- 重建最小验收节点集
- 启动各自的真实 reporter Docker 容器
- 等待真实上报节点出数

摘要文件：

- 默认主题：`.tmp/dev-preview/review-data-default.json`
- PurCarte 主题：`.tmp/dev-preview/review-data-purcarte.json`

## 前端修改后页面没更新

如果前端改了但页面没有变化，优先执行：

```bash
docker compose -f compose.dev.yml exec -T workspace sh /workspace/deploy/dev/workspace/reload-ipq.sh
```

这个脚本会：

- 重新构建前端
- 重启工作容器中的 Go 后端
- 等待健康检查恢复

建议排查顺序：

1. 先运行 `reload-ipq.sh`
2. 再检查页面 DOM、截图和接口返回
3. 不要先把问题归因到浏览器缓存

## 当前已知开发注意点（截至 2026-04-17）

1. 页面复制的安装命令当前已经改为调用**当前服务端直出的 install script**
   - reporter 配置与安装引导脚本都来自当前服务端
   - 如果准备正式发版，仍建议按目标版本完整回归一次节点侧安装流程

2. 时区能力当前已接通，并且页面已支持手动输入 IANA 时区名
   - 默认会回填当前浏览器时区
   - 用户也可以手动改成例如 `Asia/Shanghai`

3. 自动探查计划接口当前已支持“手动目标 IP + 自动发现目标 IP”并存
   - 启用的手动目标 IP 不会因为不在本次 `candidate_ips` 中而从计划里丢失

4. Javascript sender 已可用于基本流程开发验证
   - 但异步执行模型仍建议继续加固

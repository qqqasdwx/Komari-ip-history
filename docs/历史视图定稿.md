# 历史视图定稿

更新时间：2026-04-03

本文档用于固定“历史记录”功能的产品与交互口径。  
后续历史相关实现应以本文档为准，不再把“快照列表页”直接视为最终历史主页面。

---

## 1. 历史功能拆分

历史能力拆成两个页面：

1. 历史主页面
2. 快照对比页面

两者不是同一个页面的不同状态。

---

## 2. 历史主页面

### 2.1 主视角

历史主页面的主视角不是“快照列表”，而是：

- **字段变化列表**

也就是说：

- 每一行代表一次字段值变化
- 不是每一行代表一次上报快照

### 2.2 排序与分页

- 默认按时间倒序展示
- 时间列显示的是**新值时间**
- 默认每页 `10` 条
- 可切换每页数量：
  - `10`
  - `20`
  - `50`
  - `100`

### 2.3 查询能力

历史主页面需要支持：

- 按时间范围查询
- 按目标 IP 查询
- 按字段查询

目标 IP 查询规则：

- 使用下拉选择
- 默认不传 `target_id`，表示查询**所有 IP**
- 用户可以切换到某一个具体 IP，只看该 IP 的变化
- 历史列表中每条变化也需要标明是哪个 IP 发生了变化

字段查询规则：

- 不使用自由文本搜索
- 使用下拉选择
- 默认不传字段参数，表示查询全部字段变化
- 下拉选项必须使用用户可理解的展示名
- 不允许出现技术字段路径，如：
  - `Factor.IsProxy`
  - `Info.ASN`

### 2.4 每一行展示内容

每条变化至少展示：

- 变化时间
- 目标 IP
- 分组名
- 字段名
- 旧值（旧时间）
- 新值（新时间）

分组名采用细版路径，例如：

- `IP类型属性 / ipinfo / 使用类型`

而不是只保留一级分组。

### 2.5 变化判定规则

历史主页面只收录**真正发生过变化**的字段事件。

如果某次上报里字段没有变化：

- 不进入历史变化列表

以下都算变化：

- `A -> B`
- `N/A -> B`
- `A -> N/A`

其中：

- `A -> N/A` 第一版需要保留
- 但样式上应标记为**可能缺失**
- 不应和正常变化完全同权重
- 第一版默认**不直接过滤掉**这类变化
- 如果后续联调中这类记录噪音明显，再补一个显式过滤开关：
  - 隐藏“变为 `N/A`”的变化

### 2.6 时间语义

变化列表里的每一行不是简单的“相邻两次快照 diff 项”。

而是：

- 记录某个字段最近一次发生变化时的旧值
- 与本次变化后的新值
- 旧值和新值都要带上各自对应的时间

例如：

- `2026-03-01`
- `风险评分 / IP2Location`
- `20（2026-02-01） -> 30（2026-03-01）`

### 2.7 第一次变化

如果某个字段第一次出现变化，没有更早的旧值：

- 旧值显示 `N/A`

### 2.8 当前实现口径

当前历史主页面先按以下口径继续推进：

- 主体使用列表，而不是卡片
- 每页条数属于分页区，不属于筛选区
- 默认进入“所有 IP”视角，不预先锁定某一个目标 IP
- 时间范围使用一个整体控件，不拆成两个孤立输入框
- 时间范围支持快捷区间：
  - 今天
  - 近 7 天
  - 本周
  - 近 30 天
  - 本月
- 时间范围筛选支持完整时间，不再只按日期
- 当前版本以“先把信息放对”为优先，不急着过度压缩交互

但要注意：

- 当前历史页虽然已经不是快照列表，但**可读性仍未达标**
- 后续仍需要继续把变化行收成更接近自然语言的一句话
- 当前阶段先不因为 `A -> N/A` 过早加过滤逻辑，避免把真实变化一并压掉

## 3. 快照对比页面

### 3.1 入口

快照对比页面不是历史主页面本身。

它应从历史主页面右上角按钮进入：

- `快照对比`

### 3.2 顶部时间轴

页面顶部有一个时间轴：

- 起点：第一次上报时间
- 终点：最后一次上报时间

时间轴上有两个滑块：

- 左侧快照时间点
- 右侧快照时间点

用户可以拖动这两个滑块选择要比较的两个时间点。

时间轴交互要求：

- 使用**一条连续时间轴**承载两个滑块
- 时间轴中间应标明**有数据的历史时间点**
- 滑动时应实时显示当前选中的具体时间
- 时间轴应支持在页面滚动时保持可视：
  - 建议使用 `sticky`
- 页面文案中不强制出现“起始 / 终止”这类词
  - 可直接显示两个时间点各自的具体时间

### 3.3 页面内容

页面下方显示两个快照：

- 左边：起始时间快照
- 右边：终止时间快照

要求：

- 左右都复用详情页同一套展示布局
- 不再使用“技术字段对照表”式的 compare table
- 不同值高亮显示
- 高亮以**字段级**为准，不高亮整个分组
- 仍然使用展示页的最终翻译与值渲染

## 4. 统一翻译与渲染要求

这是历史功能的全局约束。

### 4.1 禁止暴露技术字段

在历史主页面和快照对比页面中：

- 严禁直接显示技术字段路径
- 严禁让用户看到类似：
  - `Factor.IsProxy`
  - `Score.IP2Location`
  - `Media.Netflix.Status`

### 4.2 与展示页统一

历史相关页面必须复用展示页同一套字段元数据与值渲染规则，包括：

- 分组名称
- 字段名称
- 值翻译
- 值颜色
- badge / 标签样式
- 缺失值显示方式

要求是：

> 用户只要能看懂 IPQuality 结果页，就一定能看懂历史页和快照对比页。

例如：

- 展示页里 `家宽` 是绿字
- 展示页里 `机房` 是红字

那么历史页里的：

- `家宽 -> 机房`

也必须延续同样的颜色语义。

## 5. 与当前实现的关系

当前已有实现可以视为：

- 历史接口基础能力
- 快照读取能力
- 基础 diff 能力

但这些仍然只是底层能力，不等于最终历史产品形态。

当前需要重构的关键点是：

- 历史主页面从“快照列表”改为“字段变化列表”
- 快照对比页从附属能力提升为独立页面
- 所有历史展示切换到统一的展示名和统一的值渲染

# 代码影响清单

本文档用于根据当前已经确认的新方向，梳理现有代码库里**会受影响的模块、文件和实现假设**。

当前判断基于以下最新口径：

- 默认按**跨域独立部署**设计
- Header 只做**桥接和入口注入**
- 产品体验上仍然**优先争取页内弹窗**
- 工程上必须保留**独立页面兜底**
- 注入弹窗鉴权以 `docs/注入弹窗鉴权定稿.md` 为准
- 节点下支持多个手动维护的目标 IP
- 当前重点包括：
  - 当前 IP 质量结果
  - 按目标 IP 查看历史

---

## 1. 当前最主要的代码影响面

以下五条主线目前都已经落了一版代码，文档保留它们主要是为了指导下一轮继续收尾：

从代码结构看，当前受影响最大的不是某一个函数，而是下面五条线：

1. **权限与公开接口线**
   - 需要把：
     - 游客只读
     - 管理员完整链路
     真正拆开

2. **弹窗状态机与回跳线**
   - `Komari` 游客和管理员的点击行为已经拍板
   - 但代码里还没有按这个状态机落地

3. **详情 / 历史接口边界线**
   - 当前详情接口仍然默认带历史
   - 已不符合“详情只看当前状态”的最新结论

4. **loader 与遗留实现收尾线**
   - iframe 弹窗骨架可以继续保留
   - 但 loader 扫描策略、legacy 残留和旧 `connect` 叙事都需要收口

5. **多 IP 与历史线**
   - 节点不再只对应一个当前结果
   - 当前结果和历史都已经下沉到目标 IP
   - 详情、公开接口、上报接口、前端节点详情都受影响

## 2. 当前最重要的风险点

### 2.1 多 IP 还缺接入命令展示

多 IP 主链路已经落地，但节点详情页里的“接入命令”还没有真正按 IP 列表生成，这会继续带来：

- 理解成本
- 文档与界面脱节
- 节点端实际接入动作还不够完整

### 2.2 匿名只读接口目前仍信任注入链路的前端分流

当前游客公开边界已经可用，但仍然主要依赖：

- `Komari /api/me`
- `Komari /api/nodes`

这意味着：

- 公开接口本身不再重复校验 `Komari` 节点公开状态
- 公开边界当前主要靠注入链路保证

这个边界是当前已接受的产品取舍，但后续要继续留意。

# 代码影响清单

本文档用于根据当前已经确认的新方向，梳理现有代码库里**会受影响的模块、文件和实现假设**。

当前判断基于以下最新口径：

- 默认按**跨域独立部署**设计
- Header 只做**桥接和入口注入**
- 产品体验上仍然**优先争取页内弹窗**
- 工程上必须保留**独立页面兜底**
- 当前重点是**当前 IP 质量结果**
- “历史变化 / 变化视图 / 趋势分析”不再作为当前主产品方向
- 开发环境改为**Debian 原生优先 + Docker 联调/CI**

---

## 1. 总结：当前最主要的代码影响面

从代码结构看，当前受影响最大的不是某一个函数，而是下面四条线：

1. **部署前提线**
   - 默认根路径独立部署已经落地
   - 但子路径兼容、旧文档、legacy 兼容逻辑仍有残留

2. **嵌入实现线**
   - 当前 header loader 已经实现了“页内弹窗 + iframe 嵌入”
   - 但它仍然需要按“跨域独立部署下是否稳得住”重新评估

3. **前端产品边界线**
   - React 主线已经收口到“当前结果优先”
   - 但 legacy 和部分辅助逻辑仍保留着旧方向实现

4. **开发与验证线**
   - Compose、CI、Playwright 已基本切到“Komari 联调入口 + 应用独立入口”
   - 但整套验证体系仍需要继续围绕真实跨域弹窗主链路做收口

---

## 2. 当前最重要的风险点

我认为当前最需要优先处理的，不是“删哪个页面”，而是下面三个结构性问题：

### 2.1 双前端并行

当前前端实际存在两套：

- `web/src/legacy-app.ts`
- `web/src/react-preview/*`

当前入口已经是：

- 默认加载 React
- `?ui=legacy` 才进入旧前端

对应文件：

- `web/src/main.ts`
- `web/src/legacy-app.ts`
- `web/src/react-preview/App.tsx`

这意味着：

- React 主线已经拍板并落地
- 但 legacy 仍然作为兼容入口留在仓库里
- 只要 legacy 还保留，就仍然会继续带来理解成本和维护噪音

这是当前仍然存在的工程风险之一，但已经不是“方向未拍板”问题。

---

### 2.2 子路径兼容能力与默认部署口径还需要继续拆开

默认入口已经收口到根路径，但子路径兼容仍然存在于：

- Header / loader
- BasePath 兼容逻辑
- legacy 前端
- 部分历史文档与实施记录

对应文件：

- `internal/config/config.go`
- `internal/service/header.go`
- `internal/app/app.go`
- `internal/render/static.go`
- `web/vite.config.ts`
- `web/src/legacy-app.ts`
- `README.md`

这不代表这些都要立刻删掉，但代表：

- 当前“兼容子路径”和“默认独立部署”虽然已经分开了一大半
- 但还没有完全收干净

---

### 2.3 历史 / 变化相关实现占比过高

当前代码里，历史、变化、变化规则已经形成一整条实现线：

- 后端有 history 数据结构与接口
- legacy 前端仍有历史页
- legacy 前端仍有变化页
- legacy 前端仍有变化规则设置页
- 还有 compare / priority 相关逻辑
- 一部分辅助逻辑和类型仍围着这条线设计过

这与当前产品方向已经不一致。

对应文件主要包括：

- `internal/service/nodes.go`
- `internal/service/settings.go`
- `internal/httpx/handlers/nodes.go`
- `internal/httpx/handlers/admin.go`
- `web/src/compare.ts`
- `web/src/legacy-app.ts`
- `web/src/react-preview/App.tsx`
- `web/src/react-preview/lib/changes.ts`
- `web/playwright/verify-stage1-flows.mjs`
- `web/playwright/verify-react-preview-nodes.mjs`

这里后续一定要做减法，但不建议现在盲删。

## 3. 建议按优先级看代码影响

## P0：必须先收口的代码影响

这些部分直接影响“当前架构是否和新方向一致”。

---

### 3.1 Header 生成与 loader 注入逻辑

对应文件：

- `internal/service/header.go`
- `internal/httpx/handlers/admin.go`
- `internal/httpx/handlers/embed.go`

当前现状：

- Header preview 已支持 `loader` / `inline`
- loader 能根据 `PublicBaseURL` 或 `window.location.origin + BasePath` 生成脚本地址
- loader 内已经实现：
  - 按钮注入
  - 登录跳转
  - 节点注册
  - 弹窗
  - iframe 嵌入
  - 独立页打开

当前问题：

1. 默认地址生成仍然保留了“当前站点 origin + BasePath”路径
2. loader 目前的主弹窗方案本质上是：
   - 打开 iframe
   - iframe 指向 `APP_BASE + "/#/nodes/:uuid?embed=1"`
3. 跨域独立部署时，这条链路是否稳定，还没有验证
4. 登录流程当前是：
   - `window.open(LOGIN_URL, "_blank")`
   - 但登录后如何回到弹窗可用状态，还没有收口成明确产品策略

影响判断：

- 这是当前**最核心**的一块
- 因为它直接关系到“页内弹窗优先、独立页兜底”到底能不能成立

建议动作：

- **优先保留现有 loader 骨架**
- 不要现在直接推翻
- 先把它标记为“重点验证对象”
- 先做技术验证，再决定：
  - 保留 iframe 弹窗
  - 改成轻量弹层 + API 请求
  - 还是弹层只做摘要，完整页独立打开

---

### 3.2 公开访问地址与默认部署口径

对应文件：

- `internal/config/config.go`
- `.env.example`
- `internal/httpx/handlers/auth.go`
- `internal/httpx/handlers/admin.go`
- `web/src/legacy-app.ts`
- `web/src/react-preview/lib/types.ts`

当前现状：

- 已有 `IPQ_PUBLIC_BASE_URL`
- `Me` 接口也会返回 `public_base_url`
- Header 预览也会用到这个配置
- `IPQ_BASE_PATH` 默认已改为空
- `VITE_BASE_PATH` 默认已改为 `/`

当前问题：

1. 默认值已经切到独立部署
2. 子路径兼容能力仍然保留
3. 但相关文档、legacy 和少量说明还没完全同步

影响判断：

- 这是部署口径层面的基础问题
- 不需要取消 `BasePath`
- 但需要持续把“默认值”和“兼容值”表达清楚

建议动作：

- 保留 `BasePath` 能力，作为兼容子路径部署的能力
- 但后续要考虑：
  - 是否在 UI 中更明确地突出 `IPQ_PUBLIC_BASE_URL`
  - 是否在生成 header 时优先要求用户确认公开地址

---

### 3.3 SPA 路由挂载方式

对应文件：

- `internal/app/app.go`
- `internal/render/static.go`
- `web/vite.config.ts`
- `web/index.html`

当前现状：

- 后端 SPA 挂载完全依赖 `cfg.BasePath`
- 前端构建 base 也依赖 `VITE_BASE_PATH`

当前问题：

- 当前实现更像“本服务挂在某个路径前缀下”
- 这对兼容子路径是好的
- 但当前产品已经不应该把它当默认前提

影响判断：

- 这是兼容性层，不是立即要删的点
- 但后续需要明确：
  - 哪些是“必须保留的兼容能力”
  - 哪些是“当前默认行为”

建议动作：

- **短期保留**
- 不建议立刻改路由挂载
- 先从文档和 UI 口径收口，再看是否需要调整默认值

## P2：开发环境与验证体系的影响

这些部分更多影响开发效率与回归方式。

---

### 5.1 Docker / Caddy / Compose

对应文件：

- `compose.dev.yml`
- `deploy/dev/caddy/Caddyfile`
- `deploy/dev/workspace/*`

当前现状：

- 现有联调环境完全围绕：
  - `proxy`

  - `workspace`

影响判断：

- 这套环境仍有价值
- 但现在应被视为：
  - 联调模型
  - E2E 模型
  - CI 模型
- 不再是唯一开发模式

建议动作：

- 先保留
- 不急着改 Compose
- 等代码主方向收口后，再决定是否补独立部署联调方案

### 5.2 Playwright 验证脚本

  - `/ipq/`
  - iframe 弹窗
  - history

  - change-priority

影响判断：

- 当前测试覆盖的重点，和当前产品重点已经有明显偏差

建议动作：

  - header 注入是否成功
  - 节点按钮是否出现

## 6. 建议“暂时不要删”的能力

虽然当前方向已经变化，但我不建议现在立刻删除下面这些：

### 6.1 `BasePath` / 子路径兼容能力

原因：

- 它仍然可以作为兼容部署方式存在
- 当前问题不是“支持子路径”，而是“不能把它当默认前提”

---

### 6.2 节点历史数据表与历史存储结构

原因：

- 即便前端先不展示，底层历史沉淀未必没有价值
- 现在直接删模型和删数据结构，收益不高

---

### 6.3 reporter 上报接口

原因：

- 它属于结果来源能力，不一定和“历史变化 UI”绑死
- 当前更多是前端展示位置问题，而不一定是后端接口本身有问题

---

## 7. 建议优先处理顺序

如果后面开始动代码，我建议顺序是：

### 第一步：先决定前端主线

先拍板：

- 继续以 `legacy-app.ts` 为主
- 还是切到 `react-preview` 为主

在这件事没定之前，不建议大规模改 UI。

---

### 第二步：只收口嵌入主链路

先围绕这条主链路收口：

1. Header 生成
2. 节点按钮注入
3. 页内弹窗
4. 独立页兜底
5. 当前结果展示

---

### 第三步：再做减法

减掉或降级：

- history 入口
- changes 入口
- change-priority 设置
- 结果页里过重的 reporter 配置展示

---

### 第四步：最后调整开发与验证

再去更新：

- Playwright
- CI
- Compose 文档
- README

---

## 8. 当前结论

当前这份代码影响清单的核心判断是：

1. **最先要处理的不是“删历史页”，而是“收口嵌入主链路”。**
2. **当前最大工程风险已经收口到“legacy 残留 + loader 跨域链路”这两条。**
3. **`/ipq/` 子路径能力可以保留，但现在已经不再是默认实现。**
4. **history / changes / change-priority 这条线明显已经该降级。**
5. **Playwright 和 CI 当前验证目标也需要后续一起收口。**
